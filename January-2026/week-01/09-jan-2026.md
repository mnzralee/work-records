**Title:** Tiered Registration Flow Refactor: 5-Step User Onboarding, Liveness Detection, 4-Step KYC Wizard with Face Verification & Comprehensive API Integration

**Objective:**
To complete the comprehensive registration flow refactor implementing a tiered 5-step user onboarding experience with liveness detection, construct a separate 4-step KYC verification wizard with document upload and biometric face verification, integrate complete API route layer for KYC application management, implement onboarding portal with dynamic status display based on verification stage, and establish route protection middleware ensuring authenticated users complete required verification steps before accessing dashboard functionality.

### Registration Step Components with Inline Verification

EmailWithOtpStep component created totaling 344 lines implementing email input with inline OTP verification transitioning through five states: input for email entry, sending during API submission, verify displaying OTP input, verifying during code validation, and verified showing success confirmation. Six-digit OTP input implements auto-advance functionality moving focus to next digit on entry, paste support detecting clipboard events and distributing digits across inputs, and complete callback triggering navigation when all digits entered and verified. Sixty-second resend cooldown prevents OTP spam with countdown timer display updating every second until cooldown expires enabling resend button.

PhoneWithOtpStep component created totaling 426 lines implementing phone input with country dial code selector automatically populating dial code when user selects country in previous step. Country selector displays flag emoji, country name, and dial code enabling users to change selection if needed. Phone number validation ensures proper format with country code prefix before allowing progression. Inline OTP verification mirrors email step pattern with identical state machine transitions, auto-advance digit entry, paste support, and resend cooldown timer.

### Liveness Detection Component Implementation

LivenessStep component created totaling 501 lines implementing motion-based liveness verification requiring three challenge completions: blink detection tracking eyelid movement, turn detection measuring head rotation, and smile detection analyzing facial expression changes. Camera access request implemented with permission handling displaying appropriate messages when camera unavailable or permission denied. Face detection overlay guide provides visual reference showing optimal face positioning within detection boundary with real-time feedback adjusting display based on detected face location.

Challenge progress indicators display current challenge instruction, completion status for each challenge type, and overall progress bar showing verification advancement. Scoring system accumulates points as user completes challenges with minimum threshold of 70 points required to pass verification ensuring sufficient liveness evidence captured. Component maintains challenge state tracking current challenge index, completed challenges array, accumulated score, and detection status enabling progression through verification sequence. Motion detection algorithms analyze frame differences between captures identifying significant movements corresponding to requested challenges with configurable sensitivity thresholds preventing false positives.

### API Route Implementation and Route Protection

Five KYC API routes created implementing BFF (Backend-for-Frontend) pattern proxying requests to backend svc-identity service. /api/kyc/personal-info POST route accepts date of birth, gender, and nationality creating or updating KYC application draft record. /api/kyc/documents POST route accepts document type and front/back image files uploading to storage and recording metadata. /api/kyc/biometric POST route accepts selfie image file uploading for face verification matching. /api/kyc/summary GET route retrieves current application status and collected data enabling review page to display complete information. /api/kyc/submit POST route finalizes application changing status from DRAFT to SUBMITTED and updating user profile kycStatus field to KYC_SUBMITTED.

RegistrationWizard redirect logic updated routing tiered registration completions to /onboarding instead of /verify preserving legacy flow redirect for backward compatibility. Middleware configuration updated adding /onboarding/:path* to protected routes array ensuring authentication required for onboarding portal access. Unauthenticated users attempting to access onboarding paths redirected to /login with return URL preserved enabling automatic redirect after authentication.

### Production Build Fixes and User Testing

Initial production build failed due to route group naming conflict where app/(onboarding)/ directory created route group not generating URL segments causing conflict with landing page at app/page.tsx. Resolution implemented renaming app/(onboarding)/ to app/onboarding/ converting from route group to regular route segment generating proper /onboarding URL path. Layout and page components relocated maintaining directory structure without parentheses notation.

User testing session identified three issues with initial inline OTP implementation: email step skipped OTP verification proceeding directly to next step, duplicate navigation buttons displayed with inline OTP components providing their own buttons conflicting with wizard navigation, and phone validation error where PhoneWithOtpStep validation differed from standard PhoneStep causing inconsistent behavior. Resolution implemented reverting to standard wizard flow using original EmailStep and PhoneStep components instead of inline OTP variants, removing custom navigation from inline components to rely solely on wizard controls, and including separate email-verify and phone-verify steps in tiered flow configuration ensuring proper OTP verification sequence.
