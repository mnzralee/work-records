# Daily Work Record

**Date:** 6th January 2026 (Monday)

**Title:** Major Architecture Decision - Multi-Cluster Environment Separation Strategy and VPS4 Infrastructure Redesign

---

## Objective

To analyze the current single-cluster Kubernetes architecture running all environments (MainNet production, TestNet staging, DevNet development) on shared infrastructure, identify the operational risks and compliance concerns inherent in this shared-control-plane design, and design a comprehensive multi-cluster separation strategy that would isolate each environment into independent K3s clusters on VPS4. The objective included proposing a three-cluster architecture (DevNet cluster on port 6443, TestNet cluster on port 6444, Monitoring cluster on port 6445) to eliminate blast radius risks, achieve compliance with financial blockchain regulatory requirements for production isolation, prevent operational errors from affecting MainNet during development work, and establish a foundation for independent environment lifecycle management as the GX Protocol prepares for extensive development of Modules 3-17. This architectural redesign was initiated based on the recognition that the current shared-cluster model, while functional for initial deployment, presents unacceptable risks for enterprise financial blockchain operations requiring production stability during aggressive development cycles.

## Work Carried Out

**Current Architecture Risk Analysis and Problem Identification.** The session began with a comprehensive analysis of the existing GX Protocol Kubernetes infrastructure spanning four VPS nodes (VPS1/72.60.210.201, VPS2/72.61.116.210, VPS3/72.61.81.3, VPS4/217.196.51.190) running as a single K3s cluster with approximately 70 pods across all environments. The current architecture was documented showing VPS1-VPS3 forming the MainNet control plane with three etcd replicas providing high availability, while VPS4 was tainted with `environment=devnet:NoSchedule` and `environment=testnet:NoSchedule` to host development and testing workloads. While namespace isolation and node taints provided some separation between environments, six critical risk categories were identified that justified the architectural redesign. Risk 1: Shared Control Plane with HIGH severity was identified because DevNet and TestNet API calls traverse the same Kubernetes API server components that serve MainNet production traffic, creating potential for development activities to impact production API availability or introduce instability through excessive API load. Risk 2: etcd Contention with MEDIUM severity was documented because all cluster state for MainNet production, TestNet staging, and DevNet development was stored in the same three-node etcd cluster, meaning development activity generating high key-value operations could degrade etcd performance affecting production. Risk 3: Blast Radius with HIGH severity identified that a single cluster-level failure such as API server crash, etcd corruption, or CNI plugin failure would simultaneously affect all three environments, making it impossible to maintain MainNet availability during TestNet or DevNet infrastructure experimentation. Risk 4: Compliance Risk with HIGH severity noted that financial regulators and enterprise blockchain standards typically expect production environments to be physically or logically isolated from development and testing environments, with the current shared-cluster model potentially failing audit requirements. Risk 5: Operational Error with HIGH severity recognized that administrator operations using kubectl commands could accidentally affect the wrong environment despite context switching, particularly during incident response when stress levels are high and mistakes more likely. Risk 6: Security Boundary with HIGH severity identified that all environments share the same RBAC (Role-Based Access Control) system, service accounts, and network policies, meaning a security compromise in DevNet could potentially provide lateral movement paths to MainNet resources.

**Strategic Timing Analysis for Architectural Change.** The question of why this architectural change should be pursued now rather than maintaining the status quo was analyzed through five strategic considerations. Consideration 1 identified Future Module Development as Modules 3-17 of the GX Protocol will require extensive development, testing, and debugging work that could span months, during which MainNet must remain stable and available for production operations. Consideration 2 emphasized Production Stability requirements recognizing that the GX Protocol MainNet is a financial blockchain system where downtime or instability directly impacts users, token transactions, governance voting, and organizational credibility. Consideration 3 documented Compliance Requirements noting that as the GX Protocol scales and attracts enterprise partners or regulatory scrutiny, the ability to demonstrate proper production isolation will become a mandatory rather than optional requirement. Consideration 4 highlighted Operational Safety acknowledging that the current shared-cluster model increases cognitive load on operators who must constantly verify environment context before executing commands, whereas separate clusters provide inherent safety through physical separation. Consideration 5 recognized Independent Scaling needs as DevNet may require rapid resource scaling during development sprints while TestNet needs production-like stability and Monitoring needs guaranteed resources, requirements that conflict when managed within a single cluster's resource allocation system.

**Multi-Cluster Architecture Design and Specification.** A comprehensive three-cluster architecture was designed for deployment on VPS4 (217.196.51.190) to replace its current role as a tainted node in the shared cluster. The Cluster 1 DevNet specification defined a K3s single-node cluster listening on API port 6443 with SQLite datastore for lightweight state management, Traefik ingress controller configured for ports 80/443 for HTTP/HTTPS traffic, pod CIDR 10.42.0.0/16 and service CIDR 10.43.0.0/16 for network addressing, namespaces fabric-devnet and backend-devnet for component organization, and external access through devnet.gxcoin.money and admin.gxcoin.money DNS records. The Cluster 2 TestNet specification defined a K3s single-node cluster on API port 6444 with SQLite datastore, Traefik ingress on ports 8080/8443 to avoid conflict with DevNet, pod CIDR 10.44.0.0/16 and service CIDR 10.45.0.0/16, namespaces fabric-testnet and backend-testnet, and external access through testnet.gxcoin.money DNS. The Cluster 3 Monitoring specification defined a K3s single-node cluster on API port 6445 with SQLite datastore, Traefik ingress on ports 9080/9443, pod CIDR 10.46.0.0/16 and service CIDR 10.47.0.0/16, namespaces prometheus, grafana, and loki for observability stack components, and external access through monitor.gxcoin.money DNS. The three-cluster design was chosen over a two-cluster alternative (combining DevNet and TestNet) to provide DevNet independence ensuring development work wouldn't affect TestNet staging environments, TestNet production-like configuration allowing exact mirroring of MainNet setup for accurate testing, monitoring isolation preventing observability stack failures from impacting workloads, and independent upgrades enabling each cluster to be maintained on different schedules without coordination overhead.

**Docker Compose Alternative Evaluation and Rejection.** Docker Compose was evaluated as a lighter-weight alternative to the three-cluster Kubernetes architecture and ultimately rejected based on six critical limitations. Limitation 1: Deployment Parity Gap recognized that using Docker Compose for DevNet/TestNet while MainNet runs on Kubernetes would create fundamental differences in how applications behave, making it impossible to have confidence that testing results would translate to production behavior. Limitation 2: No Kubernetes-Specific Testing identified that critical Kubernetes features including NetworkPolicies for pod-to-pod security, HorizontalPodAutoscaler for load-based scaling, and Ingress controllers for external access would be completely untestable in Docker Compose, creating blind spots in the testing process. Limitation 3: No Self-Healing or Rolling Updates noted that Docker Compose lacks Kubernetes features like automatic pod restart on failure, rolling deployment strategies, and health check integration that are essential for production operations. Limitation 4: Different Manifests Required recognized that maintaining separate docker-compose.yml files for development and Kubernetes YAML manifests for production doubles the infrastructure-as-code maintenance burden and creates opportunities for configuration drift. Limitation 5: Hyperledger Fabric Complexity documented that the GX Protocol's Hyperledger Fabric blockchain network with 5 orderers, 4 peers, 5 CAs, and 4 CouchDB instances is complex enough that managing it in Docker Compose would be significantly more difficult than using Kubernetes operators and Helm charts designed specifically for Fabric. Limitation 6: Team Skill Investment noted that the operations team has already invested significant effort in learning Kubernetes operations, and introducing Docker Compose for some environments would require maintaining expertise in two orchestration platforms.

**Dynamic Resource Allocation Strategy for VPS4.** A sophisticated resource allocation strategy was designed to enable three clusters to coexist on VPS4's 8 CPU cores, 32 GB RAM, and 400 GB storage without requiring static partitioning that would waste resources. The allocation strategy defined four resource pools: DevNet Cluster with 2 CPU and 8 GB RAM reserved (guaranteed minimum) and 5 CPU and 20 GB RAM maximum burst limit, Normal priority, used during active development work. TestNet Cluster with 2 CPU and 8 GB RAM reserved and 5 CPU and 20 GB RAM maximum burst limit, Normal priority, used during staging and QA validation. Monitoring Cluster with 1 CPU and 4 GB RAM reserved and 2 CPU and 8 GB RAM maximum burst limit, High priority to ensure observability remains operational even during high load. System/OS with 1 CPU and 2 GB RAM reserved and no burst capability, Critical priority to guarantee operating system stability. The dynamic allocation strategy was designed to allow DevNet and TestNet to burst beyond their reserved allocations when the other is idle, automatically enforced through cgroup v2 CPU and memory controllers, with the Monitoring cluster receiving priority during resource contention to prevent observability blackouts during incidents. Disk allocation was specified as DevNet 100 GB reserved with 150 GB maximum for Fabric ledger, PostgreSQL databases, and application logs, TestNet 100 GB reserved with 150 GB maximum for the same purposes, Monitoring 50 GB reserved with 80 GB maximum for Prometheus time-series database and Loki log storage, and System/OS 20 GB reserved for operating system, K3s binaries, and system packages. Network CIDR allocation was carefully designed to prevent any overlap between the three clusters, with DevNet using 10.42.0.0/16 for pods and 10.43.0.0/16 for services, TestNet using 10.44.0.0/16 for pods and 10.45.0.0/16 for services, and Monitoring using 10.46.0.0/16 for pods and 10.47.0.0/16 for services.

**Five-Phase Migration Plan Development.** A detailed implementation plan was developed to guide the migration from the current shared-cluster architecture to the three-cluster separated architecture. Phase 1 Preparation & Backup was defined with tasks to export all DevNet and TestNet Kubernetes resources using `kubectl get all --all-namespaces -o yaml`, backup all PostgreSQL databases using pg_dump for each database, backup Redis data using RDB snapshots, document the current exact versions of all deployed applications, and create a rollback plan in case migration encounters critical issues. Phase 2 Remove VPS4 from MainNet Cluster was defined with tasks to cordon VPS4 using `kubectl cordon srv1089624` to prevent new pod scheduling, drain VPS4 using `kubectl drain srv1089624 --ignore-daemonsets --delete-emptydir-data` to migrate all pods to other nodes, verify MainNet health after drainage to confirm the three-node control plane (VPS1-VPS3) is sufficient for production operations, uninstall K3s on VPS4 using the official uninstall script, and remove the node from the cluster using `kubectl delete node srv1089624`. Phase 3 Install Fresh K3s Clusters on VPS4 was defined with tasks to install the first K3s cluster with `curl -sfL https://get.k3s.io | INSTALL_K3S_EXEC="--disable traefik --kube-apiserver-arg=secure-port=6443" sh -` for DevNet, install the second K3s cluster with similar command but port 6444 for TestNet, install the third K3s cluster on port 6445 for Monitoring, configure systemd services for each cluster to enable independent lifecycle management with services named k3s-devnet.service, k3s-testnet.service, and k3s-monitoring.service, and verify all three clusters are operational with separate kubeconfig files. Phase 4 Deploy Components was defined with tasks to deploy the Hyperledger Fabric network to DevNet and TestNet clusters using Helm charts configured for single-node topology (replication factor 1 for all components), deploy backend microservices with single replicas rather than the 3-replica HA configuration used in MainNet, deploy frontend applications with appropriate environment-specific configurations, configure Traefik ingress routes for each cluster on their designated ports, deploy the monitoring stack (Prometheus, Grafana, Loki) to the dedicated Monitoring cluster, and configure cross-cluster monitoring so the Monitoring cluster can scrape metrics from both DevNet and TestNet. Phase 5 Verification & Cutover was defined with tasks to verify all services functional by running health checks on every component, test end-to-end functionality in both DevNet and TestNet including wallet operations and governance transactions, update DNS records to point devnet.gxcoin.money, testnet.gxcoin.money, and admin.gxcoin.money to the VPS4 IP with appropriate port configurations, configure Cloudflare or reverse proxy for port mapping if non-standard ports (8080, 9080) need to be hidden from users, clean up old DevNet/TestNet resources from the MainNet cluster namespaces, and update CI/CD pipelines to target the new cluster API endpoints.

**Architectural Decision Recording and Justification.** The architectural decision to proceed with multi-cluster separation rather than maintaining the status quo was formally recorded with explicit rationale. Decision 1: Separation Approach chose three K3s clusters over alternatives such as remaining with single cluster, using two clusters (merging DevNet/TestNet), or using Docker Compose for non-production environments, with rationale being complete environment isolation, independent lifecycle management, and elimination of blast radius risks. Decision 2: DevNet Platform chose Kubernetes over Docker Compose despite the latter's simplicity, with rationale being deployment parity with production ensuring testing accuracy, ability to test Kubernetes-specific features like NetworkPolicies and Ingress controllers, and leveraging existing team Kubernetes expertise. Decision 3: Resource Strategy chose dynamic allocation with cgroups over static partitioning, with rationale being efficient resource utilization allowing burst capacity when needed, automatic enforcement through kernel-level controls, and flexibility to adjust allocation as usage patterns change. Decision 4: Monitoring Location chose dedicated cluster over shared deployment in DevNet or TestNet, with rationale being isolation preventing monitoring stack failures from affecting workloads, guaranteed resources ensuring observability during high-load scenarios, and independent maintenance schedule. Decision 5: Implementation Timing chose to proceed after current Module 2 work completion rather than immediately, with rationale being minimizing disruption to ongoing development, allowing time for thorough planning and documentation, and ensuring rollback procedures are tested before production migration.

## Outcome

The January 6 architecture planning session successfully produced a comprehensive multi-cluster separation strategy addressing the six identified risk categories (shared control plane, etcd contention, blast radius, compliance, operational error, security boundary) that pose unacceptable hazards in the current shared-cluster architecture. The three-cluster design for VPS4 was fully specified with DevNet cluster on port 6443, TestNet cluster on port 6444, and Monitoring cluster on port 6445, each with dedicated network CIDRs (10.42-47.x.x/16 ranges), independent storage allocation (100-150 GB per environment cluster, 50-80 GB for monitoring), and separate ingress configurations preventing port conflicts. The dynamic resource allocation strategy was designed to efficiently utilize VPS4's 8 CPU cores and 32 GB RAM through cgroup-enforced reservations (2 CPU/8 GB per environment, 1 CPU/4 GB for monitoring) with burst capabilities allowing environments to expand to 5 CPU/20 GB when idle capacity exists. The Docker Compose alternative was evaluated through comprehensive feature analysis and rejected based on six critical limitations including deployment parity gap, inability to test Kubernetes-specific features, lack of self-healing capabilities, manifest duplication burden, Hyperledger Fabric complexity, and team skill investment fragmentation. Two major documentation artifacts were created totaling over 5,000 lines of comprehensive architecture documentation: the Kubernetes Architecture Report at `/home/sugxcoin/prod-blockchain/gx-infra-arch/docs/kubernetes-architecture-report.md` documenting the complete current state including all 70 pods, 15+ namespaces, 380 GB storage allocation, and three-layer architecture patterns, and the Environment Separation Architecture Study at `/home/sugxcoin/prod-blockchain/gx-infra-arch/docs/environment-separation-architecture.md` serving as a formal ADR with problem analysis, best practices research, option evaluation, and detailed migration planning. The five-phase implementation plan was developed covering preparation and backup (Phase 1), VPS4 removal from MainNet cluster (Phase 2), fresh K3s installation for three independent clusters (Phase 3), component deployment with single-node optimization (Phase 4), and verification with cutover including DNS updates (Phase 5). Enterprise best practices research validated the decision through citations from groundcover.com confirming "multi-cluster provides a high isolation level" with small blast radius, DataCamp analysis showing Docker Compose lacks "comprehensive deployment features needed for enterprise-grade production environments," and blockchain compliance resources confirming financial regulators expect production isolation. Historical work reconstruction completed four missing security incident records from December 15-18, 2025, using forensic analysis of package timestamps, security logs, and fail2ban statistics. The architectural decision was formally recorded with explicit rationale for each choice including separation approach (three K3s clusters for complete isolation), platform selection (Kubernetes for deployment parity), resource strategy (dynamic allocation for efficiency), monitoring location (dedicated cluster for reliability), and implementation timing (after current module work to minimize disruption). The proposed architecture represents a fundamental evolution from the initial shared-cluster deployment suitable for MVP launch to an enterprise-grade multi-cluster infrastructure capable of supporting financial blockchain operations requiring production stability during extensive development cycles for Modules 3-17.
