## Work Carried Out

### Automatic Face Detection and Capture System Implementation

User acceptance testing revealed friction in manual capture workflow where users struggled to simultaneously position their face correctly and press capture button resulting in misaligned or poorly-timed photographs. Product requirement redefined specifying automatic capture triggering when face detection algorithms confirm correct positioning eliminating manual coordination requirement. Implementation strategy leveraged existing MediaPipe Face Mesh integration from registration liveness testing providing consistent technical foundation across verification workflows.

MediaPipe Face Mesh hook integration replaced manual camera initialization resolving black camera display issue where previous getUserMedia implementation failed to properly stream video to display element. Face mesh hook provides managed camera lifecycle with proper initialization through start() method, video stream binding, and cleanup on unmount. Head pose detection utilizes headYaw metric from face mesh analysis measuring horizontal rotation angle enabling directional position verification for straight, left, and right capture requirements.

Threshold calibration implemented using empirically-derived values from liveness detection real-world testing. Rest position data analysis showed headYaw values ranging -0.01 to 0.01, left turn producing 0.03 to 0.05 positive values, and right turn producing -0.06 to -0.09 negative values. CENTER_ZONE threshold established at 0.02 for straight detection requiring headYaw absolute value below threshold, CLEAR_THRESHOLD established at 0.06 for directional detection requiring headYaw exceeding positive threshold for left turn or below negative threshold for right turn. Initial implementation contained inverted left/right logic where positive yaw incorrectly mapped to right turn; correction applied aligning with MediaPipe coordinate system conventions.

Stable position detection implemented requiring 8 consecutive frames with face in correct position before capture initiation preventing premature captures from transient movements. 500-millisecond delay after position confirmation provides visual feedback opportunity and ensures user awareness of imminent capture. Position checking callback evaluates current direction requirement against headYaw metrics: straight requires absolute value below 0.02, left requires value above 0.06, right requires value below -0.06. Infinite loop bug discovered where startFaceMesh() called repeatedly during effect execution; resolution implemented using cameraStartedRef guard preventing redundant initialization calls and Maximum update depth exceeded React error.

### Blur Detection Algorithm Implementation

Image quality assurance requirement specified automatic rejection of blurry captures ensuring all submitted photographs meet clarity standards for administrative review. Blur detection algorithm implemented using Laplacian variance method computing second-order derivative of grayscale image intensity measuring edge sharpness as proxy for focus quality. Algorithm processes captured canvas converting RGB pixel data to grayscale using luminance formula (0.299R + 0.587G + 0.114B).

Laplacian kernel convolution applied calculating each pixel's Laplacian value as -4 times center pixel plus sum of four cardinal neighbors (top, bottom, left, right). Variance of Laplacian values across entire image computed providing single blur score metric where higher values indicate sharper images with more defined edges. Threshold established at 50 based on empirical testing; images scoring below threshold rejected as insufficiently sharp with automatic retry triggering new capture attempt. This prevents submission of motion-blurred photographs from users moving during capture sequence.

### Visual Feedback System for Automated Capture

User interface redesigned providing clear visual feedback throughout automated capture process. Face detection indicator implemented as colored dot in top-right corner: green when face detected and positioned correctly, red when no face detected or positioned incorrectly. Border color system provides position status feedback: white at 50% opacity during active searching, yellow during "hold still" countdown when correct position detected, green during capture trigger, and red border with "No face detected" message when face mesh fails detection.

Direction-specific visual guides implemented with face oval overlay shifting position based on current capture requirement. Straight capture displays centered oval, left capture shifts oval leftward with animated chevron-left icon using Tailwind animate-pulse creating attention-drawing movement, right capture shifts oval rightward with animated chevron-right icon. Progress indicator displays current capture sequence (1/3, 2/3, 3/3) with completed captures shown as filled dots enabling user awareness of completion progress. Instruction overlay at bottom displays direction-specific guidance text updated dynamically based on current capture requirement.

### Enterprise Terminology Standardization Initiative

Product management review identified "biometric" terminology as potentially concerning for end users creating privacy apprehension despite actual implementation being simple facial photography without biometric template extraction. Decision made to rename all references to "Face Verification" providing clearer description of actual functionality while maintaining enterprise-appropriate professional terminology. Comprehensive audit identified 47 references requiring update across frontend types, components, pages, API routes, backend controllers, services, DTOs, and database schema.

Frontend type definitions updated renaming BiometricRequest to FaceVerificationRequest, BiometricDirection to FaceVerificationDirection, BiometricCapture to FaceVerificationCapture, and BIOMETRIC_DIRECTIONS constant to FACE_VERIFICATION_DIRECTIONS. Page directory renamed from /biometric to /face-verification with corresponding API route rename from /api/kyc/applications/[id]/biometric to /api/kyc/applications/[id]/face-verification. KYC context method updateBiometric renamed to updateFaceVerification with endpoint URL updated accordingly. KYC step progress component updated changing step type from 'biometric' to 'face-verification' and display label from 'Biometric' to 'Face Verification'.

Backend route definition updated from /biometric to /face-verification with controller method updateBiometric renamed to updateFaceVerification. Service layer DTO renamed from KYCBiometricDTO to KYCFaceVerificationDTO with all field references updated from biometric prefix to faceVerification prefix. Request body field names updated: biometricImageId to faceVerificationImageId, biometricHash to faceVerificationHash maintaining API consistency with frontend expectations.

### Database Schema Migration for Terminology Update

Database schema required column renames to maintain consistency with codebase terminology updates. Prisma schema updated renaming UserProfile columns: biometricHash to faceVerificationHash, biometricLivenessScore to faceVerificationLivenessScore, biometricFaceMatchScore to faceVerificationFaceMatchScore, biometricVerifiedAt to faceVerificationVerifiedAt, biometricNotes to faceVerificationNotes. KYCApplication columns renamed: biometricHash to faceVerificationHash, biometricImageId to faceVerificationImageId, biometricVerifiedAt to faceVerificationVerifiedAt.

Migration SQL script created executing ALTER TABLE RENAME COLUMN statements for all affected columns. Unique index on biometricHash dropped and recreated on faceVerificationHash maintaining constraint integrity. Migration executed directly via kubectl exec to PostgreSQL pod avoiding Prisma migration complexity for simple rename operations. Migration record inserted into _prisma_migrations table ensuring Prisma tracking recognizes applied changes preventing future migration conflicts.

Backend service deployment restarted using kubectl rollout restart ensuring new code loads with updated schema expectations. Health check verification confirmed service operational with correct database connectivity. Multiple atomic git commits created for both frontend and backend repositories: 8 frontend commits and 6 backend commits providing granular change tracking and potential selective revert capability.

### Legacy Single-Document System Decommissioning

KYC Step 2 documents module previously upgraded to multi-document architecture using KYCApplicationDocument junction table supporting multiple document submissions per application. Legacy single-document fields remained in KYCApplication table causing validation confusion where submit logic checked deprecated fields instead of populated documents table. Fields identified for removal: documentType, documentNumber, documentFrontId, documentBackId, documentExpiryDate, documentIssuingCountry, and OCR-related fields (ocrData, ocrNameMatch, ocrDobMatch, ocrNationalityMatch, ocrConfidenceScore).

Prisma schema updated removing eleven legacy document fields from KYCApplication model with comment added documenting migration to KYCApplicationDocument table. Schema push executed with --accept-data-loss flag acknowledging intentional field removal. Backend service layer updated removing deprecated KYCDocumentDTO interface and updateDocuments() method. Submit application validation logic rewritten querying KYCApplicationDocument table for application's documents, validating GOVERNMENT_ID document presence with required documentNumber and frontImageId fields, and constructing expiresAt from document-level expiry dates.

Controller endpoint updateDocuments() modified returning HTTP 410 Gone status with deprecation message directing API consumers to use document-specific endpoints. Frontend KYC context updateDocuments() method removed since documents page uses direct API calls to document-specific endpoints. Review page enhanced with useEffect fetching documents from /api/kyc/applications/:id/documents endpoint displaying each document type with upload status, document summary showing count of uploaded and skipped documents.

### Document Type Enumeration Extension for Face Verification

Face verification capture upload testing revealed HTTP 500 Internal Server Error with Prisma validation message indicating invalid value for documentType argument. Investigation identified DocumentType enum in Prisma schema missing face verification specific values required for storing directional captures. Enum extension implemented adding four new values: FACE_STRAIGHT for center-facing capture, FACE_LEFT for left-turned capture, FACE_RIGHT for right-turned capture, and FACE_RECENT for optional uploaded recent photograph.

Schema update applied using prisma db push with subsequent prisma generate ensuring TypeScript type definitions reflect new enum values. Docker image rebuild executed using --no-cache flag ensuring fresh layer creation incorporating schema changes. Image pushed to internal container registry and deployment updated using kubectl set image triggering rolling update with new container version. Backend health check confirmed successful startup with updated schema.

Frontend bug discovered during testing where retaking capture created duplicate array entries with same direction key causing React duplicate key error. Fix implemented modifying setCaptures callback to filter existing captures matching current direction before appending new capture ensuring array contains single entry per direction regardless of retake count. Pattern: setCaptures(prev => [...prev.filter(c => c.direction !== newCapture.direction), newCapture]).
