## Work Carried Out

### KYC Backend/Frontend Schema Alignment Initiative

Comprehensive schema alignment audit executed addressing multiple inconsistencies between Prisma database schema, backend DTOs, and frontend types causing KYC submission failures. Investigation triggered by 500 Internal Server Error responses with message "Invalid value for argument 'gender'. Expected Gender." Root cause analysis identified four distinct issues requiring systematic resolution.

**Backend Route Path Misalignment:** Frontend API routes incorrectly included application ID in paths (`PUT /api/v1/kyc/applications/${id}/personal-info`) while backend routes use JWT-based user identification without ID parameter (`PUT /api/v1/kyc/applications/personal-info`). Four frontend API route files updated removing application ID from backend URL construction: personal-info, submit, documents, and face-verification routes. This architectural difference reflects backend design principle where authenticated user identity derives from JWT payload rather than URL parameters.

**Status Enum Inconsistency:** Frontend types used Prisma enum values (SUBMITTED, UNDER_REVIEW) while backend read model returns transformed values (PENDING, IN_REVIEW). Frontend `types/kyc.ts` updated aligning KYCApplicationStatus enum with backend read model values. This mapping difference exists because backend domain layer transforms database states for API consumers, requiring frontend alignment with API contract rather than database schema.

**Missing DTO Validation Field:** Backend updateDocumentsSchema lacked proofOfAddressUploaded field required by controller and use case. Field added to `dto/request/index.ts` ensuring Zod validation passes for complete document status updates. Integration test suite created following Testing Trophy methodology covering personal info update flows, gender case conversion, date parsing, genesis eligibility calculation (age 13-73), and error handling scenarios. Unit tests confirmed 75 of 75 passing with successful TypeScript build.

### Gender Enum Consistency Standardization

Gender enum values standardized to uppercase throughout entire system stack matching Prisma enum definition. Analysis identified lowercase values ('male', 'female') in frontend forms conflicting with Prisma Gender enum (MALE, FEMALE). Repository layer previously performed case conversion, but inconsistent handling caused validation failures at various layers.

Backend DTO schema updated changing gender Zod enum from lowercase to uppercase values. Repository implementation simplified removing now-unnecessary toUpperCase() conversion. Frontend types updated using MALE/FEMALE values. UI components modified sending uppercase values while displaying user-friendly labels. Frontend API route validation changed from checking `['male', 'female'].includes(body.gender.toLowerCase())` to `['MALE', 'FEMALE'].includes(body.gender.toUpperCase())` with corresponding uppercase conversion before backend transmission.

Step 1 data persistence issue resolved where nationality and country fields failed to populate when returning from Step 2. Root cause identified as conditional truthy checks failing for null values. Fix implemented changing `if (personalInfo.nationalityCountryCode)` patterns to unconditional `setNationalityCountryCode(personalInfo.nationalityCountryCode || '')` assignments ensuring form fields populate regardless of null values.

### KYC Documents Page Response Format Resolution

Documents page TypeError "data.data is not iterable" resolved through response structure alignment. Investigation revealed backend returns nested object structure `{ success: true, data: { documents: [...] } }` while frontend incorrectly attempted direct array iteration on `data.data`. Fix implemented accessing `data.data.documents` with fallback to `data.data` for backwards compatibility, wrapped with `Array.isArray()` safety checks preventing iteration errors on non-array values.

Document type validation extended adding PROOF_OF_ADDRESS and DRIVER_LICENSE to validDocTypes array enabling complete document workflow support. Document save endpoint mismatch resolved where frontend called non-existent `PUT /applications/:id/documents/:docType` returning HTML 404 page causing JSON parse error. Frontend route updated to map document types to appropriate boolean flags (governmentIdUploaded, proofOfAddressUploaded) and call existing `PUT /applications/documents` endpoint.

### KYC Document Upload DTO Schema Correction

Document upload 400 Bad Request errors resolved through DTO schema analysis identifying required fields unavailable at validation time. Backend uploadDocumentSchema required fileName and mimeType fields validated via Zod on request body before controller could extract these values from multipart file object. Schema corrected making fileName and mimeType optional with runtime validation in use case layer after multipart processing.

Repository interface extended adding side parameter (`'FRONT' | 'BACK'`) to CreateDocumentInput interface enabling proper front/back image tracking. Prisma repository implementation updated storing URL in frontImageId or backImageId based on side parameter value. Use case layer updated passing side parameter to repository. Frontend upload route enhanced adding fileName and mimeType to outgoing FormData for defense-in-depth validation. Docker image rebuilt with --no-cache flag resolving containerd mount callback errors caused by cached layer corruption, successfully deployed with healthy pod verification.

### Document Metadata Persistence System Implementation

Critical bug identified where uploading back image of two-sided document (passport, national ID) deleted the front image. Root cause analysis revealed upload logic performing delete-then-create pattern destroying previously uploaded side data. Resolution implemented changing upload logic to UPDATE existing documents when adding secondary side image, preserving frontImageId when uploading backImageId and vice versa.

**Repository Interface Extension:** KYCDocumentReadModel extended adding documentNumber, frontImageId, backImageId, skipped, and skipReason fields enabling complete document state tracking. UpdateDocumentInput interface gained metadata fields for upsert operations. upsertByType method signature added to IKYCDocumentRepository interface enabling find-or-create semantics by document type.

**Prisma Repository Implementation:** upsertByType method implemented finding existing documents by applicationId and documentType, creating new record if not found, or updating existing record while preserving unchanged fields. documentSubType stored in ocrData JSON field until schema migration, with extraction in toReadModel for response mapping. This approach enables incremental schema evolution without breaking changes.

**Backend Endpoint Creation:** PUT /documents/:docType endpoint added to DocumentsController enabling direct metadata save without file upload. saveDocumentMetadataSchema created validating document type path parameter and body fields (documentNumber, expiryDate, frontImageId, backImageId, skipped, skipReason). SaveDocumentMetadataUseCase created and wired in DI container with repository dependencies.

**Admin Response Mapping Fix:** mapDocumentToResponse function updated adding frontImageId, backImageId, skipped, skipReason to response object. docType and fileId aliases added for frontend compatibility ensuring consistent field naming across API consumers.

### Face Verification Pre-Upload Workflow Implementation

Face verification workflow enhanced supporting dual-flow architecture accommodating both pre-uploaded and legacy processing patterns. Pre-uploaded flow accepts metadata when face images already uploaded via /documents endpoint (faceVerificationImageId, scores), while legacy flow maintains backwards compatibility accepting base64 images for real-time biometric processing.

**Schema Extension:** updateFaceVerificationSchema modified accepting faceVerificationImageId, faceVerificationHash, faceMatchScore, and livenessScore fields for pre-uploaded flow. Legacy selfieBase64 and documentBase64 fields retained for backwards compatibility. Face verification document types added to uploadDocumentSchema: FACE_STRAIGHT, FACE_LEFT, FACE_RIGHT, and FACE_RECENT enabling face capture storage in document system.

**Use Case Enhancement:** UpdateFaceVerificationUseCase updated with conditional logic detecting flow type. When faceVerificationImageId present without base64 images, use case accepts metadata directly storing provided hash and scores. When base64 images present, use case performs biometric processing generating hash and verification scores. This allows frontend to upload face images separately via /documents endpoint then finalize step with metadata submission.

**Response Field Alignment:** Face capture upload response updated returning actual fileId and fileUrl with Google Drive URL instead of synthetic documentId. Frontend mapping updated preferring fileId/fileUrl over documentId ensuring faceVerificationImageId populated correctly. Previous implementation returned documentId causing undefined faceVerificationImageId triggering 400 Bad Request on face verification save.

Initial state had 12 use-case domain directories with only auth, rbac, and admin wired; app.ts imported from legacy /routes/ directory. Final state achieves zero TypeScript errors with comprehensive Clean Architecture pattern implementation.

**Response DTO Enhancement:** DTOs widened using Partial<> wrappers adding missing properties: DeviceDetailsResponseDTO gained sessionHistory, name, type fields; SessionStatsResponseDTO gained byDeviceType percentage field with date type expanded to `string | Date`; SessionDetailsResponseDTO gained optional adminId, location, status fields; ExportUsersResponseDTO gained data, createdAt, createdBy fields; UserDetailsResponseDTO gained profileId, fullName, kycDocuments, freezeHistory fields.

**Admin User Creation Fix:** profileId made optional in CreateAdminUserInput interface since system admins may not have user profiles. PrismaAdminUserRepository enhanced with automatic UserProfile creation when profileId not provided, generating unique faceVerificationHash using timestamp and random string concatenation. Circular import errors resolved adding .js extensions for ESM module resolution in approval.service.ts and deployment.service.ts dynamic imports.

**Controller Creation:** Nine new controllers implemented following thin controller pattern with dependency injection: ApprovalController (8 methods for approval workflow), AuditController (5 methods for audit logging), DashboardController (6 methods for statistics), DeploymentController (5 methods for deployment management), EntityAccountsController (9 methods for entity lifecycle), NotificationController (8 methods for notification handling), ReconciliationController (5 methods for reconciliation workflow), SessionController (6 methods for session/device management), and UserManagementController (9 methods for user lifecycle).

**Route Factory Implementation:** Ten route factory functions created applying Clean Architecture pattern with middleware options interface: createApprovalRoutes, createAuditRoutes, createDashboardRoutes, createDeploymentRoutes, createEntityAccountsRoutes, createNotificationRoutes, createReconciliationRoutes, createSessionRoutes, createDeviceRoutes, createUserManagementRoutes, and createHealthRoutes. DI container expanded from approximately 468 lines to 709 lines wiring all 12 domains with use cases, repositories, and controllers. app.ts updated importing route factories from interface layer and using DI container for controller instantiation.

**Middleware Type Standardization:** Route factory middleware types standardized creating shared types file using Express RequestHandler. BaseRoutesOptions, ProtectedRoutesOptions, and SuperOwnerRoutesOptions interfaces established providing type-safe middleware injection. Thirteen route files updated removing `as never` type casts, replacing unknown parameter types with proper RequestHandler typing.
