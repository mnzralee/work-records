**Title:** Stateless Registration Architecture Implementation: In-Memory OTP Caching, Resend Email & Twilio SMS Integration, MediaPipe Liveness Calibration & Enterprise UI Redesign

**Objective:**
To refactor registration system from stateful database-dependent architecture to stateless in-memory design eliminating PendingRegistration table dependency, integrate production email delivery via Resend API and SMS delivery via Twilio with graceful dev mode fallback, calibrate MediaPipe liveness detection thresholds using real-world measurement data, redesign liveness verification interface with compact enterprise-grade styling, and resolve development environment port conflicts enabling isolated parallel development workflows.

## Work Carried Out

### Stateless Registration Architecture Implementation

Previous registration architecture analysis revealed complex stateful design where frontend state synchronized with backend PendingRegistration database table through multiple API calls, followed by migration transaction copying data to UserProfile table upon completion. This architecture introduced authentication bug identified in previous session where password endpoint failed to execute migration leaving users in PendingRegistration without UserProfile records. Additionally, database dependency created cleanup burden for expired registrations and complicated horizontal scaling due to state coupling.

Complete architectural refactoring executed replacing database-backed state with in-memory OTP caching and single-call registration completion. OTP cache service created implementing Map-based storage with configurable TTL expiration (10 minutes) and maximum 5 verification attempts per OTP preventing brute force attacks. Cache service stores email and phone verification status separately enabling frontend to track completion state without backend database queries. Test OTP "111111" supported in non-production mode enabling development testing without email or SMS provider integration.

Registration service rewritten with seven stateless API endpoints. checkEmail validates email uniqueness against UserProfile table, generates 6-digit OTP, stores in cache, and triggers email delivery returning success response. verifyEmailOtp validates submitted OTP against cache marking email verified status. resendEmailOtp regenerates OTP and triggers new email delivery. checkPhone mirrors email pattern validating phone uniqueness and triggering SMS delivery. verifyPhoneOtp validates phone OTP marking phone verified status. resendPhoneOtp regenerates phone OTP. completeRegistration accepts all registration data in single request creating UserProfile record, generating JWT tokens, and returning authenticated session.

Frontend registration wizard refactored removing registrationId state dependency since backend no longer tracks registration sessions. Name and country step now operates frontend-only without backend API call since data submitted with final registration. Final password step collects all accumulated form data and submits to single completeRegistration endpoint receiving JWT tokens and user profile in response enabling immediate authenticated session establishment.

### Email OTP Integration with Delivery Service

Email service examination revealed existing comprehensive implementation with branded HTML templates including OTP display with countdown timer, security warning notices, and mobile-responsive design. sendOTPEmail method accepts recipient email, 6-digit OTP, and purpose parameter (registration, password-reset) generating professionally formatted email with application branding.

Registration service integration implemented importing email service and invoking sendOTPEmail asynchronously after OTP generation ensuring API response not blocked by email delivery latency. checkEmail method triggers email delivery passing generated OTP and registration purpose. resendEmailOtp method triggers fresh email delivery with regenerated OTP. Email delivery failures logged but not propagated to API response maintaining graceful degradation where users can proceed with dev mode OTP if email service unavailable.

Kubernetes configuration updated adding email service environment variables including EMAIL_ENABLED feature flag, sender address configuration, sender name for email headers, and application URL for email template links. Email API credentials added to secrets store. Deployment verified through pod log examination confirming successful email delivery with message identifiers returned from delivery service API.

### SMS OTP Integration with Fallback Pattern

SMS service created integrating Twilio SDK for phone OTP delivery. Service implements sendOTPSms method accepting phone number with international format and 6-digit OTP, constructing branded message text, and invoking Twilio messages API. isOperational method checks for required credentials enabling graceful degradation when Twilio configuration unavailable. Error handling wraps SDK calls logging failures without propagating exceptions maintaining service resilience.

Configuration schema extended with Twilio environment variables including account SID, authentication token, sender phone number, and SMS_ENABLED feature flag. Registration service modified detecting SMS service operational status before attempting delivery. When SMS disabled, backend returns devOtp field in API response enabling development testing without external SMS provider. This pattern maintains consistent API structure while providing OTP visibility during development.

Frontend enhancement implemented detecting devOtp field in phone verification responses. When present, OTP logged to browser console with styled formatting (green bold text with DEV MODE prefix) providing clear developer feedback without modifying user-facing interface. Console message format distinguishes development mode from production behavior where OTPs delivered via actual SMS.

Dev mode pattern extended to email verification providing consistent development experience across both verification channels. Backend checks email service operational status returning devOtp when email delivery unavailable. Frontend logs email OTP to console with purple styled formatting matching email branding. This dual-channel dev mode enables complete registration flow testing without external service dependencies.

### Liveness Detection Threshold Calibration

User testing revealed liveness detection scoring 17% despite genuine facial movements indicating significant threshold miscalibration. Developer overlay data captured during testing provided actual MediaPipe output values enabling evidence-based threshold adjustment. Eye Aspect Ratio measurements showed eyes open ranging 0.35-0.45 with eyes closed ranging 0.17-0.27, significantly different from theoretical values used in initial implementation.

EAR thresholds recalibrated setting closed eye threshold to 0.28 (previously 0.19) and open eye threshold to 0.33 (previously 0.24) matching measured ranges. Head turn yaw measurements revealed turn angles of 0.03-0.09 compared to theoretical 0.22-0.30 requiring substantial threshold reduction to 0.03 for turn detection and 0.06 for clear turn classification. Center zone threshold reduced to 0.02 matching measured rest position range of -0.01 to 0.01.

Smile detection algorithm completely rewritten addressing fundamental calculation flaw. Original approach calculated mouth width divided by height producing unstable output when lips closed caused height approaching zero creating ratio approaching infinity with measurements fluctuating 0-90+. New approach implements corner lift detection measuring mouth corner Y position relative to mouth center. Smiling causes corners to lift producing measurable Y coordinate decrease in normalized face coordinates. Algorithm calculates lift score normalized 0-1 range combined with width expansion score producing stable smile metric. Thresholds set at 0.3 for neutral, 0.3-0.5 for slight smile, and 0.5+ for clear smile detection.

Multi-frame consistency requirements added requiring sustained detection across consecutive frames preventing spurious single-frame detections. Blink requires 2 consecutive frames with EAR below threshold. Head turn requires sustained position confirmation. Smile requires 2 consecutive frames above threshold. Quality metrics tracking implemented recording blink duration, turn frame counts, center crossings, and movement variance enabling detailed scoring based on action quality rather than binary detection.

Anti-spoofing analysis added calculating depth variance across facial landmarks (flat photographs exhibit low variance), micro-movement detection (natural faces exhibit involuntary micro-movements between frames), and symmetry analysis (photographs may exhibit unrealistic perfection). These metrics inform spoofing risk assessment though not yet integrated into pass/fail determination pending further calibration.

### Liveness Interface Enterprise Redesign

Liveness step interface redesigned implementing compact enterprise-grade styling suitable for financial application context. Icon sizes reduced from 20x20 to 14x14 and 16x16 pixels maintaining clarity while reducing visual weight. Button padding reduced from px-6 py-3 to px-4 py-2 creating tighter professional appearance. Face detection border reduced from 4px to 2px with green glow effect when face detected providing subtle feedback without visual dominance.

Challenge progress display simplified from verbose progress text to compact labels showing "Done", "Blink again", or current challenge instruction. Tips box styling reduced from prominent blue container to subtle gray with minimal borders. Overall padding reduced from py-8 to py-6 decreasing vertical space consumption. Color scheme aligned with application branding using purple (#470A60) and green (#17A210) accent colors with gradient status bar displayed during active challenges.

Video element retry crash resolved by modifying conditional rendering approach. Original implementation rendered video element only when state indicated active camera usage, unmounting element when state reached failed status. Retry attempt triggered useEffect accessing videoRef before React re-rendered causing null reference exception. Solution implemented always rendering video element with CSS visibility control using hidden class when not active, ensuring ref maintains valid reference throughout component lifecycle enabling reliable retry functionality.

### Development Environment Port Conflict Resolution

Registration endpoint 404 errors investigated revealing requests reaching incorrect backend service despite correct route definitions in code. Container inspection confirmed registration routes properly defined with all seven endpoints accessible via internal container HTTP requests returning expected responses with devOtp field present. Direct container testing via kubectl exec confirmed correct code deployment and route functionality.

Root cause identified as port conflict between parallel development environments. Port 3031 occupied by port-forward targeting shared development namespace rather than isolated developer namespace. Requests from frontend reached shared namespace service lacking stateless registration implementation while isolated namespace service contained correct code.

Port allocation strategy implemented assigning 304x series ports to isolated developer namespace while shared namespace retains 303x series. Environment configuration updated changing svc-identity from 3031 to 3041, svc-tokenomics from 3032 to 3042, svc-admin from 3033 to 3043, svc-government from 3034 to 3044, svc-onboarding from 3035 to 3045, and svc-organization from 3036 to 3046. Frontend environment variables updated reflecting new port assignments. Backend client default ports updated with documentation explaining port separation rationale. Port-forward commands updated in development documentation ensuring consistent environment setup.
