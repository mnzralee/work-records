Monday, February 23rd, 2026 — File Storage Service Production Hardening, Marketing Site Security Remediation, KYC Integration Repair, and Human-Readable Object Storage Architecture

File Storage Service Security Hardening (50-Finding Audit)
The day began with a comprehensive production readiness audit of the file storage microservice, where three parallel review agents examined all 35 backend source files, the full cross-service integration chain, and deployment infrastructure. The audit surfaced 50 findings across four severity levels: 10 critical, 14 high, 14 medium, and 12 low. The most critical security vulnerability was a timing attack on the API key middleware where a simple string equality comparison allowed character-by-character extraction of the secret key through response-time measurement. This was replaced with crypto.timingSafeEqual() using pre-computed buffer comparison with a length-safe guard. Additional security remediations included magic-byte file validation using the file-type npm package to prevent executable uploads disguised with false MIME types, Content-Disposition header injection protection via a character whitelist regex on filenames, reversal of the ClamAV virus scan bypass default from true to false, CORS restriction from wildcard to specific service origins, Zod schema validation on all endpoint parameters including UUID format enforcement and presigned URL expiry bounds (60-86400 seconds), and caller profile identity extraction via a new X-Caller-Profile-Id header for establishing file ownership chains.

Cross-Service Authentication Architecture Resolution
An architecturally significant authentication gap was discovered between the admin portal's Storage Browser and the file storage service. The admin frontend authenticates via JWT bearer tokens while the storage service expects service-to-service API key authentication via X-Service-Api-Key headers. The previous implementation used a Next.js URL rewrite rule that could perform routing but not header injection, rendering the entire admin file browser silently broken with 401 responses. The solution replaced the rewrite rule with a proper Next.js App Router API route that intercepts all storage requests server-side, injects the API key from environment variables never exposed to the browser, forwards the original JWT for audit purposes, and handles multipart form data by forwarding raw array buffers with intact boundary markers while stripping hop-by-hop headers. A parallel integration was implemented for the admin service using a factory pattern where a STORAGE_TYPE environment variable selects between the new HTTP storage client and the existing cloud drive client, ensuring zero-change backward compatibility.

Data Integrity, Performance Remediation, and Test Suite Development
Several data integrity vulnerabilities were identified and corrected. The file download pipeline was loading entire files into memory buffers requiring 25MB of contiguous allocation per concurrent download. This was replaced with a streaming architecture where downloadStream() returns a Readable stream piped directly to the HTTP response, reducing memory usage from O(fileSize) to O(chunkSize) at approximately 16KB per stream. A cross-bucket deduplication leak was discovered where findByHash(hash) searched across all storage buckets, potentially exposing files from restricted buckets. The fix scoped deduplication to the target bucket by adding a bucket parameter. A soft-delete restore paradox was resolved where findById() filtered out deleted records making it impossible to locate deleted files for recovery. Decompression bomb protection was added to the thumbnail service with a 100-megapixel limit. A test suite of 90 tests across 10 specification files was created following the testing trophy pattern, covering domain errors (9 tests), request DTO validation with SQL injection and path traversal prevention (23 tests), response DTO mapping (8 tests), upload use case with MIME rejection and SHA-256 hash computation (17 tests), five additional use cases (24 tests), thumbnail service (8 tests), and API key authentication middleware (8 tests). All 90 tests passed in 2.44 seconds.

KYC Integration Chain Repair and Marketing Site Full Security Remediation
Five integration mismatches between the identity verification service and the storage service were corrected: the verification service was passing application UUIDs instead of user profile UUIDs as the uploadedBy field, the uploaderType field was never included in upload requests, face verification photographs were routed to the wrong bucket, virus scan status was never checked allowing infected files to be linked, and file re-uploads left orphaned files in storage. Separately, a marketing website's production server received full-stack security remediation following a 30-finding audit (7 critical, 13 high, 10 medium). Docker container hardening was applied across seven dimensions: localhost port binding, 512MB memory cap, one-core CPU limit, PID ceiling of 256, read-only root filesystem with tmpfs, all Linux capabilities dropped, and privilege escalation blocked. Two deploy script logic bugs were fixed. Nginx hardening addressed a critical header inheritance flaw where child location blocks silently stripped parent security headers from static assets. System-level hardening covered SSH configuration conflict resolution, IPv6 firewall rules restricting HTTP/HTTPS to CDN IP ranges, fail2ban escalation to 24-hour SSH bans with three new web server jails, Docker log rotation, and automatic security updates.

Stale Container Image Diagnosis, Human-Readable Storage Keys, and Frontend Document Previews
During manual testing, city and country of birth fields were not persisting despite being present in local source code across 13 files. Investigation revealed the deployed container image predated the feature implementation. A cache-disabled rebuild was required, followed by pushing with a unique timestamp tag to force a genuine Kubernetes image pull. The namespace resource quota of 15 pods was exhausted and patched to 20. The most architecturally significant feature work implemented a human-readable key structure for object storage, organising files as {bucket}/{OwnerName_email}/{DocumentType}/{timestamp}-{uuid}.ext instead of opaque UUID paths, enabling administrators to locate documents without database queries. Three optional fields were added to the upload schema with Zod validation. The identity verification service was updated to fetch the user's name before uploading with try/catch ensuring uploads never fail due to name resolution errors. A server-side Next.js API route was created for document previews, authenticating user sessions, validating file identifiers as UUIDs, stripping storage:// protocol prefixes, and returning 15-minute presigned URLs. All changes were deployed through parallel Docker builds with unique timestamp image tags and verified end-to-end via browser automation confirming the human-readable key format, backward compatibility fallback, and valid presigned URL chain. Seventeen atomic commits were made across three repositories.

Outcome: The file storage microservice was elevated from development-functional to production-ready through remediation of 50 audit findings including timing-attack prevention, magic-byte validation, streaming downloads, and cross-bucket data leak fixes, validated by a 90-test suite passing in 2.44 seconds. A marketing site received full-stack security hardening across 30 findings spanning container runtime, web server, firewall, and SSH configurations. Tuesday, February 24th, 2026 — Enterprise Exception Handling Architecture, Full-Stack Storage Pipeline Hardening, Admin Storage Browser Audit Remediation, and KYC Document Preview Resolution

Discovery of Silent Fallback and Exception Handling Philosophy
The day began with a routine end-to-end test of the KYC document upload flow, where an uploaded government identification document was expected to appear in object storage with a human-readable folder structure but was instead silently saved to the container's ephemeral /tmp filesystem because the identity verification service had fallen back to a local storage adapter. The root cause was a missing environment variable combined with a Zod configuration schema that defaulted to local file storage when no value was provided. This single discovery exposed a systemic philosophy problem across the codebase: silent fallbacks, swallowed errors, and catch blocks that logged warnings without propagating failures. A comprehensive audit of both the storage service and the identity verification service mapped every error handling gap, identifying zero try-catch blocks across all seven object storage provider methods, zero try-catch blocks on database repository write operations, an error handler middleware recognising only two error categories, a virus scanner throwing generic errors indistinguishable from application crashes, and client services discarding structured error responses in favour of generic failure messages.

Domain Error Class Hierarchy and Middleware Overhaul
Six new domain error classes were implemented extending a base domain error class, each carrying the appropriate HTTP status code and structured detail payloads: validation error (400), database error (500), virus scanner unavailability (503), file-already-deleted (410), file download error (502), and presigned URL generation error (502). The error handler middleware was completely rewritten from a two-handler implementation to a five-category system covering domain errors, Zod validation errors parsed into field-level issue arrays, Prisma known request errors mapped by error code, Prisma initialisation errors returning 503, and multer file upload errors matched by code property rather than fragile string comparison. A critical Docker build compatibility issue was resolved when Prisma namespace exports behaved differently in the monorepo Docker context compared to local development, with all instanceof checks replaced by runtime duck-typing using error.constructor.name.

Infrastructure and Application Layer Error Wrapping
All seven methods in the object storage provider were wrapped with try-catch blocks following a consistent pattern: 404 responses from the cloud SDK are translated to domain file-not-found errors while all other failures are wrapped in the appropriate domain error with the original message preserved for logging. Five database repository write methods received error wrapping with special handling for the P2025 record-not-found case. Three generic error throws in the virus scanner service were replaced with a domain-specific unavailability error. The identity verification service's HTTP storage client was rewritten to parse structured error responses from the storage service, mapping storage error codes to appropriate domain errors. The configuration schema's silent default was removed entirely, and the dependency injection container's default case was changed from returning a local storage adapter to throwing an explicit error.

Comprehensive Test Suite, Code Audit, and Admin Storage Browser Remediation
The first testing pass confirmed 113 tests passing (90 existing plus 23 new), covering all six domain error classes and 15 error handler middleware integration tests including a secret-leak prevention test verifying internal error messages never escape to API responses. A second session produced 30 additional unit tests for the HTTP storage client covering every public method and edge case. A code audit across 10 files identified 1 critical finding (a non-null assertion on a cloud SDK response body that could crash on empty downloads), 7 warning-level findings, and 12 informational findings. Seven systematic fixes were executed: standardising download URL patterns to use the server-side proxy, correcting misleading icons, adding screen reader accessibility descriptions, fixing broken useCallback memoisation caused by unstable state object references in dependency arrays, wiring preview and delete actions into the user-grouped file view, adding image error fallback handling, and hardening the preview dialog with PDF iframe sandboxing. KYC Document Preview Resolution and Kubernetes Deployment
Three distinct frontend rendering bugs were resolved on the KYC documents page. The first bug was a classic React dependency cycle: a useCallback hook included state variables in its dependency array, and the function body called state setters that updated those same variables, creating an infinite re-render loop. The fix replaced state-based guards with useRef equivalents that do not trigger re-renders. The second bug involved the storage:// protocol prefix being passed through encodeURIComponent(), converting colons and slashes into percent-encoded characters that broke dynamic route matching and returned 404 responses. The third bug was CSS-related: saved document image labels rendered as inline elements, and CSS aspect-ratio does not apply to inline elements, causing image containers to collapse to zero width, resolved by adding display: block. The storage service was rebuilt and deployed to the Kubernetes cluster. The complete KYC document upload chain was verified end-to-end at the API level. The full Playwright suite of 5 tests passed and the storage browser's 8 existing E2E tests maintained 7 passing with 1 flaky timeout unrelated to code changes. The day concluded with 41 commits, 146 unit tests, and 13 end-to-end tests.

Outcome: The entire file storage and identity verification pipeline was transformed from a system riddled with silent fallbacks and swallowed errors into an enterprise-grade architecture with structured error propagation across every layer, validated by 146 unit tests and 13 end-to-end tests across 41 commits. The administrative storage browser was systematically audited and remediated from a 53-finding code review to a polished operational interface, and three distinct frontend rendering bugs were resolved to deliver working document image previews across all viewport sizes.

PROBLEMS ENCOUNTERED
The week surfaced a broad range of technical problems spanning infrastructure, security, data integrity, and frontend rendering. Twenty-one systemic bugs were discovered propagating silently across three repositories, including service client misrouting where API routes dispatched requests to incorrect backend microservices, JWT token leakage through raw error object serialisation in seven files, and incomplete schema migrations where smart contract events still emitted legacy field names. The file storage microservice had critical security vulnerabilities including a timing attack on API key comparison, missing magic-byte file validation allowing disguised executable uploads, cross-bucket deduplication leaks exposing files from restricted buckets, and entire file downloads loaded into memory buffers. A pervasive silent-fallback philosophy was uncovered where missing environment variables caused the identity verification service to silently write KYC documents to ephemeral pod filesystem storage instead of object storage, and catch blocks throughout the codebase swallowed errors without propagation. Stale Kubernetes container images predating feature implementations blocked KYC data persistence, Prisma schema columns defined in compiled output artefacts were never back-ported to source schema files meaning no migrations were ever created, and face verification uploads returned 500 errors due to seven critical gaps between the hardened document upload route and the unhardened face verification route. Frontend rendering bugs included React infinite re-render loops from useCallback dependency cycles, storage protocol prefixes breaking URL routing through percent-encoding, CSS aspect-ratio failing on inline elements, and data table filter popovers unmounting on every render cycle due to unstable arrow function references in column definitions.

SOLUTIONS FOUND
The systemic bugs were remediated through a three-tier parallel orchestration architecture with prompt-writing, implementation, and validation agents operating across zero-conflict file boundaries, producing seventeen file-by-file commits across three repositories with backward-compatible field mapping using nullish coalescing for historical blockchain data. The timing attack was eliminated by replacing string equality with crypto.timingSafeEqual() using pre-computed buffer comparison, magic-byte validation was added via the file-type npm package, cross-bucket deduplication was scoped by adding a bucket parameter to the hash lookup, and the streaming download architecture replaced full-buffer loading reducing memory from O(fileSize) to O(chunkSize) at approximately 16KB per stream. The silent-fallback philosophy was addressed by implementing a six-class domain error hierarchy with structured HTTP status codes, rewriting the error handler middleware from a two-category to a five-category system, wrapping all seven object storage provider methods and five database repository write methods with try-catch blocks, and removing the Zod schema default that allowed silent local storage fallback. Stale container images were resolved through cache-disabled Docker rebuilds with unique timestamp tags to force genuine Kubernetes image pulls, and namespace resource quotas were patched from 15 to 20 pods. The face verification pipeline was hardened with optional recent photo fields, capture preservation on failure, retry logic for transient errors, a FaceVerificationInfrastructureError domain error, and adaptive calibration backported from the registration wizard with 40-frame baseline collection and per-participant z-score thresholds. Frontend rendering issues were resolved by replacing state-based guards with useRef equivalents to break dependency cycles, stripping and encoding storage protocol prefixes before URL construction, adding display block to image containers for CSS aspect-ratio compliance, and wrapping column definitions in three tiers of useMemo for reference-stable filter popovers.
