Title: File Storage Service Production Hardening, Marketing Site Security Remediation, KYC Integration Repair, and Human-Readable Object Storage Architecture

Objective:
To execute a comprehensive production-readiness audit on a newly built file storage microservice by addressing 50 findings across security, data integrity, observability, and cross-service integration domains; to remediate 30 security findings on a marketing site's web server, container runtime, and system infrastructure; to resolve a stale container image deployment blocking KYC data persistence; and to implement a human-readable object storage key structure with presigned URL document previews across the frontend application layer, culminating in full Kubernetes deployment and browser-automated end-to-end verification.

Work Carried Out:

**File Storage Service Security Hardening (50-Finding Audit):** The day began with a comprehensive production readiness audit of a recently developed file storage microservice, where three parallel review agents examined all 35 backend source files, the full cross-service integration chain, and deployment infrastructure. The audit surfaced 50 findings across four severity levels: 10 critical, 14 high, 14 medium, and 12 low. The most critical security vulnerability was a timing attack on the API key middleware, where a simple string equality comparison on line 28 of the authentication middleware allowed character-by-character extraction of the secret key through response-time measurement. This was replaced with `crypto.timingSafeEqual()` using pre-computed buffer comparison with a length-safe guard. Additional security remediations included magic-byte file validation using the `file-type` npm package to prevent executable uploads disguised with false MIME types, Content-Disposition header injection protection via a `[a-zA-Z0-9._-]` whitelist regex on filenames, reversal of the ClamAV virus scan bypass default from `true` to `false` so production deployments receive scanning by default, CORS restriction from wildcard to specific service origins, Zod schema validation on all endpoint parameters including UUID format enforcement and presigned URL expiry bounds (60-86400 seconds), and caller profile identity extraction via a new `X-Caller-Profile-Id` header for establishing proper file ownership chains.

**Cross-Service Authentication Architecture Resolution:** An architecturally significant authentication gap was discovered between the admin portal's Storage Browser and the file storage service. The admin frontend authenticates via JWT bearer tokens, while the storage service expects service-to-service API key authentication via `X-Service-Api-Key` headers. The previous implementation used a Next.js URL rewrite rule that could perform routing but not header injection, rendering the entire admin file browser silently broken with 401 responses. The solution replaced the rewrite rule with a proper Next.js App Router API route at `app/api/v1/storage/[...path]/route.ts` that intercepts all storage requests server-side, injects the API key from environment variables (never exposed to the browser), forwards the original JWT for audit purposes, and handles multipart form data by forwarding raw array buffers with intact boundary markers while stripping hop-by-hop headers. A parallel integration was implemented for the admin service using a factory pattern where a `STORAGE_TYPE` environment variable selects between the new HTTP storage client and the existing cloud drive client, ensuring zero-change backward compatibility for existing deployments.

**Data Integrity and Performance Remediation:** Several data integrity vulnerabilities were identified and corrected. The file download pipeline was loading entire files into memory buffers, requiring 25MB of contiguous allocation per concurrent download. This was replaced with a streaming architecture where `downloadStream()` returns a `Readable` stream piped directly to the HTTP response, reducing memory usage from O(fileSize) to O(chunkSize) at approximately 16KB per stream. A cross-bucket deduplication leak was discovered where `findByHash(hash)` searched across all storage buckets, potentially exposing a file from a restricted bucket when an identical file existed in a different bucket. The fix scoped deduplication to the target bucket by adding a bucket parameter to the query. A soft-delete restore paradox was resolved where `findById()` filtered out deleted records, making it impossible for the restore endpoint to locate deleted files for recovery. A new `findByIdIncludeDeleted()` method was added with explicit deleted-state validation. Decompression bomb protection was added to the thumbnail service with a 100-megapixel limit via Sharp's `limitInputPixels` configuration, graceful shutdown was enhanced with proper database client disconnection, and missing database indexes on `deletedAt` and `createdAt` columns were added to eliminate sequential scans on frequently filtered and sorted columns.

**KYC Integration Chain Repair:** Five integration mismatches between the identity verification service and the storage service were corrected. The verification service was passing application UUIDs instead of user profile UUIDs as the `uploadedBy` field, making per-user file queries impossible. The `uploaderType` field was never included in upload requests, causing all user documents to appear as system-uploaded. Face verification photographs were routed to the general documents bucket instead of the dedicated face verification bucket despite having different retention policies and sensitivity classifications. The virus scan status returned by the storage service was never checked, allowing infected files to be linked to verification applications without warning. Finally, file re-uploads left orphaned files in storage with no reference, which was resolved by adding best-effort deletion of the old file before storing replacements.

**Comprehensive Test Suite Development (90 Tests):** A test suite of 90 tests across 10 specification files was created following the testing trophy pattern. Domain error class tests (9 tests) verified error message format, operational flags, and HTTP status code mappings. Request DTO validation tests (23 tests) covered schema enforcement including SQL injection rejection, path traversal prevention, UUID format validation, and pagination bounds. Response DTO mapping tests (8 tests) verified public field inclusion and internal field exclusion. The upload use case received 17 tests covering MIME type rejection, size limits, SHA-256 hash computation, deduplication with soft-delete awareness, virus scan invocation, and thumbnail generation. Download, restore, presigned URL, delete, metadata, and statistics use cases received 24 tests combined. The thumbnail service (8 tests) verified format detection and output validation via JPEG magic byte verification. API key authentication middleware tests (8 tests) covered timing-safe comparison, header extraction, and missing credential handling. All 90 tests passed in 2.44 seconds with zero TypeScript compilation errors.

**Marketing Site Full Security Remediation:** A separate workstream executed a complete security remediation on a marketing website's production server following a post-migration audit that had identified 7 critical, 13 high, and 10 medium severity findings. Docker container hardening was applied across seven dimensions: port binding was restricted from all interfaces to localhost only, memory was capped at 512MB, CPU limited to one core, the PID ceiling was set to 256, the root filesystem was made read-only with tmpfs for temporary files, all Linux capabilities were dropped, and privilege escalation was blocked. Two deploy script logic bugs were fixed: a race condition where the deployment state file was written during cleanup rather than immediately after traffic switching, and a rollback destruction issue where the old container was removed instead of merely stopped. Nginx hardening addressed a critical header inheritance flaw where child location blocks with their own `add_header` directives silently stripped all parent-level security headers from static assets. A shared snippet file included in every location block resolved this. Additional Nginx changes included a default server block returning connection drops for direct-IP access, Content-Security-Policy in report-only mode, rate limit status code correction from 503 to 429, and expanded sensitive path blocking. System-level hardening covered environment file permissions (644 to 600), SSH configuration conflict resolution across drop-in files that were re-enabling password authentication, IPv6 firewall rules restricting HTTP/HTTPS to CDN IP ranges only, fail2ban escalation to 24-hour SSH bans with three new web server jails, Docker log rotation, automatic security updates, and audit rules monitoring sensitive configuration files.

**Package Removal Incident and Recovery:** During post-hardening cleanup, removal of the legacy web server triggered a dependency cascade that unexpectedly removed the active Nginx reverse proxy as well, due to a shared package dependency. The site immediately returned origin-unreachable errors from the CDN. The hardened Nginx configuration was restored from session notes within approximately two to three minutes of downtime. The incident reinforced the operational lesson that package removals sharing dependency trees with active services require explicit package pinning or immediate post-removal verification.

**Stale Container Image Diagnosis and Kubernetes Quota Resolution:** During manual testing of the identity verification flow, city and country of birth fields were not persisting despite being present in the local source code across 13 files. Investigation via pod inspection revealed the deployed container image predated the feature implementation. The initial rebuild still produced a stale image due to Docker's build cache reusing layers where the COPY hash appeared unchanged. A cache-disabled rebuild resolved this, but the Kubernetes node then served its locally cached image due to the `IfNotPresent` pull policy on reused tags. The definitive fix was pushing with a unique timestamp tag to force a genuine image pull. Additionally, the namespace resource quota of 15 pods and 15 services was exhausted, silently blocking new service creation. The quota was patched to 20, enabling the storage service to be created and a previously scaled-down service to be restored.

**Human-Readable Object Storage Key Architecture:** The most architecturally significant feature work implemented a human-readable key structure for the object storage layer. The previous format stored files under opaque UUID-based paths such as `{bucket}/USER/{uuid}/{timestamp}.ext`, which were completely unintelligible when browsing the storage system directly. The new format organises files as `{bucket}/{OwnerName_email}/{DocumentType}/{timestamp}-{uuid}.ext`, enabling administrators to locate any user's documents without database queries. Three optional fields (owner name, owner email, and category) were added to the upload schema with Zod validation enforcing 200-character name limits and email format validation. The upload use case branches on metadata availability: when owner information is present, a `sanitizePathSegment()` helper replaces spaces with underscores, strips characters outside `[a-zA-Z0-9@._-]`, and truncates at 200 characters to produce S3-compatible paths. When metadata is absent, the original UUID-based structure serves as a backward-compatible fallback. The identity verification service was updated to fetch the user's name from the user repository before uploading, with a try/catch ensuring the upload never fails due to a name resolution error. A hand-written Prisma migration added three nullable columns and an owner email index, necessitated by the development environment's PostgreSQL instance lacking shadow database creation permissions required by Prisma's automated migration tooling.

**Frontend Document Preview via Presigned URL Proxy:** The KYC documents page previously displayed a static placeholder after page reload, with no mechanism to convert stored file references back into viewable images. Since the documents page is a client-side rendered component that cannot access server-side environment variables, a direct browser-to-storage request was architecturally impossible without exposing the API key. A server-side Next.js API route was created at `app/api/kyc/documents/[fileId]/url/route.ts` that authenticates the user session, validates the file identifier as a UUID, strips any `storage://` protocol prefix, proxies the request to the storage service with server-side credentials, and returns a 15-minute presigned URL. The documents page gained `documentUrls` and `loadingUrls` state objects, a `fetchDocumentUrl` callback, and a `useEffect` that auto-fetches presigned URLs for all saved image identifiers on mount. The rendered document cards display the actual uploaded image with a hover overlay pattern showing a semi-transparent "Click to replace" affordance, using plain HTML `<img>` tags rather than the framework's optimised Image component due to the environment-variable nature of the object storage hostname.

**Kubernetes Deployment Pipeline and Playwright Verification:** All changes were deployed through parallel Docker builds for the storage and identity verification services with unique timestamp image tags, followed by database migration application and port-forward refresh. Browser-automated end-to-end verification confirmed the admin Storage Browser loaded without errors and displayed file grids with bucket tabs, a direct upload with owner metadata produced the human-readable key format `kyc-documents/E2E_TestUser_e2e-test@test/GOVERNMENT_ID/{ts}-{uuid}.jpg`, an upload without metadata fell back to the UUID-based structure confirming backward compatibility, and the presigned URL chain returned valid time-limited download URLs. Seventeen atomic commits were made across three repositories following conventional commit specifications, with 12 backend commits, 2 frontend commits, and 3 parent repository pointer updates.

Outcome:
A file storage microservice was elevated from development-functional to production-ready through remediation of 50 audit findings including timing-attack prevention, magic-byte validation, streaming downloads, and cross-bucket data leak fixes, validated by a 90-test suite passing in 2.44 seconds. A marketing site received full-stack security hardening across 30 findings spanning container runtime, web server, firewall, and SSH configurations. A stale container deployment blocking KYC data persistence was resolved through cache-busted rebuilds and unique image tagging. A human-readable object storage key architecture was implemented across 13 files with backward-compatible fallback, complemented by presigned URL document previews in the frontend, all deployed to Kubernetes and verified end-to-end with browser automation.
