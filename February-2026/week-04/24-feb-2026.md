Title: Enterprise Exception Handling Architecture, Full-Stack Storage Pipeline Hardening, Admin Storage Browser Audit Remediation, and KYC Document Preview Resolution

Objective:
To eliminate silent failure modes across the entire file storage and identity verification pipeline by designing and implementing a multi-layer enterprise exception handling architecture, to systematically audit and remediate the administrative storage browser interface through a comprehensive 53-finding code review and 9-finding UI assessment, and to resolve three distinct frontend rendering bugs preventing document image previews from loading on the KYC documents page — validating all changes through 146 unit tests, 13 end-to-end tests, live API verification, and Kubernetes deployment.

Work Carried Out:

**Discovery of Silent Fallback and Exception Handling Philosophy:** The day began with a routine end-to-end test of the KYC document upload flow, where an uploaded government identification document was expected to appear in the object storage service with a human-readable folder structure. Instead, the file was silently saved to the container's ephemeral `/tmp` filesystem because the identity verification service had fallen back to a local storage adapter. The root cause was a missing environment variable combined with a Zod configuration schema that defaulted to local file storage when no value was provided. This single discovery exposed a systemic philosophy problem across the codebase: silent fallbacks, swallowed errors, and catch blocks that logged warnings without propagating failures. A comprehensive audit of both the storage service and the identity verification service mapped every error handling gap before any code was written, identifying zero try-catch blocks across all seven object storage provider methods, zero try-catch blocks on database repository write operations, an error handler middleware that only recognised two error categories, a virus scanner that threw generic errors indistinguishable from application crashes, and client services that discarded structured error responses in favour of generic failure messages.

**Domain Error Class Hierarchy and Middleware Overhaul (Phase 1-2):** Six new domain error classes were implemented extending a base domain error class, each carrying the appropriate HTTP status code and structured detail payloads: a validation error mapping to 400, a database error mapping to 500, a virus scanner unavailability error mapping to 503, a file-already-deleted error mapping to 410, a file download error mapping to 502, and a presigned URL generation error mapping to 502. The error handler middleware was completely rewritten from a two-handler implementation to a five-category system covering domain errors with structured JSON responses, Zod validation errors parsed into field-level issue arrays, Prisma known request errors mapped by error code (P2025 to 404, others to 500), Prisma initialisation errors returning 503, and multer file upload errors matched by code property rather than fragile string comparison. A critical Docker build compatibility issue was resolved when Prisma namespace exports behaved differently in the monorepo Docker context compared to local development — all `instanceof` checks were replaced with runtime duck-typing using `error.constructor.name`, which is decoupled from the import chain and works reliably across all build environments.

**Infrastructure and Application Layer Error Wrapping (Phase 3-6):** All seven methods in the object storage provider were wrapped with try-catch blocks following a consistent pattern: 404 responses from the cloud SDK are translated to domain file-not-found errors, while all other failures are wrapped in the appropriate domain error with the original message preserved for logging. A private helper method handles the detection of not-found errors across both the SDK's `error.name` and `error.$metadata.httpStatusCode` properties, as the cloud SDK reports errors differently depending on the operation. Five database repository write methods received error wrapping with special handling for the P2025 record-not-found case, which is elevated to a domain file-not-found error rather than a generic database error. Three generic error throws in the virus scanner service were replaced with the domain-specific unavailability error, covering unexpected scan responses, file scan timeouts, and health check timeouts. The upload use case's magic byte detection catch block was extended to handle module import failures gracefully, logging a warning and proceeding with the declared MIME type as a defence-in-depth fallback.

**Cross-Service Error Propagation and Configuration Hardening (Phase 7-8):** The identity verification service's HTTP storage client was rewritten to parse structured error responses from the storage service. A private error parsing method and a typed error response interface now map storage error codes to the appropriate domain errors: invalid file type codes map to invalid file type errors, file too large codes map to document size errors, and file-not-found or file-already-deleted codes are treated as idempotent non-fatal conditions during cleanup operations. The configuration schema's silent default was removed entirely, and the dependency injection container's default case was changed from returning a local storage adapter to throwing an explicit error. All document use cases had their silent catch blocks replaced with structured error logging, ensuring that no failure in presigned URL generation or document retrieval would be silently absorbed.

**Comprehensive Test Suite and Code Audit Remediation:** The first testing pass confirmed 113 tests passing (90 existing plus 23 new), covering all six domain error classes and 15 error handler middleware integration tests that simulate every error category including a secret-leak prevention test verifying that internal error messages never escape to API responses. A second session produced 30 additional unit tests for the identity verification service's HTTP storage client, covering every public method and edge case including owner metadata forwarding, bucket routing for face capture types versus standard documents, structured error parsing from JSON and non-JSON responses, and network failure handling. A comprehensive code audit across 10 files then identified 1 critical finding (a non-null assertion on a cloud SDK response body that could crash on empty downloads), 7 warning-level findings (including missing fetch timeouts that could hang indefinitely, 5xx errors leaking internal details, and a missing database validation error handler), and 12 informational findings. All critical and warning findings were fixed, bringing the total to 146 passing unit tests across both services. Five end-to-end Playwright tests were also created for the KYC document upload flow, covering login, navigation, file upload with synthetic test images, page load verification, and error handling with mocked failure responses.

**Admin Storage Browser Audit and Systematic Remediation:** The administrative storage browser interface underwent a deep quality pass triggered by the discovery that nothing was clickable inside the file details modal. Investigation revealed that a third-party scroll area component does not participate in CSS flex layout the way native overflow scrolling does — the component's internal wrapper layers expand to content height regardless of flex constraints, pushing action buttons off-screen and blocking pointer events. The fix replaced the component with native CSS `overflow-y-auto`, which correctly respects flex constraints. This initial bug led to a comprehensive two-agent audit: one agent reviewed all 15 storage browser component files and 3 API route handlers producing 53 findings, while a second agent performed Playwright-driven UI testing of every interactive element producing 9 findings. Seven systematic fixes were executed: standardising all three download URL patterns to use the server-side proxy endpoint instead of unreachable internal presigned URLs, correcting a misleading clipboard icon on the preview button to the correct eye icon, adding screen reader accessibility descriptions to the upload dialog, fixing broken `useCallback` memoisation where every callback was recreating on every render due to unstable state object references in dependency arrays (resolved by destructuring stable `useState` setter functions), wiring preview and delete actions into the user-grouped file view that previously had no action buttons, adding image error fallback handling in the grid view file cards, and hardening the preview dialog with PDF iframe sandboxing and a corrected zoom-out floor. A duplicated bucket path in the object storage key construction was also corrected — the bucket name was being included in both the S3 API's bucket parameter and the object key, creating a doubled path prefix on every upload. A dialog stacking bug where row clicks and dropdown menu actions fired simultaneously was resolved with a ref-based guard pattern. Twelve commits addressed the admin interface across 8 files in 2 repositories.

**KYC Document Preview Resolution — Three Bugs, Three Root Causes:** The final phase addressed broken document image previews on the KYC documents page, where uploaded images showed blank areas instead of rendered previews and the page entered an infinite re-render loop. The first bug was a classic React dependency cycle: a `useCallback` hook included state variables (`documentUrls` and `loadingUrls`) in its dependency array, and the function body called state setters that updated those same variables, which changed the callback's identity, which triggered a dependent `useEffect`, which called the function again. The fix replaced state-based guards with `useRef` equivalents that do not trigger re-renders, stabilising the callback identity with an empty dependency array. The second bug involved the `storage://` protocol prefix in file identifiers being passed through `encodeURIComponent()`, converting the colons and slashes into percent-encoded characters that broke the framework's dynamic route matching and returned 404 responses. A helper function was added to strip the prefix and extract the clean UUID before constructing API URLs. The third bug was CSS-related: saved document image labels were rendered as inline elements (the HTML default for `<label>`), and CSS `aspect-ratio` does not apply to inline elements, causing the image containers to collapse to zero width. Adding `display: block` to both front and back image labels resolved the rendering. Additional hardening included filtering out stale local filesystem paths from pre-fix uploads and gracefully handling 404 responses for files that no longer exist in object storage. Verification was performed across desktop, tablet, and mobile viewport sizes confirming correct rendering and responsive behaviour.

**Kubernetes Deployment and End-to-End Verification:** The storage service was rebuilt as a Docker image using the three-stage build pattern and deployed to the Kubernetes cluster after correcting both the container registry address and the deployment namespace. The complete KYC document upload chain was verified end-to-end at the API level: authentication, document upload through the identity verification service delegating to the storage service, object storage key structure verification confirming the duplicated bucket path fix, presigned URL generation, and file visibility in the administrative storage browser with correct metadata. The full Playwright end-to-end test suite of 5 tests passed, and the storage browser's 8 existing end-to-end tests maintained 7 passing with 1 flaky timeout unrelated to the code changes. The day concluded with 41 commits across all sessions, 146 unit tests, and 13 end-to-end tests, with the entire storage and identity verification pipeline operating under enterprise-grade exception handling.

Outcome:
The entire file storage and identity verification pipeline was transformed from a system riddled with silent fallbacks and swallowed errors into an enterprise-grade architecture with structured error propagation across every layer, validated by 146 unit tests and 13 end-to-end tests across 41 commits. The administrative storage browser was systematically audited and remediated from a 53-finding code review to a polished operational interface, and three distinct frontend rendering bugs were resolved to deliver working document image previews across all viewport sizes.
