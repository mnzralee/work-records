**User Profile Microservice Creation and Clean Architecture Implementation:** The integration test exposed a critical architectural gap: the user-facing frontend application had no dedicated backend service for profile data queries. The deprecated monolithic identity service was still the only option for serving user profile endpoints, and the wallet application's profile route was returning stale session data due to the absence of a proper backend data source. A new user profile microservice was built from scratch as a purpose-built Express.js application following the established Clean Architecture patterns across the monorepo. The service was assigned port 3049 extending the existing sequential port convention and was structured across four architectural layers with over fifteen source files. The domain layer defined a user profile entity with status, blockchain synchronisation status, KYC status, and wallet reference fields, along with trust score and address value objects. The application layer implemented four focused use cases: authenticated profile retrieval joining data across five database tables, profile field updates restricted to mutable fields only, a limited public profile view by user identifier, and a lightweight status polling endpoint for onboarding progress checks. The infrastructure layer contained five Prisma ORM repository implementations for user profiles, addresses, trust scores, wallet references, and KYC status lookups. The interface layer provided route handlers with JWT authentication middleware and request validation. The wallet frontend was updated to fetch profile data from the new service with a graceful fallback to session data if the service was unavailable, ensuring the frontend never breaks due to a backend service outage.

**Docker Build Marathon and Kubernetes Deployment Pivot:** Five components required fresh Docker images reflecting the CQRS bug fixes and the new service. The projector and administrative service images built successfully, as did the KYC service and the new user profile service. The wallet service build required a Dockerfile correction to remove a reference to a renamed package and to add the missing transitive dependency that had been causing failures across multiple services. All five images were built, tagged, and pushed to the private container registry. During Kubernetes deployment, the new user profile service crashed immediately due to a missing Redis connection environment variable in the deployment manifest, which was resolved by sourcing the value from the existing cluster secrets. Two other services crashed with the familiar transitive dependency error in their production Docker images. Rather than spending additional time rebuilding images for services under active development, the pragmatic decision was made to run actively-developed services locally using the TypeScript executor while keeping stable services in Kubernetes with port-forwarding. This hybrid approach — local execution for services under active change, containerised deployment for stable services — reduced the iteration cycle from five to ten minutes per Docker build to sub-second feedback for code changes.

**Systemic Dockerfile Audit and Transitive Dependency Resolution:** A comprehensive verification agent auditing seventy-eight checks across the entire monorepo completed its analysis, scoring seventy-four passes against four critical failures. All four failures were the same systemic bug — a shared error-handling package used as a transitive dependency of the HTTP middleware was missing from the production stages of three additional service Dockerfiles. This package is resolved during the build phase because the monorepo build tool includes the full dependency graph, but the slimmed production stage only copies explicitly listed packages. The fix required adding the package in exactly three locations per Dockerfile: the build filter for type resolution during compilation, the production copy command for runtime availability, and a symbolic link step for Node.js module resolution. The same three-location fix was applied to the administrative, authentication, and tokenomics service Dockerfiles, each committed individually. This brought the total number of services fixed for this transitive dependency issue to six across the current and preceding sessions, establishing the pattern as a known monorepo packaging concern requiring a systematic solution such as a shared Dockerfile template.

**CQRS Projector Monolith Decomposition into Enterprise Handler Pattern:** The most architecturally significant refactoring addressed the projector worker — the component responsible for subscribing to the Hyperledger Fabric blockchain event stream, processing events in order, and writing resulting state into the PostgreSQL read models that all REST APIs serve. This critical infrastructure had accumulated to 2,903 lines in a single file, with infrastructure concerns such as the gRPC subscription, distributed advisory locking, checkpoint management, Prometheus metrics, and graceful shutdown interleaved with business logic spanning thirty event handlers across ten domain areas. The decomposition extracted the infrastructure into a dedicated projector class of 440 lines owning all operational concerns, while the entry point was reduced to 41 lines of pure bootstrap. A handler registry of 159 lines maps event names to handler instances using auto-registration from domain barrel exports. Four parallel implementation agents simultaneously created handler files across ten domain subdirectories — identity, tokenomics, governance, organisation, administration, government, loan, tax, composite events, and stub handlers for recognised but unimplemented events. The composite event handler required a callback-based delegation pattern to avoid circular dependencies: sub-events are routed back through the registry without the handler importing the registry directly. A reconciliation pass after the parallel creation corrected twenty-one import path inconsistencies, four registry name mismatches, and removed three phantom handlers for events that do not yet exist in the chaincode. Twelve dead code files were cleaned up and the original monolith was archived for reference. TypeScript compilation completed with zero errors across all forty-eight new files, confirming correct wiring of every import, type, and handler registration.

**Outbox-Submitter Decomposition and Command Handler Architecture:** The same decomposition pattern was applied to the outbox-submitter — the complementary CQRS worker that maps database commands to blockchain transactions. The outbox-submitter monolith at 2,385 lines packed six distinct concerns into a single file: configuration and constants, Prometheus metrics, Fabric identity management with attribute-based access control, a thirty-five case command routing switch statement, post-commit side effects including balance synchronisation and wallet creation, and the worker infrastructure for polling, distributed locking, batch processing, and graceful shutdown. The command handler pattern was designed with a richer lifecycle than the projector's event handlers, implementing four distinct hooks: payload enrichment for pre-submission database lookups, chaincode resolution mapping commands to contract functions and serialised arguments, post-commit side effects for database updates after blockchain confirmation, and dead letter queue handling for exhausted retries. Three parallel agents created forty-one command handler files across seven domain subdirectories. A naming convention reconciliation was required as the independent agents used inconsistent suffixes — some used a command handler suffix while others used a command suffix — resolved pragmatically by updating the single registry file rather than modifying thirteen handler files. The core submitter extraction produced a 600-line file preserving all infrastructure complexity including multi-organisation multi-identity Fabric client initialisation, atomic command claiming with row-level locking and exponential backoff, and the critical blockchain idempotency check that prevents duplicate registrations after crash recovery. TypeScript compilation verified zero errors across all forty-eight new files. The combined decomposition across both sessions eliminated 5,288 lines of monolithic code, producing 96 new files with 63 focused handler classes, each testable in isolation and navigable by domain.

**Deployment Pipeline and Genesis Distribution Root Cause Resolution:** With the decomposed CQRS workers ready, Docker images were built for both workers and two additional services. The projector and outbox-submitter built cleanly on the first attempt as they do not depend on the problematic transitive dependency chain. The administrative and authentication service images also built successfully with the Dockerfile fixes applied earlier. The tokenomics service build failed due to a test utilities package compilation issue inside Docker, which was resolved by replacing the monorepo build tool invocation with explicit per-package TypeScript compilation commands. After deploying the new projector image and resetting its checkpoint to replay the critical blockchain block, two bugs emerged during the replay. The first was a database schema inconsistency where five columns defined in the ORM schema and recorded as migrated were physically absent from the database table — the columns were added via direct SQL. More critically, a transaction pattern bug was discovered where a catch-and-continue approach on a database operation within a transaction silently allowed PostgreSQL to mark the transaction as aborted, causing all subsequent operations in the same transaction to fail. This was replaced with an existence check before the update operation. The second and most significant bug was the root cause of the zero wallet balance that had persisted through all testing: the genesis distribution event handler was using the blockchain fabric user identifier to look up wallet records, but the wallet table is keyed by the platform's internal profile identifier — a UUID with no resemblance to the blockchain identifier. The update operation silently matched zero rows and committed successfully, logging a misleading success message. The fix implemented a resolution step that maps the blockchain identifier to the internal profile identifier via the user profile table before performing the wallet balance update. Both fixes were deployed in a rebuilt projector image with the checkpoint configured to replay the critical block, positioning the system to correctly distribute genesis token allocations on the next processing cycle.
