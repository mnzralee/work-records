Title: Backend Service Deployment Pipeline Overhaul, Domain-Driven Financial Arithmetic Foundation, CQRS Composite Event Architecture, and Full-Stack E2E Integration Testing

Objective:
To resolve cascading backend service deployment failures across a Kubernetes-hosted microservices platform by rewriting broken Dockerfiles, fixing TypeScript compilation errors, and correcting port-mapping misconfigurations; to eliminate dangerous floating-point arithmetic from the entire currency processing layer by implementing a BigInt-backed Amount value object and rewriting all conversion utilities with string-split integer arithmetic; to design and deploy a three-layer CQRS integrity architecture addressing a critical Hyperledger Fabric single-event-per-transaction limitation that silently discarded domain events; and to execute comprehensive end-to-end integration testing across the full user lifecycle from registration through KYC document upload, uncovering and resolving critical integration bugs along the way.

Work Carried Out:

**Backend Service Deployment and Port-Forward Script Correction:** The day began with an assessment of backend microservice health across the Kubernetes cluster, where three services had been running stale container images built over three weeks prior. An immediate bug was discovered in the port-forwarding helper script: the administrative service was mapped to the wrong port (3043 instead of the correct 3040, which belongs to a different service entirely). The port convention assigns sequential ports per service — 3040 for the administrative service, 3041 for authentication, 3042 for identity verification, and 3043 for the wallet service. Beyond the port fix, the administrative service was found in a CrashLoopBackOff state due to a fundamentally broken Dockerfile that used a single-stage build pattern expecting pre-built artifacts and pre-installed dependencies to exist before Docker build time. This pattern fails in monorepo environments where interdependent packages must be built in sequence. The Dockerfile was completely rewritten to match the proven multi-stage build pattern used by sibling services: a dependencies stage using clean install from the lockfile, a builder stage that compiles TypeScript within the full monorepo context, and a production runner stage containing only the built artefact. This rewrite also incorporated the Prisma ORM client generation step, which must execute during the build phase before the server can start.

**Docker Build Iteration Odyssey and Kubernetes Deployment Alignment:** What followed was a four-iteration Docker build cycle, each iteration exposing a distinct category of failure. The first iteration failed because both the administrative and authentication service Dockerfiles referenced a removed CLI tool package that was explicitly excluded by the Docker ignore rules. The second iteration progressed to TypeScript compilation but surfaced three separate type errors: a required database field that was never populated during user registration (resolved by making the field nullable in the Prisma schema), missing interface method signatures in a notification repository port, and test specification files being included in the production build due to absent exclusion patterns in the TypeScript configuration. The third iteration revealed a Prisma client module resolution mismatch specific to monorepo Docker builds — the ORM generates its client to a package-specific location, but TypeScript import resolution looks in the workspace root. A post-generation copy step was added to synchronise both locations. The fourth iteration built successfully but the deployed container was missing an internal package because Docker BuildKit had cached layers from an older Dockerfile structure, serving a stale 22-step production stage instead of the current 24-step version. Rebuilding with cache disabled resolved this, and the diagnostic lesson — comparing step counts in build output against the actual Dockerfile — was documented. After all three images were pushed to the private container registry and Kubernetes deployments were updated, a final port mismatch between the Kubernetes service target port (3002) and the container's actual listen port (3006) required a service patch along with readiness and liveness probe endpoint corrections. All three backend services reached healthy Running status with verified health check responses.

**Domain Foundation Package and BigInt-Backed Amount Value Object:** The most architecturally significant work of the day addressed a quietly dangerous flaw in the platform's financial arithmetic layer. The backend's currency conversion utilities processed token amounts through JavaScript's `parseFloat` and `Math.round` functions — two operations that introduce precision loss in a system designed to handle trillions of tokens with sub-unit granularity across billions of users. A new shared domain package was created as the home for domain-driven design primitives that the codebase had been entirely lacking: domain entities existed only as plain objects, aggregates were absent, and the "domain layer" across all services contained nothing but error classes. The centrepiece of this package is a 280-line Amount value object that represents all monetary values as BigInt integers in the smallest currency unit (Qirat), using a private constructor that forces all instances through validated factory methods. The critical innovation is the string-split parsing algorithm: when converting a human-readable amount like "1.5" to its internal representation, the string is split at the decimal point, each side is independently converted to BigInt, and they are combined — no floating-point intermediate, no rounding, zero precision loss. This is the same algorithm used by the Go smart contract on the blockchain layer. A denial-of-service guard limits integer digit length to 30 characters, preventing memory exhaustion from maliciously large BigInt allocations. Immutable arithmetic operations return new instances, and subtraction throws on negative results because negative amounts are invalid in a financial context. An Entity base class and AggregateRoot with domain event collection were also implemented, along with a Rust-inspired Result pattern for explicit error handling that makes failure cases visible in function signatures rather than relying on implicit exception propagation.

**Currency Utility Rewrite and Tokenomics Constants Centralisation:** With the Amount value object establishing the gold standard for new code, the existing currency utilities that all eight-plus services depend on were rewritten to eliminate every remaining floating-point operation. The GX-to-Qirat conversion function was transformed from a `parseFloat`-multiply-round-BigInt pipeline to the same string-split BigInt algorithm used in the Amount value object. The return type of the inverse Qirat-to-GX function was changed from `number` to `string`, eliminating precision loss for large values — impact analysis confirmed zero direct external call sites, making this breaking change safe. Two functions in the fee calculation service were also purged of floating-point: a display formatter that previously divided through `Number()` was rewritten with BigInt division and modulo, and a Prisma Decimal-to-Qirat converter that routed through `parseFloat` was replaced with the string-split pattern. Additionally, a comprehensive tokenomics constants file was created as a single TypeScript source of truth mirroring every constant from the 519-line Go smart contract constants file, organised by technical design specification section with explicit citations. Pool allocation sums were verified at runtime to equal the maximum supply of 1.25 trillion tokens. Frontend formatting utilities were created using pure string arithmetic (inserting a decimal point six characters from the right) to avoid BigInt requirements in the browser environment, supporting locale-aware grouping, compact notation for large values, and inverse parsing of user input back to the smallest unit representation.

**Test Suite Development and Monorepo Build Verification:** A comprehensive test suite of 168 new tests across four files validated the entire domain foundation. The Amount value object alone received 110 tests across 23 describe blocks covering every factory method, arithmetic operation, comparison, conversion, immutability guarantee, and financial scenario — including denial-of-service protection verification, precision boundary testing at the maximum safe integer limit, and compound operation pipelines simulating fee calculation and split payment workflows. The currency utility tests (67 tests) validated that the BigInt rewrite produces identical results to the old implementation for all standard inputs while correctly rejecting oversized inputs. Entity and AggregateRoot tests (45 tests) covered identity equality, timestamp tracking, domain event accumulation and clearing, and optimistic concurrency version management. Result pattern tests (52 tests) verified monadic composition, error propagation through flatMap chains, and array aggregation via combine. All 168 new tests passed, with only two pre-existing failures in unrelated services. The full monorepo build succeeded for 15 of 16 packages, the single failure being a pre-existing missing module reference in a service unrelated to the changes. Go smart contract tests were also verified with five passes and one pre-existing failure, confirming that the TypeScript constants mirror does not affect the deployed smart contract.

**Git Branch Recovery and Commit Surgery:** The implementation work had been performed on a submodule that had drifted into a detached HEAD state — thirteen implementation commits were sitting on a headless branch disconnected from the development branch, while a separate test commit existed on the actual branch. A cherry-pick strategy was chosen over rebase to preserve the correct commit ordering (implementation before tests) and avoid the complexity of interactive rebasing within a submodule context. The test commit was cherry-picked onto the detached commit chain, which automatically reattached the branch pointer because git recognised the commit chain as extending from the branch's original base. All remaining uncommitted files were then committed individually with descriptive messages following the conventional commits specification, and parent repository submodule pointers were updated in separate commits to record the new submodule states. The final tally was 18 commits in the backend submodule, 1 commit in the frontend submodule, and 2 parent repository pointer updates, all on clean branch states.

**Frontend SPA Navigation Performance Fixes and E2E User Lifecycle Testing:** Three performance antipatterns were identified and corrected across five frontend files in the wallet application. Plain HTML anchor tags were causing full HTTP page reloads instead of client-side SPA transitions, adding 2-3 seconds of blank screen time between authentication pages. These were replaced with the framework's Link component for prefetched SPA routing. A programmatic navigation call was replaced with a declarative Link, and an unnecessary client-only wrapper around the session provider was removed as it forced an extra render cycle on every page load. These changes reduced page transition times from 2-3 seconds to approximately 300-400 milliseconds. Following the frontend fixes, a comprehensive end-to-end integration test was executed across both frontend applications with parallel database verification at each step. The test walked a new user through account registration with email and phone OTP verification, KYC personal information entry, document upload for government identification, admin portal login and KYC application review, admin approval, and blockchain registration queue submission. Four critical integration bugs were discovered: a Prisma schema mismatch in the deployed identity verification service image, a Hyperledger Fabric event overwrite where the smart contract's `SetEvent` function silently discards all but the last event in multi-contract transactions, an outbox processor using a non-unique lookup that activated the wrong user's status, and a stale local development process occupying a port and masking Kubernetes connectivity.

**CQRS Three-Layer Integrity Architecture Design and Implementation:** The most critical bug — the Fabric event overwrite — required not a simple code fix but an architectural solution. Hyperledger Fabric only delivers the last `SetEvent` call per transaction, which means that when the user creation function calls a sub-function that distributes genesis tokens and emits three additional events, only the final treasury allocation event survives to the block. The event projector subscribes to the user creation event and never receives it, leaving the user permanently stuck in a pending synchronisation state. The solution was a three-layer integrity model: Layer 1 (chaincode) replaces independent `SetEvent` calls with an event accumulator pattern where sub-functions contribute events to a shared collection and only the top-level function emits a single typed composite event; Layer 2 (projector) validates incoming composite events against predefined event contracts that specify required and optional sub-events per operation type, catching malformed emissions before they corrupt the read model; Layer 3 (outbox-submitter) performs delayed post-commit reconciliation, verifying that the projector correctly updated user status, wallet existence, and balance synchronisation, and logging warnings for operational alerting if any projection was missed. Additionally, nine `findFirst` database lookups in the outbox-submitter and projector were replaced with `findUnique` using the correct unique identifier fields, and a database migration was created and applied to correct the column nullability mismatch for the biometric verification hash field.

**Deployment of CQRS Bug Fixes and E2E Test Continuation:** The CQRS fixes were deployed through parallel Docker builds for the outbox-submitter and projector workers, with the smart contract deployed via the Chaincode-as-a-Service image swap pattern on the Fabric network. Two latent TypeScript errors surfaced during the Docker builds: an implicit `any` type in an admin notification callback and a stale enum value reference where a "FROZEN" status had been renamed to "LOCKED" in a previous schema migration but not updated in the freeze-account handler. The identity verification service also required a Dockerfile fix to include a transitive dependency package that the HTTP middleware layer imports for error handling — four changes were needed across the dependency, builder, production, and symlink stages of the multi-stage build. After deploying all updated images and verifying health across all six backend services, both frontend applications, the PostgreSQL database, the Fabric peers and orderer, the smart contract pod, and both CQRS worker processes, the end-to-end integration test was resumed. A new test user was successfully registered through the full account creation flow including OTP verification, and KYC personal information was submitted and government identification documents were uploaded to both front and back sides, with the test paused at the proof of address upload stage for continuation in the next session.

Outcome:
All three backend microservices were brought from stale or crashed states to healthy running status with current code through a systematic four-iteration Docker build pipeline, a domain-driven financial arithmetic foundation was established that eliminates all floating-point currency operations across the platform with 168 new tests validating correctness, a three-layer CQRS composite event architecture was designed and deployed to resolve Hyperledger Fabric's single-event-per-transaction limitation, and end-to-end integration testing validated the user registration and KYC pipeline while uncovering and resolving four critical integration bugs. The platform is now positioned for complete user lifecycle verification from registration through blockchain onboarding once the remaining KYC document upload steps are completed.
