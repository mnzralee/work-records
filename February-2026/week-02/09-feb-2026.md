Title: Strategic Architecture Review and Infrastructure Migration Planning — Development Team Decision Session on Codebase Foundation, Repository Consolidation, Server Platform Migration, and Technical Roadmap

Objective:
To convene a comprehensive architecture review and planning session with the development team and project founder, evaluating the findings from the preceding day's forensic codebase comparison across 300+ files, making definitive strategic decisions on which codebase should serve as the foundation for all future development, planning the complete repository consolidation and organisational migration, deciding on a cross-platform server infrastructure migration from AlmaLinux to Ubuntu, and establishing the technical execution roadmap for framework upgrades, backend service restoration, domain-driven design adoption, and deployment automation improvements.

Work Carried Out:

**Codebase Comparison Findings Review and Foundation Decision:** The session opened with a thorough review of the previous day's comprehensive codebase comparison analysis that had examined over 300 files across two protocol codebases—a remote development prototype and the production platform—cataloguing 60 distinct issues across frontend, backend, and chaincode layers. The findings were unambiguous: the production platform demonstrated enterprise-grade Clean Architecture with proper four-layer separation, 50+ focused use cases, 13 framework-independent domain error classes, and Go-based smart contracts with dual-mode role-based access control and idempotency patterns. The remote prototype, in contrast, exhibited 15 critical vulnerabilities including missing access control on privileged chaincode operations, non-deterministic timestamp usage that would break peer consensus, hardcoded development credentials exposed in frontend bundles, in-memory repository storage with TODO comments, and a monolithic identity service containing 18+ modules that should have been decomposed into separate microservices. The team unanimously decided that the production platform would serve as the sole foundation for all future development. However, the prototype's stronger domain-driven design patterns—particularly its BigInt-backed Amount value object for financial arithmetic and its TDS-cited tokenomics constants—were identified as candidates for selective adoption into the production architecture, to be re-implemented following the production codebase's established Clean Architecture patterns rather than copy-pasted with their embedded issues.

**Repository Consolidation and Organisation Migration Strategy:** With the codebase foundation decision settled, the team addressed the fragmented state of the project's source code management. Six repositories were scattered across two GitHub organisations and a personal account, following inconsistent naming conventions and bound together by gitignore entries rather than proper git submodule references. A comprehensive migration plan was designed: all repositories would move to a single unified GitHub organisation with clean, descriptive two-word directory names that clearly communicate each component's purpose—the admin portal, backend core, user wallet, blockchain fabric, infrastructure operations, and the marketing website. The parent monorepo would graduate from gitignore-based directory isolation to proper git submodules, establishing formal dependency tracking and enabling independent versioning of each component. A five-track parallel execution strategy was outlined to manage the 300+ file changes required across all repositories: a sequential foundation track for physical directory renames and git infrastructure changes, followed by four parallel content tracks handling core configuration, skill definitions, documentation, and sub-repository internals. The team also designed a branch architecture to support multi-developer collaboration: a main branch for production-ready code, a development integration branch, and individual developer branches providing isolated working environments that converge through the integration branch before reaching production.

**Server Infrastructure Platform Migration Decision:** The discussion turned to a strategic infrastructure decision that had been developing since the initial backup and migration planning conducted earlier in the week. The development and test environments had been running on AlmaLinux—a RHEL-compatible enterprise distribution—since the project's inception, but recurring friction points had accumulated: package availability issues where many Node.js and blockchain tools had better support on Debian-based systems, community resources overwhelmingly assuming Ubuntu/Debian environments, developer familiarity favouring apt-based package management, and compatibility issues with certain npm packages and build tools on RHEL-based systems. After evaluating the trade-offs, the team decided on a complete infrastructure overhaul: all servers across the DevNet, TestNet, and MainNet environments would be formatted and rebuilt from scratch on Ubuntu LTS. This was deliberately chosen as a clean-slate rebuild rather than an in-place upgrade, as the format-and-reinstall approach eliminates accumulated configuration drift and legacy cruft that inevitably accumulates over months of iterative infrastructure changes. The 102GB infrastructure backup completed on the previous Thursday provided the safety net enabling this decision, with database dumps, Fabric cryptographic materials, and application secrets preserved for selective restoration. DevNet was designated as the first rebuild target to validate the process with lowest risk before progressing to TestNet and ultimately MainNet.

**Frontend Framework Upgrade Planning:** The team reviewed the current frontend technology stack and planned a major framework upgrade for the marketing website, which would serve as the pilot project for modernisation before applying similar upgrades to the admin portal and wallet application. The upgrade scope encompassed Next.js 15 to 16, React 19.1 to 19.2, and a comprehensive Tailwind CSS migration from v3 class syntax to v4. The Tailwind v4 migration was identified as particularly labour-intensive, requiring systematic replacement of deprecated utility class names across potentially hundreds of files—flex-grow to grow, flex-shrink to shrink, and numerous other renamed utilities. A known hero section loading deficiency was also prioritised for resolution during this upgrade window, as the current three-stage visual sequence of white flash, fallback image, then video created a poor first impression. The team agreed that the framework upgrade and deployment would be followed immediately by CI/CD pipeline activation through GitHub Actions, establishing automated deployment so that future changes would flow automatically to production without manual intervention.

**Backend Service Restoration and Domain Foundation Roadmap:** The team assessed the state of the backend microservices running in the Kubernetes cluster, where three core services had been operating on stale container images built weeks prior and one service was in a persistent crash loop. The restoration plan was outlined in phases: first, fix the broken Dockerfiles using the proven multi-stage build pattern already established in the sibling services; second, resolve the cascading TypeScript compilation errors and ORM client generation issues that had accumulated; third, push fresh images to the private container registry and update Kubernetes deployments with corrected port mappings and health check configurations. Beyond service restoration, the team planned a foundational architectural intervention: the introduction of a shared domain package implementing domain-driven design primitives that the codebase had been entirely lacking. The centrepiece would be a BigInt-backed Amount value object that eliminates the dangerous floating-point arithmetic currently used throughout the currency processing layer—where parseFloat and Math.round operations introduce precision loss in a system handling trillions of tokens. The team also discussed the Hyperledger Fabric single-event-per-transaction limitation that had been silently discarding domain events during blockchain registration, and outlined a three-layer CQRS integrity architecture to address it: a composite event accumulator in the outbox submitter, a projection layer for read model updates, and a reconciliation safety net for eventual consistency verification.

**Execution Timeline and Risk Assessment:** The team established a two-day execution window for the planned changes: the repository migration, framework upgrades, production deployment, and CI/CD setup would be executed on the first day, followed by backend service restoration, domain foundation implementation, CQRS architecture deployment, and comprehensive end-to-end integration testing on the second day. Risk mitigation measures were defined for each workstream: the repository migration would be executed in a specific track order to prevent file path conflicts, the framework upgrade would begin with the marketing site as a lower-risk pilot, the backend restoration would proceed service-by-service with health check verification at each stage, and the domain foundation would include comprehensive test coverage to prevent regressions. The 102GB cloud backup and the codebase comparison reports were identified as the two reference artefacts that would inform decision-making throughout the execution phase, ensuring that architectural choices aligned with the forensic analysis findings.

Outcome:
The strategic planning session produced definitive decisions across five workstreams: the production platform was confirmed as the sole development foundation, a comprehensive repository consolidation with submodule architecture was designed, the full infrastructure migration from AlmaLinux to Ubuntu was approved, frontend framework upgrades and CI/CD activation were scoped, and a backend restoration roadmap including domain-driven design adoption and CQRS architecture was outlined. These decisions established the technical execution plan for the following two days of intensive implementation work.
