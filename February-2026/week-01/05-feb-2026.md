Title: Blockchain Registration Sub-Tab Visibility, Biometric Hash Validation Pipeline, Full Infrastructure Backup with Cloud Verification, and Cross-Platform Migration Planning

Objective:
To enhance the administrative dashboard's blockchain registration workflow with granular status sub-tabs for pending on-chain users, close the remaining biometric hash validation gap that allowed invalid placeholder values to reach the chaincode, execute a comprehensive 102GB infrastructure backup across all Kubernetes namespaces and databases with byte-verified cloud upload, reclaim 242GB of accumulated disk space, and initiate the strategic planning phase for a complete infrastructure migration from AlmaLinux to Ubuntu across all server environments.

Work Carried Out:

**Pending Onchain Registration Status Sub-Tabs:** The administrative dashboard's Pending Onchain tab previously displayed all users awaiting blockchain registration as a single undifferentiated list, offering no visibility into where each user sat within the registration pipeline. Three nested sub-tabs were implemented within the Pending Onchain section to segment users by their blockchain synchronisation status: a Ready tab for users with onchainStatus NOT_SYNCED who are queued for batch registration, a Processing tab for users with onchainStatus SYNC_PENDING whose registration transactions are currently in flight, and a Failed tab for users with onchainStatus SYNC_FAILED whose registration attempts have exhausted retries and require administrative intervention. Each sub-tab displays a badge count reflecting the current population, enabling administrators to monitor registration pipeline throughput at a glance and identify bottlenecks without navigating to separate pages.

**Biometric Hash Validation and Fallback Generation Pipeline:** Despite the previous session's implementation of real SHA-256 biometric hash generation from face descriptors, a validation gap remained in the fallback path. When the primary face-api.js biometric processing failed, the UpdateFaceVerificationUseCase accepted whatever hash value the frontend provided without validation—including development bypass strings such as "dev-bypass-straight" and placeholder values like "pending_face_UUID" that did not conform to the chaincode's strict 64-character hexadecimal format requirement. Two validation functions were introduced: an isValidBiometricHash function using the regex pattern `/^[0-9a-f]{64}$/i` to verify conformance to SHA-256 hex format, and a generateFallbackBiometricHash function that produces a deterministic SHA-256 hash from a combination of the application identifier, current timestamp, and 32 bytes of cryptographic randomness. The fallback method in the use case was modified to validate any provided hash before acceptance, generating a proper fallback hash when validation fails. Separately, the placeholder hash generation in the authentication service's user registration repository—which had been creating "pending_face_UUID" strings at account creation time—was removed entirely, establishing that users now begin with a null faceVerificationHash until KYC face verification completes. This ensures the complete flow produces only valid hashes: real biometric processing yields a genuine SHA-256 from face descriptors, failed processing yields a cryptographically valid fallback, and the approval pipeline's existing 64-character hex validation guard prevents any non-conforming values from reaching the blockchain.

**Comprehensive Infrastructure Backup to Cloud Storage:** A full infrastructure backup was executed across the production VPS hosting the complete distributed ledger development and test environments. The backup pipeline progressed through eight phases: pre-flight checks validating kubectl and rclone availability, Kubernetes resource export across seven namespaces (fabric-testnet, fabric-devnet, backend-testnet, backend-devnet, ingress-nginx, registry, and kube-system), database dumps for PostgreSQL across both DevNet and TestNet environments plus Redis session caches and CouchDB world state for both organisation peers, system configuration capture including firewall rules and crontab entries, and filesystem backup of four major directory trees. The filesystem phase dominated the operation—the /root directory required 30 minutes, /home required 21 minutes, and the /var/lib/rancher Kubernetes data directory consumed 68 minutes for its 86GB of container images, persistent volumes, and runtime state. Archive compression into a single tarball produced a 102GB file (108,865,566,312 bytes). During the compression phase, the SSH session timed out after approximately two hours of continuous operation. Investigation revealed the archive had completed successfully server-side, but the upload phase had not initiated due to the session termination. Multiple approaches to restart the upload as a persistent background process were attempted—nohup and setsid both failed to survive SSH session boundaries, and neither screen nor tmux was installed on the server. The solution was systemd-run, which created a transient systemd unit (gdrive-upload) to execute the rclone copy operation independently of any terminal session. The upload completed after approximately 90 minutes, and byte-count verification confirmed an exact match between the local archive and the Google Drive copy, achieving a total backup duration of approximately four and a half hours.

**Post-Backup Disk Space Reclamation:** The backup process had consumed substantial disk space, pushing VPS utilisation to 87% with only 53GB remaining free. The SSH session timeout had prevented the backup script's built-in cleanup phase from executing, leaving 204GB of staging files alongside the compressed archive. A systematic disk space audit identified multiple categories of reclaimable storage: backup staging files already verified on cloud storage, a duplicate 12GB backup in a secondary directory, accumulated npm caches across three user accounts, outdated VS Code Server and Claude Code binary versions, and Docker BuildKit cache. A three-phase cleanup removed the staging files (recovering approximately 216GB), eliminated the duplicate backup directory (recovering 12GB), and purged development caches (recovering approximately 5GB). The operation reduced disk utilisation from 87% to 27%, recovering 242GB and restoring 295GB of free space. A recommendation was documented to add pre-flight staging cleanup to the backup script to prevent future accumulation from interrupted runs.

**Cross-Platform Infrastructure Migration Strategy:** A strategic decision was made to migrate the entire server infrastructure from AlmaLinux to Ubuntu LTS, affecting all three environments: DevNet, TestNet, and MainNet. The migration rationale centred on improved package availability for the Node.js and blockchain toolchain, stronger community support with Ubuntu-first documentation and container images, greater developer familiarity with apt-based systems, and resolution of RHEL-based compatibility issues encountered with npm packages and build tools. The migration was planned as a complete infrastructure overhaul—formatting and reinstalling all servers rather than performing an in-place upgrade—structured across seven phases: backup verification, Ubuntu installation across all servers with base security configuration, infrastructure foundation including Docker and K3s cluster setup, DevNet rebuild as the lowest-risk validation environment, TestNet rebuild, MainNet rebuild with additional security hardening, and DNS cutover with 48-hour monitoring. The day's 102GB backup served as the critical safety net enabling this migration, with database dumps, Fabric cryptographic materials, application secrets, and CouchDB state exports preserved for selective restoration. DevNet was designated as the first rebuild target on VPS1, with the existing VPS4 infrastructure maintained as a rollback option until the new environment was validated.

Outcome:
The blockchain registration pipeline gained granular administrative visibility through three status sub-tabs, and the biometric hash validation gap was closed by introducing format validation and cryptographic fallback generation across both the KYC and authentication services. A complete 102GB infrastructure backup was successfully created and byte-verified on cloud storage, followed by a 242GB disk space reclamation that restored the VPS to healthy capacity. The strategic planning phase for a full AlmaLinux-to-Ubuntu infrastructure migration was completed, establishing a seven-phase rebuild approach with DevNet as the first target environment.
