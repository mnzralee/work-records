Date: Thursday, October 30, 2025 
Title: Phase 4D (External Chaincode) Implementation, 6-Hour Debugging Session, & Strategic Pivot
Objective: To execute Phase 4D of the Kubernetes migration by deploying the gxtv3 chaincode using the modern "External Builder" method. This K8s-native approach involves running the chaincode as a standalone StatefulSet within the cluster, rather than as a Docker container spawned by the peer.
Work Carried Out:
Chaincode Server Implementation: To enable the external server model, the chaincode's main.go file was significantly modified. Logic was added to detect a CHAINCODE_SERVER_ADDRESS environment variable. When present, the code now initializes a shim.ChaincodeServer (listening on 0.0.0.0:7052) instead of using the traditional chaincode.Start() function. TLS was intentionally disabled at this level to offload transport security to the Kubernetes service mesh.
POSIX Compatibility Fixes: The initial deployment failed because the peer's external builder scripts (detect, build, release) were incompatible with its POSIX sh shell. The scripts used set -euo pipefail (a bash-specific command), which caused an Illegal option -o pipefail error in the peer logs. This was resolved by modifying all three scripts to use the POSIX-compliant set -eu, which maintains the necessary error-handling.
Image Distribution Resolution: After fixing the scripts, the chaincode deployment failed with ImagePullBackOff errors. This was because the chaincode image (gxtv3-chaincode:1.0) only existed on the master node's local cache and was not accessible to other nodes in the K3s cluster. This was solved by pushing a new version (1.1) to the K3s in-cluster registry (10.43.75.195:5000) and updating the gxtv3-deployment.yaml manifest to pull from this cluster-internal registry address.
Exhaustive gRPC Handshake Debugging: With all previous issues resolved, the chaincode lifecycle (install, approve, commit) was successfully executed. The external chaincode pod started, and its logs confirmed the gRPC server was active and listening. However, the peer still failed to connect, resulting in a persistent gRPC handshake timeout. This triggered an exhaustive 6+ hour debugging session which verified:
Builder Scripts: The fixed scripts did run successfully and produced the correct build-info.json.
Chaincode Pod: The pod was healthy, and its logs confirmed the server was listening.
Network Connectivity: All K8s DNS (chaincode-service.fabric.svc.cluster.local), TCP connectivity, and Service endpoints were verified as working.
Configuration: The connection.json and CHAINCODE_ID were all correct.
Conclusion: The failure was isolated to the gRPC protocol handshake itself, not basic network connectivity. The peer could find the pod but could not complete the protocol-level connection.
Strategic Pivot & Documentation: After 6+ hours of debugging this final, unresolvable blocker, a strategic decision was made to pivot away from the external builder model to unblock the overall production migration. The project will revert to the traditional peer-hosted chaincode model (which will require a different solution, like a Docker-in-Docker sidecar). All findings, logs, hypotheses, and the 6+ issues resolved were documented in a comprehensive troubleshooting report and committed to the repository to serve as a knowledge base for any future attempts.
Outcome: Phase 4D (External Chaincode Deployment) is blocked and postponed. Despite successfully modifying the chaincode, fixing all POSIX script bugs, and solving the K8s image distribution, a final, deep-seated gRPC handshake failure could not be resolved. To maintain project velocity, a strategic pivot back to a traditional peer-hosted chaincode model has been initiated.
