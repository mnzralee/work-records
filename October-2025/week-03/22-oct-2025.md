October 22, 2025 Title: Production Network TLS Troubleshooting & Core Identity Architecture Correction
Objective: To finalize the deployment of the production Hyperledger Fabric network by methodically diagnosing and resolving a cascade of script, file permission, and critical TLS handshaking errors that were preventing successful channel creation.
Work Carried Out:
Initial Failure Analysis & Permission Debugging: The day began by re-running the production deployment. The registerEnroll-prod.sh script immediately failed with two distinct errors:
A script bug: cp: target '.../server.key' is not a directory, indicating a non-robust command for handling the generated private key.
A more severe permission denied error, specifically when the script tried to write into directories for the orderer organization.
Identified "Permission Race Condition": A deep-dive investigation revealed the root cause of the permission errors was not a simple chmod issue. It was a "permission race condition" between the host and the Docker containers. Even after setting top-level folder permissions for the sugxcoin user, the Docker daemon (running as root) was auto-creating the necessary sub-directories inside the volume mounts faster than the script could. This resulted in root-owned directories that the sugxcoin user could not write to, causing the script to fail.
First Resolution Attempt (Pre-Creation Script): To solve the race condition, a full network teardown was performed. A new script was implemented to run before any containers were started, proactively pre-creating the entire required directory structure (organizations-ca/) with the correct sugxcoin ownership. This attempt successfully resolved all permission denied errors.
Diagnosed Critical "Unknown Authority" Error: With the permission and basic script bugs fixed, the identity generation completed successfully. However, the subsequent createChannel-prod.sh script failed with a new, critical TLS error: certificate signed by unknown authority. This signaled a much deeper, architectural problem.
Pinpointed Core Architectural Flaw: The root cause was identified as a fundamental flaw in the identity generation architecture. The original design incorrectly issued TLS (communication) certificates from separate organizational CAs (e.g., ca-org1 issued Org1's TLS cert, ca-orderer issued the orderer's TLS cert). For a secure TLS connection to be established, all parties must share a common root of trust. The client (Org1 Admin) was configured to trust only the central ca-tls, but the orderer was presenting a certificate from ca-orderer—an authority the client did not know, hence "unknown authority."
Established Definitive Architectural Solution: A new, correct architecture was designed and mandated:
MSP (Signing) Identities: Will continue to be issued by their respective organizational CAs (ca-org1, ca-org2, ca-orderer). This is correct as it defines who an entity is.
TLS (Communication) Identities: Will all be issued exclusively by the central ca-tls. This ensures every single peer and orderer in the entire network shares one unified trust domain for communication, resolving the "unknown authority" error.
Executed Full & Final System Reset: To implement this new architecture and ensure no trace of the flawed identity structure remained, a "scorched-earth" reset was executed. This involved not only bringing down all containers but also deleting the persistent postgres-data Docker volume (to wipe the CA databases) and recursively deleting the entire organizations-ca directory (to remove all old crypto-material).
Re-Implemented Enrollment Script: A brand-new, architecturally-correct registerEnroll-prod.sh script was authored from scratch. This new script correctly implements the centralized TLS CA model (contacting ca-tls for all TLS certs) and incorporates fixes for all previously identified minor bugs (like the non-robust cp command).
Initiated Final Deployment Sequence: The full, corrected deployment flow was executed:
Verified full system cleanup.
Started the CA network (with its fresh, empty database).
Ran the new, correct registerEnroll-prod.sh script.
Started the main production network (peers, orderers).
Outcome: The flawed identity architecture has been fully corrected and replaced with a secure, centralized TLS model. All script bugs and permission race conditions have been resolved. The production network's prerequisite services (CAs, peers, orderers) are now running, and all cryptographic material has been generated correctly. The system is stable and staged, awaiting the final createChannel-prod.sh command to complete the network setup.
