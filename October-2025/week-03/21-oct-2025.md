October 21, 2025 — Establishment of Production-Grade Identity Infrastructure & Network Deployment Troubleshooting
Objective:
 To architect, deploy, and verify a fully operational production-grade identity management system for the GX Coin Protocol using Hyperledger Fabric Certificate Authorities (CAs), and to troubleshoot the initial deployment of the production blockchain network, ensuring complete readiness for Phase 5 network launch.

Work Carried Out
Implementation of Production-Grade Certificate Authority (CA) Cluster
 The day began with the full deployment of a multi-CA infrastructure, marking the official migration from a development-only, cryptogen-based configuration to a production-grade identity management system. Using Infrastructure as Code (IaC) principles, a new Docker Compose configuration (docker-compose-ca.yaml) was authored to orchestrate a multi-container setup consisting of five distinct Fabric CA services—Root CA, Orderer CA, Org1 CA, Org2 CA, and TLS CA—backed by a persistent PostgreSQL 14 database.
A custom automation script (init-multiple-databases.sh) was written to initialize isolated databases for each CA instance upon startup, ensuring data segregation and repeatable setup. Additional scripts (register_identities.sh and fetch_ca_certs.sh) were implemented to automate identity registration, certificate enrollment, and the secure retrieval of CA root certificates for establishing the chain of trust.
All communication between clients and CAs was secured via TLS, and .gitignore rules were configured to exclude directories containing sensitive cryptographic material (organizations-ca/). Git versioning followed strict commit hygiene, ensuring every configuration and script update was traceable and well-documented.
The milestone outcome was the successful deployment of all five CA containers, each registered, healthy, and connected to the PostgreSQL backend. The system now supports Attribute-Based Access Control (ABAC), embedding role metadata (e.g., gxc_role=gx_super_admin) within certificates, fulfilling a critical component of the network’s security model.

Production Network Deployment & Troubleshooting
 Following the CA deployment, the focus shifted to initiating the production blockchain network deployment. The task involved validating connectivity, ensuring compatibility with existing development services, and resolving infrastructure-level conflicts.
The first issue identified was port conflicts—several production containers were mapped to ports already occupied by the development network (e.g., 17051, 17050). These were resolved by systematically reassigning the production ports to a new non-conflicting range (27051, 27050) within docker-compose-production.yaml, allowing both networks to coexist for testing and verification.
A second issue arose from missing cryptographic material during channel creation. Investigation revealed that the production admin identities (Admin@org1.prod.goodness.exchange/msp) had not yet been registered with the new CA cluster. To resolve this, a dedicated production registration and enrollment script (registerEnroll-prod.sh) was authored, adapted from the development script to interface directly with the new Fabric CAs.
Finally, several file permission and access errors were encountered—root-owned CA certificates were inaccessible to the service user (sugxcoin). The fix involved adjusting container mount paths, cleaning old directories as root, and then executing scripts under the application user for consistent permissions. This procedure is now codified as the standard operational workflow.

Advanced Problem Solving & System Reset
 Midway through testing, the CAs failed to regenerate fresh certificates after cleanup, indicating stale registration entries persisted in the PostgreSQL backend. A full system reset was performed: the CA network was brought down, and the persistent database volume was deleted (docker volume rm ca_pgdata). Upon restart, the CAs re-initialized successfully, enabling a clean state for registration and enrollment.
Further debugging identified flaws in the enrollment scripts where TLS CA certificates were not correctly propagated into each node’s tlscacerts directory. A corrected version of registerEnroll-prod.sh was implemented, ensuring every generated MSP included the proper certificate chain and trust anchors.
These fixes resulted in a fully functioning CA-based network setup with consistent, verifiable identity structures. The environment is now in a clean, reproducible, and production-ready state, ready for the full deployment sequence.

Outcome
A complete, secure, and automated production-grade Certificate Authority infrastructure has been established for the GX Coin Protocol. All required network identities have been registered and enrolled with ABAC attributes, and all CA services are healthy and integrated with the persistent PostgreSQL backend. The production network environment is now configured, tested, and ready for Phase 5 deployment.
This milestone marks the successful transition from a development-only setup to a professional-grade blockchain identity architecture, forming the security backbone of the GX Coin Protocol.
