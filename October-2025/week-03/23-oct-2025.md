Date: Thursday, October 23, 2025
Title: Production Network Launch: Peer Recovery, Channel Bootstrapping, and Full Chaincode Deployment
Objective:
To achieve a fully operational state for the GX Coin Protocol production blockchain. This involved a multi-phased effort to first diagnose and recover crashed peer and CouchDB containers, then to correctly bootstrap the 5-node ordering service with a genesis block, create the gxchannel application channel, and finally to deploy and initialize the gxcoin smart contract.
Work Carried Out:
Peer & CouchDB Recovery: The day began with diagnosing a critical failure in the peer layer: the peer0.org1 container was in a crash loop. Log inspection revealed the peer was panicking with connection refused errors, indicating it could not reach its state database, couchdb0. A docker ps -a check confirmed that couchdb0 was in an Exited state. An initial restart attempt failed because the command was run from the wrong directory. This was resolved by using docker inspect couchdb0 to find the container's labels, which identified its source configuration file (.../docker-compose-production.yaml). After navigating to the correct directory, docker compose ... up -d couchdb0 was executed, followed by a docker restart for the peer. The peer's logs then confirmed a successful connection and a clean startup.
Orderer System Channel Bootstrapping: With the peer layer stable, the createChannel-prod.sh script was executed. This failed immediately with a BAD_REQUEST error, stating the orderer system channel is not defined. The root cause was identified in the docker-compose-production.yaml: all five orderer services were configured with ORDERER_GENERAL_BOOTSTRAPMETHOD=none, meaning they started without a system channel. To fix this, the generate_production_genesis.sh script was run to create the genesis.block. The docker-compose-production.yaml was then modified to change the bootstrap method to file and add volume mounts for the new genesis block to all orderer containers.
Application Channel Creation & Final Recovery: After restarting the reconfigured network, the createChannel-prod.sh script was run again. It successfully created the channel and joined peer0.org1, but then failed with a connection refused error when attempting to join peer0.org2. An investigation revealed the same issue as before: couchdb2 (the database for peer0.org2) was Exited. The database and its peer were restarted. Re-running the full create script failed (as expected, since the channel now existed). The final step was resolved by manually joining peer0.org2 to the channel using the peer channel join command after setting the correct environment variables.
Production Chaincode Deployment: A new, production-ready deployCC-prod.sh script was created by adapting the development version. This involved updating all environment variables, file paths, and MSP IDs to match the production environment. The initial packaging step failed because the chaincode directory was not a valid Go module. This was fixed by navigating to the chaincode/ directory and running go mod init gxcoin and go mod tidy to vendor the dependencies.
Chaincode Initialization: The corrected deployCC-prod.sh script then successfully executed the full chaincode lifecycle: package, install, approve, and commit. The final step was initialization. An initial invoke attempt using the common InitLedger function failed with Function not found. A review of the admin_contract.go source code revealed the correct initialization function is BootstrapSystem. A final invoke command was sent (... -c '{"function":"AdminContract:BootstrapSystem","Args":[]}'), which returned a status:200 and a payload, confirming the ledger was successfully seeded.
Outcome:
The GX Coin Protocol production blockchain network is now fully deployed, bootstrapped, and operational. All peers, orderers, and their state databases are healthy. The gxchannel is active with all four peers joined. The gxcoin chaincode has been successfully deployed and initialized, making the network ready to process transactions.
