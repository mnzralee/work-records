Title
Production Monitoring Infrastructure Completion: Grafana Dashboard Development & System Observability Enhancement
Objective
To complete the production monitoring infrastructure deployment by implementing five comprehensive Grafana dashboards for blockchain network visualization, resolving critical metrics collection issues affecting orderer monitoring, deploying host-level and cluster-state metric exporters for infrastructure observability, and producing extensive operational documentation totaling 3,700+ lines to support production deployment and incident response procedures.
Work Carried Out
Orderer Metrics Collection Resolution: A critical issue was identified preventing Prometheus from collecting metrics from the Hyperledger Fabric orderer nodes. Investigation revealed that the scrape configuration was targeting port 8443, but the orderers expose their operations endpoint on port 9443. The Prometheus scrape job configuration was corrected to use the proper port, and orderer label extraction was added from pod metadata to enable better metric organization and dashboard filtering capabilities. This fix enabled successful collection of Raft consensus metrics (cluster size, leadership changes, active nodes), blockchain height and transaction throughput metrics, and system health indicators. Verification confirmed all five orderers now report healthy status with full metrics collection operational.
Infrastructure Metrics Collection Enhancement: Two critical metric exporters were deployed to enable comprehensive infrastructure monitoring. The Prometheus node-exporter was deployed as a DaemonSet ensuring one pod per Kubernetes node for host-level system metrics collection. The deployment uses prom/node-exporter:v1.8.2 with hostNetwork, hostPID, and hostIPC enabled to access host resources, mounts /proc, /sys, and /root filesystems from the host, exposes metrics on port 9100, and implements collector exclusions for virtual filesystems (sys, proc, dev) and virtual network interfaces (veth, bridge interfaces). Resource allocation was set to 100m CPU request with 200m limit and 30Mi memory request with 100Mi limit. The exporter provides CPU utilization per core and mode (idle, user, system, iowait), memory usage metrics (total, available, buffers, cache), disk I/O statistics (read/write bytes and operations), network I/O per interface (receive/transmit bytes and errors), and filesystem usage including inode counts per mount point. The kube-state-metrics deployment was added using registry.k8s.io/kube-state-metrics:v2.13.0 to expose Kubernetes API object state as Prometheus metrics. A ServiceAccount with cluster-wide ClusterRole permissions was created granting read access to core resources (pods, nodes, services, endpoints, PVCs, PVs, namespaces), application resources (deployments, statefulsets, daemonsets, replicasets), batch resources (jobs, cronjobs), storage classes and volume attachments, networking resources (networkpolicies, ingresses), pod disruption budgets, and horizontal pod autoscalers. The deployment exposes metrics on port 8080 with telemetry on port 8081, runs as non-root user 65534 with read-only root filesystem, implements HTTP liveness probe on /healthz and readiness probe on telemetry port, and has resource allocation of 100m CPU request with 200m limit and 150Mi memory request with 300Mi limit. These metrics enable monitoring of node metadata and status, pod lifecycle state, deployment replica counts, PVC status and capacity, and container restart counts.
Prometheus Scrape Configuration Extension: The Prometheus configuration was extended with two new scrape jobs to collect the infrastructure metrics. The node-exporter job uses Kubernetes pod discovery in the monitoring namespace, selects pods with label app=node-exporter, scrapes port 9100, and extracts instance labels from node names in pod metadata. The kube-state-metrics job uses static configuration targeting kube-state-metrics.monitoring.svc.cluster.local:8080. These additions enable collection of node_cpu_seconds_total (CPU time per core and mode), node_memory_MemAvailable_bytes and node_memory_MemTotal_bytes (system memory metrics), node_filesystem_avail_bytes and node_filesystem_size_bytes (disk capacity per mount), node_network_receive_bytes_total and node_network_transmit_bytes_total (network I/O), node_disk_read_bytes_total and node_disk_written_bytes_total (disk I/O), kube_node_info (node information), kube_pod_info (pod status), kube_persistentvolumeclaim_info (PVC metadata), and kubelet_volume_stats metrics (volume usage statistics).
Database Monitoring Implementation: The official Prometheus PostgreSQL exporter was deployed using prometheuscommunity/postgres-exporter:v0.15.0 to monitor the Fabric CA database health and performance. A Kubernetes Secret was created containing the database connection string for the fabricca database, and a ConfigMap with custom query definitions was added for Fabric CA-specific tables. The exporter deployment exposes metrics on port 9187 and successfully connects to the postgres-0 StatefulSet pod via the headless service. Custom queries collect database sizes for all CA databases (ca_root, ca_orderer, ca_org1, etc.), table sizes and row counts for users, certificates, and affiliations tables, connection statistics and transaction metrics, query performance indicators (blocks read/hit, tuples operations), and replication lag monitoring. The exporter reports pg_up=1 status indicating successful operation and provides critical visibility into CA database operations for capacity planning. CouchDB exporter configuration was added as optional monitoring, with deployment manifests and comprehensive documentation provided. The implementation includes three options: third-party exporter when compatible image is available, generic JSON exporter for CouchDB stats endpoint, or log-based monitoring via Loki which is currently sufficient. The Prometheus scrape configuration was updated to use dedicated exporter services: CouchDB changed from direct port 5984 scraping to couchdb-exporter service (prepared for future deployment), and PostgreSQL updated to scrape postgres-exporter service in monitoring namespace instead of searching for postgres pods in fabric namespace.
Grafana Dashboard Infrastructure Implementation: Five comprehensive Grafana dashboards were designed and implemented following industry best practices including RED methodology (Rate, Errors, Duration) for services, USE methodology (Utilization, Saturation, Errors) for resources, and the Four Golden Signals (Latency, Traffic, Errors, Saturation). Color-coding standards were established: green for healthy states (>95% success rate, <70% utilization), yellow for warning states (90-95% success rate, 70-90% utilization), and red for critical states (<90% success rate, >90% utilization). All dashboards implement inverted pyramid structure placing critical metrics at the top, utilize template variables for filtering (channel, node, namespace selection), enable 30-second auto-refresh for real-time monitoring, and provide mobile-responsive layouts with proper grid positioning.
Blockchain Overview Dashboard (11 Panels): The first dashboard provides high-level network health monitoring with health status panels showing orderers online count with color-coded status, peers online count with availability indicators, current block height with trend sparkline, and block production rate visualization. Network synchronization panels display blockchain height across all nodes (orderers and peers combined) for visual synchronization verification and transaction throughput with per-peer breakdown to identify performance disparities. Raft consensus health panels include cluster size gauge (target: 5 orderers), active nodes gauge with health thresholds, and leadership changes tracking for stability assessment. Performance metrics panels show endorsement success rate (target: >95%) and block commit latency p95 (target: <2 seconds). This dashboard enables operators to immediately assess network health and identify issues requiring deeper investigation.
Orderer Performance Dashboard (14 Panels Across 3 Rows): The second dashboard provides detailed orderer metrics organized into three functional rows. Row 1 contains six Raft consensus health panels: current Raft leader identification showing which orderer holds leadership, active nodes gauge with fault tolerance visualization indicating quorum health, leadership changes tracking over 1-hour window to detect instability, Raft term progression showing increases with elections, snapshot index tracking log compaction indicators, and committed block number per orderer for synchronization verification. Row 2 contains four block production and performance panels: blockchain height per orderer providing synchronization view across all orderers, block production rate measured in blocks per minute, gRPC request rate showing incoming traffic volume, and broadcast queue length indicating pending transactions awaiting ordering. Row 3 contains four resource utilization panels: CPU usage per orderer with thresholds at 70% and 90%, memory usage tracking with thresholds at 1.5Gi and 1.75Gi, Go goroutines count for concurrency monitoring, and open file descriptors for system resource tracking. The dashboard uses gauges for current state metrics, time series graphs for trend analysis, color-coded thresholds for at-a-glance status assessment, table legends showing last/mean/max values, and consistent 5-minute rate windows for metric stability.
Peer Performance Dashboard (10 Panels): The third dashboard covers endorsement operations, ledger performance, gossip protocol health, and resource utilization. Panels include endorsement success rate gauge with color-coded thresholds, proposal rate and failure tracking for transaction processing visibility, per-peer blockchain height monitoring to identify synchronization lag, block processing time percentiles (p95) for latency analysis, gossip protocol health metrics (known peers count, message rates), transaction throughput per peer for load distribution analysis, and CPU and memory usage graphs for resource consumption tracking. The dashboard follows RED methodology and implements the inverted pyramid structure with 30-second refresh intervals and channel template variable support.
System Resources Dashboard (16 Panels Across 5 Rows): The fourth dashboard provides cluster-wide resource monitoring covering node health, pod resource consumption, storage utilization, and network I/O. The dashboard includes cluster overview statistics (nodes online count, pods running count), CPU, memory, and disk usage gauges with color-coded thresholds, per-node resource utilization time series for trend analysis, top 10 pod CPU consumers table for identifying resource-intensive workloads, top 10 pod memory consumers table for memory usage analysis, network I/O graphs showing receive and transmit rates, disk I/O metrics tracking read and write operations, and PVC storage usage table with color-coded thresholds (green <70%, yellow 70-90%, red >90%). The dashboard includes pod restart count monitoring to identify unstable containers and utilizes node and namespace template variables for filtering capabilities.
Alert Dashboard (12 Panels): The fifth dashboard provides comprehensive alert visibility and alerting system health monitoring. Panels include active critical alert count with red background for immediate attention, active warning alert count with yellow background, total alerts defined count showing rule coverage, AlertManager status statistics, pie chart showing alert distribution by severity for pattern identification, bar gauge displaying alerts by component (orderers, peers, couchdb, system), alert firing rate time series over 6-hour window for trend analysis, current active alerts table with severity color-coding and description, component-specific alert graphs for orderers and peers showing historical patterns, alert duration tracking table with thresholds to identify long-running alerts, and alert rule evaluation performance metrics to ensure alerting system health. The dashboard enables operations teams to identify alert patterns and respond efficiently to incidents.
Dashboard Provisioning Infrastructure: A Grafana dashboards ConfigMap was created in the monitoring namespace containing JSON definitions for all five dashboards totaling approximately 105KB. The Grafana deployment was updated to mount this ConfigMap at /var/lib/grafana/dashboards enabling automatic provisioning. Dashboard provisioning configuration points to this directory with 30-second update interval and allows user modifications while maintaining version control. This infrastructure-as-code approach ensures consistent dashboard deployment across environments, enables version-controlled dashboard definitions, eliminates manual dashboard import requirements, and simplifies dashboard updates via ConfigMap changes.
Grafana Datasource Configuration Resolution: A critical issue was resolved where dashboards displayed "No data" despite operational metrics collection. Investigation revealed that dashboard JSON files referenced datasource UIDs ('prometheus' and 'loki') but the auto-generated UIDs in Grafana were different. The datasource provisioning configuration was updated to include explicit uid fields: Prometheus datasource uid set to 'prometheus' and Loki datasource uid set to 'loki'. This ensures dashboard datasource references match the provisioned datasources, allowing panels to query metrics successfully. Verification confirmed query API returning data with 5 orderers, 4 peers, block height 26, and 120 data points over 1-hour window.

AlertManager Notification Channel Configuration: The AlertManager configuration was enhanced with comprehensive notification routing and multi-channel support. Enhanced routing logic implements severity-based rules (critical alerts with 10-second group wait and 1-hour repeat interval, warning alerts with 5-minute group wait and 4-hour repeat interval) and component-specific routing for orderer-team and peer-team notifications. Advanced inhibition rules prevent alert storms by having critical alerts suppress warnings for the same alertname and instance. Four notification channels were pre-configured but commented for security: email with SMTP support and HTML templates for Gmail and Office 365, Slack with webhook integration and color-coded messages, generic webhook for custom integrations, and PagerDuty for on-call escalation of critical incidents. Comprehensive setup documentation (NOTIFICATION_SETUP.md) provides complete setup guide for all channels, security best practices using Kubernetes Secrets, testing procedures with sample alerts, troubleshooting common issues, and production recommendations. All channels are ready for immediate activation following documented procedures.
