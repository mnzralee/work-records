Blockchain Architecture Refactoring: User/Organization Account Model Restructuring & Breaking API Standardization
Objective
To execute a comprehensive architectural refactoring of the blockchain chaincode to resolve fundamental design inconsistencies in the account model by enforcing a strict one-human-one-identity principle, implementing breaking changes to core API functions (CreateUser and ProposeOrganization), hardening the system with extensive input validation across all contract functions, and producing detailed technical documentation to support the new three-tier account architecture.
Work Carried Out
Core Architectural Refactor (User vs. Organization Distinction): A critical architectural flaw was identified where the User data structure permitted multiple account type classifications (INDIVIDUAL, BUSINESS, GOVERNMENT, INSTITUTIONAL), which contradicted the foundational principle that user accounts should exclusively represent individual human identities verified through KYC and biometric authentication processes. The AccountType field was completely removed from the User struct, establishing that all users are implicitly individuals with no type distinction required. This architectural correction eliminated the contradiction where business entities could be incorrectly represented as user accounts rather than as distinct organizational constructs. The Organization struct retained its AccountType field, which was repurposed to distinguish between three explicit categories: BUSINESS for for-profit companies, NGO for non-profit organizations, and GOVERNMENT for country treasury accounts managed by ministry officials as stakeholders.
Breaking API Change 1 (CreateUser Function): The CreateUser function signature was reduced from five parameters to four by removing the accountType parameter entirely, as users are now definitionally individuals. Old signature: CreateUser(ctx, userID, biometricHash, nationality, age, accountType). New signature: CreateUser(ctx, userID, biometricHash, nationality, age). This change forces proper architectural separation between personal and organizational accounts at the API level, preventing the creation of "business users" which represented an architectural contradiction. Comprehensive documentation was added explaining that business account creation now requires a two-step process: first creating individual User identities via CreateUser, then calling ProposeOrganization with stakeholder endorsement.
Breaking API Change 2 (ProposeOrganization Function): The ProposeOrganization function signature was expanded from three parameters to four by adding an explicit orgType parameter. Old signature: ProposeOrganization(ctx, orgID, orgName, stakeholderIDsJSON). New signature: ProposeOrganization(ctx, orgID, orgName, orgType, stakeholderIDsJSON). The orgType parameter accepts three values: BUSINESS for commercial entities, NGO for non-profit organizations, or GOVERNMENT for country treasuries. An authorization check was implemented restricting GOVERNMENT organization creation to admin roles only, ensuring treasury accounts can only be established through proper administrative channels with ministry officials registered as User stakeholders.
Constants Reorganization and Backward Compatibility: All legacy AccountType constants (AccountTypeIndividual, AccountTypeBusiness, AccountTypeGovernment, AccountTypeInstitutional) were deprecated and replaced with clearer naming conventions. New OrgType constants were introduced (OrgTypeBusiness, OrgTypeNGO, OrgTypeGovernment) for organizational classification. A new SystemPoolTypeInstitutional constant was added to identify the six pre-minted system pools that are exempt from velocity tax. Backward compatibility aliases were maintained (UserAllocationsTotal = UserGenesisPoolTotal, GovernmentTreasuryTotal = GovtGenesisPoolTotal) to prevent immediate breakage of existing code references. Genesis pool constants were reorganized with new system account IDs: SYSTEM_USER_GENESIS_POOL for the 577.5B user allocation pool and SYSTEM_GOVT_GENESIS_POOL for the 152B government treasury pool.
Velocity Tax Logic Overhaul: The ApplyVelocityTax function underwent complete refactoring to align with the new architecture, eliminating complex conditional branching based on AccountType fields. The new implementation follows a straightforward three-tier detection model: User accounts unmarshal as User structs and always return accountType "INDIVIDUAL" with velocity tax applied by default; Organization accounts unmarshal as Organization structs and return their actual AccountType (BUSINESS, NGO, or GOVERNMENT) with all three types subject to velocity tax; System Pool accounts are identified by checking for the "SYSTEM_" prefix in account IDs and return an exemption error. This architectural decision ensures government treasuries remain subject to the anti-hoarding mechanism (threshold ≥100 GX held for 360 consecutive days, tax rate 3-7% with 50% distributed to government treasury and 50% to redistribution pool) to encourage active fund circulation. The _startVelocityTaxTimer and _stopVelocityTaxTimer helper functions were similarly simplified, removing all AccountType conditional checks and implementing straightforward timer management: if User found, manage timer; if Organization found, manage timer; if System Pool, no timer needed.
Comprehensive Input Validation Implementation: Extensive validation logic was added across multiple contract functions to prevent malformed data from entering the blockchain state. CreateUser now validates biometric hash format (must be 64-character SHA-256 hexadecimal string), normalizes nationality codes to uppercase two-letter ISO 3166-1 alpha-2 format, and maintains existing age validation (0-150 range). ProposeOrganization performs multiple integrity checks: validates organization name length (3-100 characters minimum and maximum), uses CouchDB rich queries to prevent duplicate organization names through case-sensitive matching, implements map-based O(n) complexity duplicate detection for stakeholder IDs to prevent the same user being added multiple times, and verifies all stakeholders have existing User accounts before organization creation. SubmitProposal validates that targetParam and newValue are non-empty, performs GetState() checks to verify parameters exist before allowing change proposals (preventing proposals for non-existent system parameters), and uses rich queries to prevent duplicate active proposals with identical targetParam and newValue combinations. DefineAuthRule validates that requiredSignatures is greater than zero and does not exceed stakeholder count to prevent impossible authorization requirements.
High-Precision Country Allocation Calculations: A new roundToInt64() helper function was implemented using math.Round() with banker's rounding (round half to even) to prevent floating-point truncation errors in percentage-based calculations for country-wise genesis distribution. This function was immediately applied to InitializeCountryData for all six genesis distribution phase allocations (Phase1UserLimit through Phase6UserLimit), ensuring countries with long decimal percentages receive accurate user allocations without systematic rounding bias. For example, India's allocation of 13.464767% multiplied by 100 million users yields exactly 13,464,767 users rather than 13 users from naive int64 casting. Go's float64 provides 15-17 decimal digits of precision, exceeding the six decimal places required for fair proportional distribution across all countries.
Tokenomics Model Simplification: The TotalCoinsMinted field was removed from the GlobalCounters struct as it was redundant under the pre-minted supply model where all 1.25 trillion GX coins are created during BootstrapSystem and distributed from genesis pools rather than minted on-demand. The struct now exclusively tracks distribution progress through phase user counts. The CalculateTransactionFee function documentation was updated to clarify that fees are transferred from sender and receiver accounts to the fee pool rather than minting new coins, maintaining the free transfer threshold for transactions ≤3 GX and applying 0.05% fees to each side (0.1% total effective rate) for transfers exceeding the threshold with a minimum fee of 1 Qirat to prevent zero-fee rounding.
Project-Wide Currency Name Standardization: A comprehensive find-and-replace operation was executed across the entire codebase to correct the currency name from "GXT" to "GX" (the official nomenclature). This standardization touched multiple source files including tokenomics_contract.go (lines 12, 18, 286, 360, 367), loan_pool_contract.go (line 24), and all major documentation files (ARCHITECTURE.md covering system overview, tokenomics architecture, state database schema, economic model, and data flow diagrams; CHAINCODE_API_REFERENCE.md covering all 38 functions across 7 contracts; CLAUDE.md covering project overview, development commands, architecture, and troubleshooting examples), ensuring consistency across 1000+ lines of documentation and code comments.
Extensive Documentation Development: The architectural refactoring was supported by a substantial documentation initiative comprising approximately 60-70% of the day's work. A new ACCOUNT_ARCHITECTURE.md document was authored as the canonical reference for the three-tier account model, containing 10 major sections: overview explaining the one-human-one-identity principle, detailed account type specifications (User for individual identity with KYC and biometric authentication, Organization for multi-stakeholder entities with three types and endorsement processes, System Pool for six pre-minted institutional funds with admin-only management), velocity tax classification rules (all Users and Organizations subject, System Pools exempt), architectural rules with five anti-patterns and three correct patterns, API integration examples with JavaScript code snippets for creating individual users, business accounts through multi-step flows, and government treasuries through admin-only flows, database schema implications for both PostgreSQL off-chain KYC data and Hyperledger Fabric on-chain state data, benefits explanation covering regulatory compliance and security, migration guidance from the old to new architecture with breaking changes enumeration, and a comprehensive FAQ addressing common questions about account creation and management. A new 100+ page comprehensive technical document was created covering the complete GX Coin system: executive summary with key statistics, system architecture (multi-contract design, network topology, CouchDB state management), complete specifications for all 7 contracts (AdminContract with 6 functions, IdentityContract with 5 functions, TokenomicsContract with 11 functions, OrganizationContract with 7 functions, LoanPoolContract with 3 functions, GovernanceContract with 6 functions, TaxAndFeeContract with 2 functions), data models for 9 core structures, economic model and tokenomics (1.25 trillion immutable supply, 6-phase distribution for 5 billion users, velocity tax and transaction fee mechanisms), security and access control (ABAC with 4-tier role hierarchy), complete API reference with invocation examples, testing and validation procedures, deployment and operations guides, and 5 comprehensive appendices (file structure, constants catalog with 50+ configuration parameters, event catalog with 18 event types, troubleshooting guide, and technical glossary). Outcome
The chaincode architecture now enforces the one-human-one-identity principle at the structural level through type design rather than runtime validation alone, with two breaking API changes successfully implemented (CreateUser 5→4 parameters, ProposeOrganization 3→4 parameters) that significantly simplify core velocity tax logic while maintaining economic policy integrity. The system has been hardened with comprehensive input validation preventing malformed data across all critical contract functions, and the entire project is supported by 150+ pages of synchronized technical documentation providing migration guidance for existing integrations and serving as the canonical reference for the three-tier account architecture.
