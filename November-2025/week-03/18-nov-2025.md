### Prisma Client Docker Container Resolution

Critical Prisma client initialization issue was identified and systematically resolved across all API services. The error manifested as MODULE_NOT_FOUND for @prisma/client despite successful builds, indicating production container missing generated Prisma client that exists only in builder stage node_modules.

Root cause analysis determined that monorepo architecture with Prisma schema in packages/core-db generates client to node_modules/.prisma within that package, but production stage npm ci --only=production excludes dev dependencies and does not regenerate client. The builder stage properly generates client during npm ci via postinstall hook, but production stage receives fresh node_modules without generated client.

Resolution strategy implemented explicit Prisma client generation in production Docker stage after npm ci completes. The fix added three commands to each service Dockerfile production stage: first copying Prisma schema to production container, then running npx prisma generate --schema=./packages/core-db/prisma/schema.prisma to generate client in production node_modules, and finally copying generated client to root node_modules for proper module resolution in monorepo structure.

svc-identity Dockerfile fix implemented in production stage adding RUN cd packages/core-db && npx prisma generate after npm ci --only=production, then RUN cp -r packages/core-db/node_modules/.prisma node_modules/ to copy generated client to root node_modules where application imports resolve. The fix ensures Prisma client properly initialized before application start command executes.

svc-admin Dockerfile fix applied identical pattern ensuring Prisma client generation in production stage. The consistency across services ensures predictable behavior and simplifies debugging when issues arise. svc-tokenomics Dockerfile fix completed the pattern for all deployed API services.

Docker image rebuild and redeployment followed systematic process. Images rebuilt with docker build -t gx-svc-identity:v2.0.14 targeting each service Dockerfile. Distribution across cluster nodes executed via docker save piped to kubectl exec for import on each control-plane node. Deployment rollout triggered via kubectl set image updating deployment to new image version. Pod verification confirmed all services transitioning from CrashLoopBackOff to Running status with Prisma client properly initializing.

Technical learning documented that monorepo structures with shared Prisma packages require explicit client generation in production Docker stages rather than relying on builder stage artifacts or postinstall hooks. The node_modules structure differs between builder (full dependencies) and production (production only), requiring regeneration after production npm install completes.

### JWT Authentication Configuration & Testing

JWT authentication was configured and validated for testnet API access enabling programmatic testing of protected endpoints. The configuration established proper token generation, validation middleware operation, and role-based access control enforcement.

JWT_SECRET configuration added 256-bit secret to backend-testnet secrets using openssl rand -base64 32 generating cryptographically secure random value. The secret stored as Kubernetes Secret and mounted as environment variable in all API service deployments. Secret rotation strategy documented with recommendation for ninety-day rotation cycle in production deployment.

Token generation testing validated JWT structure and signature verification. Test token generated with payload containing sub (subject/userId), role (user role for RBAC), iat (issued at timestamp), and exp (expiration timestamp). Token structure verified as three dot-separated base64url-encoded segments representing header (algorithm and type), payload (claims), and signature (HMAC-SHA256 using JWT_SECRET).

Role field requirement discovered during testing where initial tokens with only sub claim resulted in authorization failures. Investigation revealed requireAdmin and requireSuperAdmin middleware expecting role field in token payload. Token generation updated to include role: "SUPER_ADMIN" for administrative testing and role: "USER" for standard user testing. Successful authentication confirmed with proper 200 OK responses from protected endpoints.

Authentication middleware verification confirmed proper token validation flow including signature verification using JWT_SECRET, expiration check rejecting expired tokens with 401 Unauthorized, role extraction for RBAC middleware consumption, and user ID extraction for request context. Error responses properly structured with descriptive messages indicating authentication failure reason.

### Worker-Chaincode Parameter Alignment

Critical parameter mismatches between outbox-submitter worker command mappings and deployed chaincode function signatures were discovered and resolved. The mismatches caused transaction endorsement failures with chaincode returning errors about incorrect parameter counts, preventing all blockchain write operations.

CQRS end-to-end testing session initiated systematic validation of complete flow from API command creation through outbox pattern to blockchain transaction. Test script created user registration command writing to OutboxCommand table with CommandType.CREATE_USER and serialized payload. Outbox-submitter worker polling detected new command and attempted blockchain submission.

CREATE_USER parameter mismatch identified when chaincode returned error indicating expected four parameters but received three. Chaincode signature analysis from deployed gxcoinv1 contract revealed CreateUser function signature accepting userId (string), biometricHash (string), nationality (string), and age (int). Worker mapping only provided userId, countryCode (wrong field name), and accountType (deprecated field not in current chaincode).

Worker mapping correction for CREATE_USER updated outbox-submitter command handler to pass four parameters in correct order: payload.userId, payload.biometricHash, payload.nationality, and payload.age. The countryCode field renamed to nationality matching chaincode parameter name. The accountType parameter removed as current chaincode version does not accept account type for users (all users are individuals by architectural decision).

TRANSFER_TOKENS parameter mismatch identified when chaincode returned error about unknown function TransferTokens. Chaincode function analysis revealed correct function name is Transfer (not TransferTokens) in TokenomicsContract. Worker mapping updated to use correct function name Transfer with parameters fromUserId, toUserId, amount, and remark matching chaincode signature exactly.

DISTRIBUTE_GENESIS parameter mismatch identified when chaincode rejected three parameters for function expecting two. Chaincode GenesisMinting function accepts only userId and nationality parameters, not the userType parameter worker was sending. Worker mapping corrected to pass only two parameters removing userType which was legacy field from earlier chaincode version.

FREEZE_WALLET unmapped command type identified during comprehensive audit. The FREEZE_WALLET command type exists in worker but corresponding chaincode function does not exist in deployed contract. Error handling added to worker throwing descriptive error when unmapped command type encountered rather than attempting blockchain submission with incorrect parameters.

Error code type conversion fix addressed database insertion failure where gRPC error codes returned as integers but OutboxCommand.errorCode database column expects string type. Fix added explicit string conversion with error.code?.toString() ensuring proper type coercion before database update.

Worker rebuild and redeployment executed with clean build process. Previous build caching caused issues where TypeScript changes not reflected in compiled output. Resolution involved deleting dist directories before build ensuring fresh compilation. Docker images built as v2.0.12, then v2.0.13 after discovering localhost connection issue in v2.0.12 caused by stale build artifacts.

Chaincode signature audit documented complete mapping between CommandType enum and chaincode contract functions. CREATE_USER maps to UserContract.CreateUser with four string parameters. TRANSFER_TOKENS maps to TokenomicsContract.Transfer with three string parameters and one int parameter. DISTRIBUTE_GENESIS maps to MintingContract.GenesisMinting with two string parameters. The audit serves as reference for future worker modifications ensuring parameter alignment maintained.

### Database Connectivity Investigation

Comprehensive investigation was conducted into database connectivity failures affecting all pods in backend-testnet namespace. The investigation systematically ruled out multiple hypotheses before identifying likely node-level networking issue on the worker node hosting testnet infrastructure.

Initial symptom presentation showed all API services failing to start with Prisma client connection timeout errors. Error logs indicated P1001 Can't reach database server at postgres.backend-testnet.svc.cluster.local:5432 despite PostgreSQL pods showing Running status and passing health checks.

Hypothesis 1 testing evaluated password URL encoding issues. PostgreSQL passwords containing special characters require URL encoding in connection strings. Test involved temporarily changing DATABASE_PASSWORD to simple alphanumeric string without special characters, updating connection URL accordingly, and redeploying services. Result: Connection still failed, ruling out URL encoding as root cause.

Hypothesis 2 testing evaluated DNS resolution failures. Service discovery via Kubernetes DNS might fail if CoreDNS not properly configured for backend-testnet namespace. Test involved modifying connection URL to use direct pod IP address (10.42.3.x) instead of DNS name, bypassing DNS resolution entirely. Result: Connection still failed with same timeout error, ruling out DNS resolution as root cause.

Hypothesis 3 testing evaluated Docker image version issues. Potentially corrupted or misconfigured images might have networking problems. Test involved rebuilding images from scratch with v2.0.6 tag (known working version from mainnet), then testing with v2.0.13 (latest version). Result: Both versions exhibited identical connection failures, ruling out image version as root cause.

Hypothesis 4 testing evaluated NetworkPolicy misconfiguration. Zero-trust NetworkPolicy might be blocking PostgreSQL port 5432 traffic. Test involved examining allow-backend-egress NetworkPolicy confirming explicit allow rule for TCP port 5432 to pods with label app=postgres. Additional test deployed temporary pod with network debugging tools (nicolaka/netshoot) attempting direct TCP connection to PostgreSQL port. Result: Connection timeout from debugging pod, but Fabric connectivity to fabric-testnet namespace worked perfectly.

Critical finding: Fabric connectivity works flawlessly while PostgreSQL connectivity fails. The NetworkPolicy allows both types of traffic (Fabric on ports 7050-7054, PostgreSQL on port 5432). Fabric pods run on control-plane nodes while PostgreSQL pods run on srv1117946 worker node. This asymmetry suggested node-specific networking issue rather than general network configuration problem.

PostgreSQL local connectivity test confirmed database itself functioning correctly. Direct connection from within PostgreSQL pod using psql -U gx_admin -d gx_protocol succeeded with full database access. This proved PostgreSQL accepting connections locally but failing to receive traffic from other pods, pointing to routing issue between pods on different nodes.

Root cause hypothesis formulated: srv1117946 worker node likely has CNI (Container Network Interface) or iptables routing issues preventing cross-node traffic to pods on that node. Fabric works because all Fabric pods run on control-plane nodes, never routing to the worker node. PostgreSQL fails because two of three replicas run on worker node and service might be routing to those replicas.

Temporary workaround documented using postgres-0 specific service or direct pod addressing to force connections to replica running on control-plane node, bypassing potentially problematic worker node routing. This enables API testing while infrastructure team investigates node networking.

Investigation documented extensively for handoff to infrastructure team including all tests performed, results observed, and elimination rationale. Node-level networking investigation recommended examining CNI plugin (Flannel/Calico) configuration on srv1117946, iptables rules on worker node comparing against control-plane nodes, and kubelet networking configuration for pod network routing.

### Fabric SDK Integration & TLS Server Name Override

Fabric SDK connectivity was established with implementation of TLS server name override feature enabling proper certificate validation when connecting via Kubernetes internal DNS to peers with external TLS certificates.

Connection architecture challenge arose from backend services running in backend-testnet namespace connecting to Fabric peers in fabric-testnet namespace via Kubernetes internal DNS (peer0-org1.fabric-testnet.svc.cluster.local). Peer TLS certificates issued for external FQDN (peer0.org1.prod.goodness.exchange) causing hostname mismatch during TLS handshake. gRPC client rejected connection with certificate hostname verification failure.

TLS server name override solution implemented in core-fabric package adding optional tlsServerNameOverride field to FabricConfig.grpc interface. When configured, the field value passed as grpc.ssl_target_name_override option to gRPC client, instructing TLS verification to check certificate against override hostname rather than connection hostname.

Configuration schema update in packages/core-fabric/src/config.ts added tlsServerNameOverride?: string to grpc interface allowing optional specification of certificate hostname. Environment variable support added with FABRIC_TLS_SERVER_NAME_OVERRIDE read during configuration initialization.

gRPC client options update in packages/core-fabric/src/gateway.ts conditionally adds ssl_target_name_override to client options when tlsServerNameOverride configured. The conditional application ensures no unnecessary options added when override not needed (direct external connections).

ConfigMap update for backend-testnet deployment added FABRIC_TLS_SERVER_NAME_OVERRIDE environment variable set to peer0.org1.prod.goodness.exchange matching certificate subject. This enables workers to connect via peer0-org1.fabric-testnet.svc.cluster.local while validating certificate issued for external hostname.

Fabric credentials update resolved PERMISSION_DENIED error encountered when projector worker attempted event listening. Investigation revealed Admin@org1 certificate in fabric-credentials secret had expired or been rotated. Fresh certificate extracted from fabric-testnet namespace crypto materials dated October 2025 and updated in backend-testnet secrets. Workers successfully authenticated after credential update.

TypeScript build error resolution fixed compilation failures in tsconfig.base.json preventing successful worker builds. Path configuration adjusted ensuring proper module resolution for core-fabric package updates.

Worker deployment verification confirmed both outbox-submitter and projector workers successfully connected to Fabric network after fixes. Projector event listening initiated from block 1, outbox-submitter Fabric gateway established ready for transaction submission. Logs showed stable connections without TLS errors or permission failures.

### CQRS End-to-End Testing Execution

Comprehensive CQRS end-to-end testing was executed to validate the complete command-query flow from API request through outbox pattern to blockchain transaction and back through event projection to read model.

Test architecture overview documented the CQRS flow: API service receives request → validates input → creates OutboxCommand record → returns commandId to client → outbox-submitter polls for pending commands → submits transaction to Fabric → receives transaction result → updates OutboxCommand status → Fabric emits event → projector receives event → validates against schema → updates read model → client queries read model for final state.

Database migration verification confirmed schema ready for testing with 38 tables successfully created including OutboxCommand table with columns for commandId, commandType, payload, status, retryCount, errorCode, errorMessage, and timestamps. ProcessedEvent table ready for projector tracking with eventId, blockNumber, transactionId, eventType, and payload columns.

Outbox-submitter worker verification validated polling behavior with worker querying OutboxCommand table every five seconds for records with status PENDING. Retry logic confirmed with exponential backoff (1s, 2s, 4s, 8s, 16s) on transient failures. Circuit breaker pattern verified opening after five consecutive failures and attempting reset after thirty seconds. Dead letter queue behavior confirmed moving commands to FAILED status after maximum retry attempts exceeded.

Worker resilience features tested under various failure conditions. Transient Fabric unavailability simulated by temporarily scaling down peer pods, confirming worker retries with backoff and eventually succeeds when Fabric restored. Persistent failures tested by submitting command with invalid parameters, confirming proper FAILED status assignment with descriptive error message. Recovery testing verified worker resuming normal operation after circuit breaker reset.

Blockchain bootstrap requirements discovered during testing when CREATE_USER command failed with chaincode validation error indicating country stats not initialized. Analysis revealed chaincode CreateUser function queries stats_{nationality} ledger key to track user counts per country, but key does not exist until AdminContract.BootstrapSystem executed. Bootstrap command also initializes system parameters, treasury accounts, and genesis pools required for full system operation.

Country data initialization requirement identified as prerequisite for user registration. Chaincode validates nationality parameter against initialized countries, rejecting users with unrecognized nationality codes. Country initialization requires separate AdminContract.InitializeCountry calls for each supported country before accepting user registrations.

Testing checklist created for post-bootstrap validation including successful CREATE_USER command submission and confirmation, TRANSFER_TOKENS between two test users with proper balance updates, DISTRIBUTE_GENESIS to test user with tier-based allocation, projector event processing creating UserProfile read model entries, API query returning projected user data confirming end-to-end flow, and transaction history showing all operations with proper timestamps and status.

Build process improvement documented requiring clean builds when modifying environment-dependent code. TypeScript incremental compilation caches compiled output and may not reflect all source changes, particularly when changes affect runtime-evaluated code paths. Clean build procedure established as standard practice for worker deployments: rm -rf dist && npm run build before Docker image creation.

Image distribution optimization considered for multi-node cluster. Current process using docker save and kubectl exec for each node is time-consuming (approximately thirty minutes for full cluster distribution). Alternative approaches evaluated including internal container registry (requires additional infrastructure), node-local image build (requires build tools on production nodes), and shared NFS mount for image tarballs (requires NFS server setup). Decision to continue current process for testnet with registry consideration for production deployment.
