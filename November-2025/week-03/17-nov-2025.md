### Infrastructure Distribution Analysis & Longhorn Migration Planning

A comprehensive analysis of pod distribution across the four-node K3s cluster was conducted to assess high availability implementation and identify potential single points of failure. The investigation revealed a critical architectural limitation where database pods concentrated on a single worker node despite pod anti-affinity rules, creating substantial risk for production deployment.

Pod distribution verification confirmed microservices achieving perfect geographic distribution with each HTTP service having one replica per control-plane node. The svc-identity deployment distributed across srv1089618 (Malaysia), srv1089624 (USA), and srv1092158 (Germany) as designed. The svc-admin, svc-tokenomics, svc-organization, svc-loanpool, svc-governance, and svc-tax services similarly achieved three-way distribution across control-plane nodes in three continents. This distribution ensures regional failover capabilities and low-latency access for users worldwide.

Database distribution analysis identified a critical concentration issue where PostgreSQL and Redis StatefulSets failed to achieve proper distribution despite pod anti-affinity specifications. Investigation revealed two of three PostgreSQL replicas (postgres-1 and postgres-2) scheduled on srv1117946 (testdev worker node) with only postgres-0 on a control-plane node. Similarly, Redis showed two of three replicas (redis-1 and redis-2) concentrated on srv1117946. This concentration creates a single point of failure where srv1117946 becoming unavailable would result in loss of two-thirds database capacity, violating high availability requirements.

Root cause investigation determined that local-path storage creates PersistentVolume node affinity that overrides pod anti-affinity preferences. When the local-path provisioner creates a PV, it binds that volume to the specific node where it was first provisioned. Kubernetes scheduling honors PV node affinity as a hard constraint, superseding soft pod anti-affinity preferences. The storage architecture thus determines final pod placement regardless of pod-level scheduling preferences.

The technical explanation documented the Kubernetes scheduling constraint hierarchy where hard constraints (node selectors, PV node affinity, resource requirements) always override soft constraints (pod affinity/anti-affinity preferences, topology spread constraints). Local-path storage is inherently single-node by design, creating 1-to-1 mapping between PV and node. StatefulSets with local-path storage cannot achieve true pod distribution because each pod's PVC binds to a node-specific PV. This architectural limitation makes local-path storage unsuitable for production-grade stateful workloads requiring cross-node distribution.

Enterprise solution evaluation compared three distributed storage options. Longhorn distributed block storage provides full Kubernetes CSI driver support, three-way replication across all control-plane nodes, automatic failover on node failure, built-in backup and snapshot capabilities, web UI for volume management, zero application changes required (drop-in StorageClass replacement), and deployment time of thirty to sixty minutes. Rook-Ceph offers enterprise-grade distributed storage with block, object, and filesystem support, higher complexity requiring dedicated operator, and longer deployment time of two to four hours. NFS shared storage provides simple network file system with single NFS server creating new single point of failure, no replication built-in, and lower performance compared to block storage.

A comprehensive Longhorn migration plan was created spanning four phases with detailed step-by-step procedures. Phase 1 Longhorn Installation includes deploying Longhorn operator via Helm chart, configuring three-way replication policy, creating longhorn StorageClass with proper parameters, verifying all three control-plane nodes recognized as storage nodes, and estimated duration of thirty minutes. Phase 2 PostgreSQL Migration includes backing up current database with pg_dumpall, creating new StatefulSet with Longhorn PVCs, restoring data to new deployment, validating data integrity and replication, deleting old local-path deployment, and estimated duration of two to three hours.

Phase 3 Redis Migration includes backing up Redis data with SAVE command, creating new StatefulSet with Longhorn PVCs, restoring RDB files to new deployment, validating cache functionality, deleting old local-path deployment, and estimated duration of one to two hours. Phase 4 Verification and Testing includes confirming pod distribution across all three control-plane nodes, simulating node failure to test automatic failover, validating database write performance, verifying Longhorn web UI accessibility, documenting final architecture, and estimated duration of two to three hours.

Complete rollback procedures were documented for emergency recovery including preserving original StatefulSet manifests before migration, maintaining backup snapshots of all data, keeping local-path deployments running until Longhorn fully validated, and step-by-step rollback commands for reverting to original configuration. Post-migration success criteria include three PostgreSQL replicas distributed one per control-plane node, three Redis replicas distributed one per control-plane node, Longhorn volumes showing three healthy replicas each, automatic pod rescheduling on node failure confirmed, and backend services maintaining connectivity during failover.

Cost-benefit analysis documented that Longhorn implementation requires 300Gi total storage for PostgreSQL (100Gi × 3 replicas) and 60Gi total storage for Redis (20Gi × 3 replicas), zero additional cost using existing cluster storage, expected downtime under fifteen minutes during cutover, and performance comparable to local-path with replication overhead. ### Admin Authorization & Role-Based Access Control Implementation

Comprehensive role-based access control was implemented across twelve sensitive endpoints spanning four microservices to enforce admin-only access to privileged operations. The implementation addresses a critical security vulnerability where administrative functions were accessible to any authenticated user, creating substantial risk of unauthorized system modifications, fraudulent token operations, and governance manipulation.

The requireAdmin middleware integrates with existing JWT authentication infrastructure extracting user role from validated token payload, checking role against ADMIN or SUPER_ADMIN values, returning 403 Forbidden with clear error message for insufficient privileges, and logging authorization failures for security audit trails. The requireSuperAdmin middleware implements stricter validation requiring exactly SUPER_ADMIN role for critical system operations, blocking even standard ADMIN users from highest-privilege functions, and providing defense-in-depth for system initialization and parameter modification.

### Rate Limiting Middleware Implementation

Comprehensive rate limiting middleware was implemented to protect against denial-of-service attacks, brute force authentication attempts, API abuse, and spam operations. The implementation provides configurable in-memory rate limiting with automatic cleanup, standard-compliant HTTP headers, and pre-configured limiters for different security requirements.

Core rate limiting architecture uses Map-based in-memory storage tracking request count per client per time window, automatic counter reset after window expiration, cleanup interval running every sixty seconds preventing memory leaks, and support for distributed Redis storage for multi-instance deployments (future enhancement). The storage structure maps client identifier (IP address or custom key) to request metadata including first request timestamp, current request count, and window expiration time.

Client identification strategy implements IP-based rate limiting as default using X-Forwarded-For header when behind proxy or load balancer, falling back to direct socket remote address if header absent, and supporting custom key generators for alternative identification methods. Custom key generators enable rate limiting by authenticated user ID for higher quotas, by API key for third-party integrations, by organization ID for B2B customers, and by session ID for web applications.

Request counting algorithm tracks first request timestamp as window start time, increments counter for each subsequent request within window, resets counter and window when time exceeds configured window duration, and enforces limit by comparing current count against maximum allowed requests. The sliding window approach provides smooth rate limiting without burst spikes at window boundaries compared to fixed window implementations.

HTTP header compliance implements RFC draft standard for RateLimit-* headers including RateLimit-Limit indicating maximum requests allowed per window, RateLimit-Remaining showing requests remaining in current window, RateLimit-Reset containing Unix timestamp when limit resets, and Retry-After specifying seconds to wait before retrying (included in 429 responses). Legacy X-RateLimit-* headers optionally included for backward compatibility with older clients.

Response handling for rate limit exceeded scenarios returns 429 Too Many Requests status code per RFC 6585, includes Retry-After header with seconds until window reset, provides JSON error response with detailed message and metadata, logs rate limit violation for security monitoring, and allows custom handler function for application-specific behavior. Custom handlers enable notification systems for repeat violators, temporary IP blocking for persistent abusers, and escalation to firewall rules for DDoS attacks.

Pre-configured limiter implementations provide three security tiers. The strictRateLimiter enforces five requests per minute suitable for authentication endpoints preventing brute force password attacks, registration endpoints preventing spam account creation, password reset endpoints preventing email flooding, and financial transaction submission requiring extra caution. The moderateRateLimiter enforces sixty requests per minute suitable for public API endpoints with computation costs, KYC document submission allowing retries for errors, fee calculation endpoints preventing reconnaissance attacks, and general business logic operations.

The lenientRateLimiter enforces 200 requests per minute suitable for authenticated user operations with higher trust, read-only data retrieval endpoints, health check and monitoring endpoints requiring high availability, and internal service-to-service communication. Rate limit configuration structure accepts window duration in milliseconds defining rate limit time window, max requests specifying maximum requests allowed per window, key generator function for custom client identification, skip function for conditional rate limiting exempting certain requests, and handler function for custom limit exceeded responses.

Automatic cleanup mechanism runs setInterval every sixty seconds, iterates through all client records in storage, identifies expired windows comparing current time against window expiration, deletes expired client records freeing memory, and logs cleanup statistics for monitoring. The cleanup interval balances memory efficiency against processing overhead, preventing unbounded memory growth while minimizing CPU usage.

Implementation in core-http package provides framework-agnostic middleware compatible with Express.js, Fastify, and other Node.js frameworks, proper TypeScript type definitions for configuration and interfaces, exported factory function createRateLimitMiddleware for custom configurations, and exported pre-configured instances for immediate use. The package structure enables centralized rate limiting logic shared across all seven microservices, consistent configuration and behavior, simplified testing and maintenance, and easy future migration to Redis-based distributed rate limiting.

Protected endpoint implementation applied rate limiting to four critical endpoints across two microservices. svc-identity service protected POST /auth/login endpoint with strictRateLimiter (five requests per minute) preventing credential stuffing attacks, brute force password guessing, and automated authentication attempts. The login endpoint represents primary attack surface for unauthorized access attempts requiring strictest rate limiting.

POST /users registration endpoint received strictRateLimiter (five requests per minute) preventing bot-driven spam registrations, fake account creation at scale, and resource exhaustion through unlimited user creation. Registration rate limiting maintains service availability during abuse attempts while allowing legitimate users reasonable retry attempts for form errors. POST /users/:id/kyc endpoint received moderateRateLimiter (sixty requests per minute) balancing legitimate retry needs for document upload errors against potential submission spam, allowing multiple evidence file uploads within reasonable timeframe, and preventing KYC system abuse.

svc-tax service protected POST /fees/calculate endpoint with moderateRateLimiter (sixty requests per minute) preventing reconnaissance attacks probing fee structures, computational resource exhaustion through unlimited calculations, and denial-of-service via fee calculation flooding. Fee calculation endpoint performs database queries and mathematical operations making it vulnerable to resource exhaustion attacks, while legitimate users need multiple calculations for transaction planning.

Monitoring and observability integration added structured logging for all rate limit events including client identification, current count, window expiration, and enforcement decision. Logs enable security analysis identifying persistent attackers, usage pattern analysis informing rate limit tuning, capacity planning for infrastructure scaling, and compliance reporting for audit requirements. ### svc-tax Business Logic Implementation

Complete business logic was implemented for svc-tax service including transaction fee calculation and velocity tax eligibility checking. The implementation mirrors chaincode algorithms exactly ensuring backend-blockchain consistency while leveraging database-driven configuration for dynamic parameter management.

Transaction fee calculation logic retrieves configuration parameters from SystemParameter table using compound key tenantId_paramKey with paramKey values of transactionFeeThreshold and transactionFeeBps. Default values provide fallback when parameters not seeded in database: threshold defaults to 1,000,000 Qirat (10 coins) matching chaincode initialization, and fee rate defaults to 10 basis points (0.1%) aligning with constitutional economics specification. Parameter retrieval uses proper Prisma query syntax handling decimal conversion from database storage to JavaScript number types.

Fee calculation algorithm implements exact chaincode logic where transactions below threshold incur zero fees encouraging micropayments and reducing friction for small transactions, while transactions at or above threshold incur percentage-based fee calculated as (amount × bps) ÷ 10000 converting basis points to decimal percentage. Input validation rejects negative amounts with 400 Bad Request error, ensures integer amounts representing Qirat smallest unit, and validates amount does not exceed maximum safe integer preventing overflow errors.

Calculation example demonstrates 50,000,000 Qirat transaction (500 coins) exceeding 1,000,000 Qirat threshold, thus fee applies at 10 basis points resulting in (50,000,000 × 10) ÷ 10000 equals 5,000 Qirat fee (0.05 coins, representing 0.01% effective rate). Response DTO provides comprehensive breakdown including gross amount, fee amount, net amount (gross minus fee), threshold value, fee basis points, and whether fee applies.

Structured logging captures all fee calculations with transaction identifiers, calculated amounts, parameter values, and execution timestamps supporting financial audit trails, parameter tuning analysis, usage pattern monitoring, and compliance reporting. Logs enable detection of anomalous fee patterns indicating potential parameter manipulation or calculation errors.

Velocity tax eligibility check implements multi-criteria validation spanning system pool exemption, account existence verification, and balance threshold checking. The implementation identifies architectural limitations in current database schema requiring future migration for complete velocity tax feature implementation.

System pool exemption check queries UserProfile and Organization tables searching for accounts with ID starting with SYSTEM_ prefix indicating special system accounts (SYSTEM_GENESIS_POOL, SYSTEM_LOAN_POOL, SYSTEM_TREASURY, SYSTEM_REWARDS) that are constitutionally exempt from velocity tax. The 100 coin threshold implements constitutional requirement that only accounts with substantial holdings face hoarding penalty.

### svc-identity Command Handler Implementation

Critical command handler implementations were completed for UPDATE_USER and SUBMIT_KYC operations in svc-identity service. An important architectural decision shifted these operations from CQRS outbox pattern to direct database writes based on recognition that profile metadata and KYC submissions are off-chain administrative operations not requiring blockchain consensus.

Architectural rationale analysis determined that profile metadata updates (firstName, lastName, phoneNum, identityNum) represent off-chain personal information not requiring blockchain immutability, do not affect on-chain identity or token balances, and maintain privacy by keeping personally identifiable information off public ledger. KYC verification represents administrative approval process managed entirely off-chain by compliance officers, with verification status potentially synced to blockchain but evidence remaining off-chain for privacy compliance. Only CREATE_USER requires blockchain presence establishing on-chain identity for token transactions and governance participation.

Performance implications of outbox pattern for off-chain operations include unnecessary overhead of command serialization and outbox polling, eventual consistency delay between update request and database reflection, increased database load from outbox table writes and worker queries, and degraded user experience from asynchronous response patterns. Direct database writes provide immediate consistency, simplified code paths, reduced database load, and improved user experience through synchronous responses.

UPDATE_USER implementation replaced outbox command creation with direct UserProfile.update() Prisma operation. The service method validates user existence before update preventing errors on non-existent users, supports partial updates where only provided fields are modified allowing flexible API contract, updates firstName, lastName, phoneNum, and identityNum fields as specified in DTO, returns complete updated profile in synchronous response, and logs update with user ID and modified fields for audit trails.

Controller layer changes modified response type from {commandId: string} to {profile: UserProfileDTO} returning complete resource representation, changed HTTP status from 202 Accepted to 200 OK indicating synchronous completion, added validation requiring at least one field provided preventing no-op requests, and updated logging to reflect immediate success rather than command acceptance. The 200 OK status code properly communicates that update completed successfully and response contains final state.

SUBMIT_KYC implementation replaced outbox command creation with direct KYCVerification.create() Prisma operation. The service method validates user existence and current KYC approval status preventing duplicate submissions while approved, creates KYC verification record with PENDING status awaiting administrative review, stores evidenceHash, evidenceSize, and evidenceMime for document verification, returns created KYC record in synchronous response, and logs submission with user ID and evidence details.

Controller layer changes modified response type from {commandId: string} to {kycRecord: KYCStatusDTO} returning created resource, changed HTTP status from 202 Accepted to 201 Created indicating resource creation, maintained existing authorization and validation checks ensuring user can only submit own KYC, and updated logging to use kycId instead of commandId. The 201 Created status code properly communicates new resource created and available at returned location.

API contract changes represent breaking changes for clients expecting commandId in responses. Previous async pattern returned {commandId: "uuid"} with 202 Accepted status requiring clients to poll for completion or subscribe to events. New sync pattern returns complete resource representation with 200 OK or 201 Created status providing immediate feedback. This breaking change is acceptable for Phase 5 production enhancements occurring before public API release, with documentation updated to reflect new contract.

Data flow transformation simplified architecture from previous multi-step async flow of API → OutboxCommand → Outbox-Submitter → Fabric → Event → Projector → Read Model requiring six components and eventual consistency to new direct write flow of API → PostgreSQL (UserProfile/KYCVerification) requiring single component with immediate consistency. The simplified flow reduces failure modes, improves debugging clarity, and enhances operational simplicity.

Security posture maintained all authorization checks where user can only update own profile verified through JWT token userId matching route parameter, input validation preserves sanitization and type checking, and audit logging enhanced with structured fields for compliance reporting. The direct database approach maintains security guarantees while improving performance and user experience.

Future blockchain synchronization remains possible where final KYC approval status could sync to blockchain for on-chain identity verification, profile hash could commit to blockchain for tamper evidence, and audit trail of critical changes could replicate to immutable ledger. These enhancements can layer on top of direct database implementation without reverting to full outbox pattern.
