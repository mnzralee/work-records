### Centralized Backend API Client Implementation

Backend client infrastructure created in lib/backendClient.ts totaling 305 lines implementing multi-service axios clients with automatic token management and CQRS command utilities. Service-specific axios instances created through factory pattern producing identityClient, tokenomicsClient, and adminClient each configured with appropriate base URLs and default headers. Request interceptor automatically attaches Authorization Bearer token from sessionStorage with fallback to localStorage for persistent sessions enabling seamless authenticated requests across all API calls.

Response interceptor implements automatic token refresh on 401 Unauthorized responses, extracting refresh token from cookies, calling /auth/refresh endpoint, storing new access token, and retrying original request transparently. This mechanism ensures uninterrupted user sessions even when access tokens expire during active usage. Error handling helpers created including getErrorMessage function extracting human-readable error messages from axios error responses and isNetworkError function detecting connection failures for appropriate user feedback.

CQRS command utilities implemented handling the asynchronous command processing architecture of the backend services. submitCommand function sends POST requests and handles 202 Accepted responses extracting commandId for status polling. pollCommandStatus function repeatedly queries /api/v1/commands/{id}/status endpoint at configured intervals until terminal state reached (CONFIRMED or FAILED). Combined submitAndWaitForCommand function orchestrates complete flow from submission through polling to final result, abstracting complexity from route handlers.

Command poller utilities created in lib/commandPoller.ts totaling 193 lines providing React hooks for frontend components. useCommandExecution hook manages command execution state including isLoading for overall process, isPolling for polling phase indication, commandId tracking, and error message capture. Status display helpers created including getPollingMessage generating user-friendly progress text, getStatusColor returning Tailwind CSS classes for visual feedback, and formatExecutionTime calculating duration between command creation and completion.

Registration route refactored from legacy fetch-based implementation to CQRS-aware architecture. Previous implementation used undefined INTERNAL_GXCOIN_API_URL environment variable with old /api/gxcoin/auth/register path and no async command handling. New implementation employs submitAndWaitForCommand with identityClient sending to /api/v1/auth/register endpoint, automatically polling for blockchain confirmation of user creation. Request body properly structured with fname, lname, email, password, country, and optional fields. Response handling extracts accessToken and refreshToken from confirmed command result enabling immediate session establishment.

Login route updated replacing NEXT_PUBLIC_GXCOIN_API_URL with identityClient and old /api/gxcoin/users/login path with /api/v1/auth/login. Cookie strategy enhanced from single legacy token to dual-token architecture with separate cookies for access and refresh tokens. Access token cookie configured with 15-minute expiry (maxAge: 900), HttpOnly flag preventing JavaScript access, secure flag for HTTPS-only transmission in production, and sameSite strict preventing CSRF attacks. Refresh token cookie configured with 7-day expiry enabling persistent sessions while maintaining security through HttpOnly protection.

Logout route enhanced with comprehensive cookie cleanup clearing gxcoin_access_token, gxcoin_refresh_token, and legacy gxcoin_token for backward compatibility. All cookies set with empty value and maxAge zero triggering immediate deletion. Error handling added for cookie operation failures ensuring graceful degradation. Response returns success status with redirect URL for client-side navigation to login page.

### Cookie-Based Authentication Architecture

Authentication strategy designed around HttpOnly cookies providing security advantages over localStorage token storage. Cookie configuration implements defense-in-depth with HttpOnly preventing XSS attacks through JavaScript inaccessibility, secure flag ensuring HTTPS-only transmission in production environments, sameSite strict preventing CSRF by blocking cross-origin cookie transmission, and path '/' enabling cookie access across all application routes.

Token lifecycle management creates access token with 15-minute expiry balancing security with user experience, while refresh token maintains 7-day expiry enabling persistent sessions without frequent re-authentication. Login flow sets both cookies simultaneously with appropriate expiry settings. API routes retrieve tokens using Next.js cookies() async function providing server-side cookie access in App Router architecture. Logout flow clears all token cookies ensuring complete session termination.

### Error Handling Standardization

Error response format standardized across all 15 API routes with consistent structure enabling predictable frontend handling. Success responses return { success: true, data: {...} } pattern while error responses return { success: false, error: "message" } structure. HTTP status codes mapped appropriately with 401 for authentication failures, 404 for not found resources, 400 for validation errors, and 503 for network connectivity issues.

Network error detection implemented using isNetworkError helper checking axios error codes for ECONNREFUSED, ENOTFOUND, and ETIMEDOUT indicating backend unavailability. User-friendly message "Cannot connect to backend services. Please ensure port-forwarding is active." guides developers to check infrastructure rather than debugging application code. Authentication error handling returns "Session expired. Please log in again." prompting re-authentication flow.

### OpenAPI 3.0.3 Specification Architecture Design

Comprehensive OpenAPI 3.0.3 specification implementation initiated for all seven backend microservices establishing industry-standard API documentation. Architecture designed around shared component library pattern with central shared-schemas.yaml file containing reusable schemas, parameters, responses, and security schemes referenced by all service-specific specifications. This DRY (Don't Repeat Yourself) approach ensures consistency across services while minimizing duplication and maintenance overhead.

File structure organized in openapi/ directory with eight YAML specification files totaling approximately 3,500 lines of OpenAPI documentation. shared-schemas.yaml serves as foundation containing 15 reusable components referenced via $ref links from service specifications. Service-specific files include svc-identity.yaml for authentication and user management, svc-tokenomics.yaml for token operations and transfers, svc-admin.yaml for system administration, svc-organization.yaml for multi-signature organizations, svc-loanpool.yaml for lending operations, svc-governance.yaml for proposal and voting, and svc-tax.yaml for fee calculation and velocity tax.

### Shared Components Library Implementation

Security schemes defined in shared-schemas.yaml establishing authentication patterns for entire API ecosystem. BearerAuth scheme configured as HTTP bearer authentication with JWT format and 15-minute expiry documentation. ApiKeyAuth scheme configured for service-to-service communication using X-API-Key header enabling internal API access without user authentication. All authenticated endpoints reference security schemes through standardized security requirement objects.

Common parameters library created for reusable path and query parameters. UserIdParam defines UUID validation pattern for user identifier path parameters with format validation and example values. PageParam and LimitParam implement pagination parameters with minimum/maximum constraints (limit capped at 100) and sensible defaults (page 1, limit 20). IdempotencyKeyHeader defines X-Idempotency-Key requirement for write operations enabling safe request retries with 16-128 character length constraints.

Standardized response library created mapping all HTTP status codes to consistent error formats. ErrorResponse schema defines nested error object structure with message, statusCode, optional code, and errors array for field-level validation failures. CommandAcceptedResponse schema defines 202 Accepted response structure with commandId, status enum (ACCEPTED), message, and optional aggregateId for CQRS write operations. CommandStatusResponse schema defines polling response with status progression (PENDING, PROCESSING, CONFIRMED, FAILED), result object for success, and error string for failures.

Common HTTP error responses standardized including BadRequest (400) for invalid input, Unauthorized (401) for missing or invalid authentication, Forbidden (403) for insufficient permissions, NotFound (404) for missing resources, Conflict (409) for duplicate resources, UnprocessableEntity (422) for semantically invalid requests, TooManyRequests (429) for rate limit violations, InternalServerError (500) for unexpected failures, and ServiceUnavailable (503) for temporary unavailability. Each response includes description, content type, schema reference, and realistic example.

### CQRS Pattern Documentation in OpenAPI

Every write endpoint across all services includes comprehensive CQRS flow documentation in description field. Documentation pattern explains step-by-step async processing: request validation and outbox table insertion, 202 Accepted response with commandId, outbox-submitter worker picking up command, chaincode invocation on Hyperledger Fabric, event emission from blockchain, projector worker updating PostgreSQL read model, and command status transition to CONFIRMED. Polling example included showing POST request returning commandId followed by GET /api/v1/commands/{id}/status calls until terminal state reached.

Command status lifecycle documented with state transition diagram in descriptions: PENDING (awaiting worker pickup) transitions to PROCESSING (submitted to Fabric) which transitions to either CONFIRMED (committed to blockchain) or FAILED (error occurred). Status definitions clarify each state's meaning enabling frontend developers to implement appropriate UI feedback during async operations.
