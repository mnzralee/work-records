Title
Critical Infrastructure Remediation & Production Readiness Enhancement: Certificate Authority Storage Migration and System Architecture Documentation
Objective
To resolve critical production blockers identified during system audit by migrating all five Certificate Authority instances from ephemeral emptyDir storage to persistent volumes ensuring certificate and key material preservation across pod restarts, address Phase 2 security hardening issues including network policy corrections and PostgreSQL configuration mismatches, complete chaincode architecture refactoring by extracting domain-specific types from monolithic files for improved maintainability, and produce comprehensive system documentation including backend integration guides, load testing plans, and testnet/mainnet architecture specifications to achieve 80% production readiness.
Work Carried Out
Critical Certificate Authority Storage Migration: A critical production blocker was discovered during system audit where all five Certificate Authority StatefulSets were using emptyDir for the ca-data volume, resulting in ephemeral storage that would be lost on pod restart. This created unacceptable risk for certificate and private key material preservation in production environments. Five sequential migrations were executed to resolve this issue across the entire CA cluster.
The ca-orderer StatefulSet was migrated first, removing the emptyDir volume definition and adding a volumeClaimTemplates section specifying a 5Gi PersistentVolumeClaim with ReadWriteOnce access mode using the local-path storage class. The existing volumeMount configuration at /var/hyperledger/fabric-ca-server was maintained for compatibility. StatefulSets require delete and recreate operations to modify volumeClaimTemplates due to immutable field restrictions. The PVC named ca-data-ca-orderer-0 was successfully created and bound, the CA pod was recreated with persistent storage mounted, the CA server came online listening on https://0.0.0.0:8054, the operations server initialized on port 18054, and health probes passed with pod status reaching 1/1 Ready. No data loss occurred as the previous emptyDir contained no persistent data from prior operation.
The ca-org1, ca-org2, ca-tls, and ca-root StatefulSets were migrated using identical procedures. The ca-org1 migration created PVC ca-data-ca-org1-0 with the CA server listening on port 9054 and operations on port 19054. The ca-org2 migration created PVC ca-data-ca-org2-0 with the CA server on port 10054 and operations on port 20054. The ca-tls migration created PVC ca-data-ca-tls-0 with the CA server on port 11054 and operations on port 21054. The ca-root migration created PVC ca-data-ca-root-0 with the CA server on port 7054 and operations on port 17054. All five migrations completed successfully with all pods reaching Running status and health checks passing. This migration increased total system storage from 140Gi to 165Gi across 19 PVCs (up from 14 PVCs), ensuring certificate authorities can survive pod restarts, node failures, and cluster maintenance operations without loss of critical PKI infrastructure.
Phase 2 Security Hardening Issue Resolution: Multiple critical issues emerged during Phase 2 security hardening deployment requiring immediate remediation. A comprehensive system recovery was executed resolving four major issues in 55 minutes with zero blockchain downtime. The first issue involved secrets encryption at rest deployment across all three K3s nodes for CIS Benchmark 3.1.1 compliance. The kube-apiserver was configured with AES-256 CBC encryption for etcd storage, but kubelet secret cache sync encountered timeout issues after encryption enablement requiring cache invalidation and pod restarts across the cluster.
The second issue was a PostgreSQL data directory ownership mismatch discovered after applying pod security contexts requiring non-root execution. The postgres:14-alpine container image runs as UID 70 (Alpine Linux postgres user) while the init container chown script assumed UID 999 (standard Postgres UID on Debian/Ubuntu distributions). This mismatch caused "Permission denied" errors preventing database initialization. The init container was corrected to chown the data directory to 70:70 instead of 999:999, aligning with the Alpine container's user ID. The Postgres pod successfully started reaching 1/1 Running status with database operations functional.
The third issue was CA-Postgres network connectivity blocked by incorrect label selectors in the network policy. The policy ingress rule used podSelector matching app.kubernetes.io/component: ca, but actual CA pods have the label app: ca as defined in their StatefulSet specifications. This mismatch blocked all CA-to-Postgres traffic causing all five CA pods to enter CrashLoopBackOff state. The network policy was corrected to use app: ca selector, immediately resolving connectivity issues. All five CAs transitioned to Running status with logs showing successful Postgres database connections.
The fourth issue was Prometheus metrics scraping failure from Fabric components after zero-trust network policy deployment. A new network policy named allow-prometheus-scraping was created in the fabric namespace allowing ingress from monitoring namespace pods on metrics ports 9443 (peers/orderers), 7052 (chaincode), and 18054 (CAs). The policy uses namespaceSelector to match the monitoring namespace and applies to all pods in the fabric namespace with empty podSelector. This enabled cross-namespace metrics collection resolving Grafana dashboards showing "no data" while maintaining zero-trust network security. Prometheus now successfully scrapes 25 of 30 targets with all critical components (5 orderers, 4 peers) reporting metrics. An additional correction was required for CA operations port specification, changing from 19054 to 18054 in the monitoring network policy to match actual CA pod configuration, enabling successful CA metrics scraping including database API request metrics.

Chaincode Architecture Refactoring for Maintainability: The monolithic types.go file containing 228 lines with all data structure definitions was refactored into seven domain-specific type files for improved code organization and maintainability following Go best practices for package structure.
Genesis distribution types (GlobalCounters and CountryStats structs) were extracted to genesis_types.go, grouping types that track overall genesis distribution progress across six phases and manage per-country phase allocations with population-based quotas. Identity management types (User, Relationship, Profile, and RecoverySession structs) were extracted to identity_types.go, grouping core individual identity with biometric verification and velocity tax tracking, social graph links between users with relationship types (FRIEND_OF, SIBLING_OF, PARENT_OF, CHILD_OF, SPOUSE_OF), profile DTOs combining user data with confirmed relationships, and social recovery process with guardian confirmations and time-locked execution.
Organization types (Organization, Rule, and PendingTransaction structs) were extracted to organization_types.go, grouping business/NGO/cooperative entities with stakeholder endorsements and multi-signature authorization, authorization rules based on amount thresholds defining required approvals, and pending transactions awaiting required approvals from stakeholders. Loan pool types (Loan struct) were extracted to loan_types.go containing interest-free loan structures with collateral tracking, trust score requirements, repayment status, and approval workflows.
Governance types (Proposal and SystemParameter structs) were extracted to governance_types.go, grouping on-chain governance proposals with voting tracking, quorum validation, and execution status, and configurable system parameters managed through decentralized governance processes including velocity tax rates, fee thresholds, and loan limits. Transaction types (Transaction struct) were extracted to tokenomics_types.go for recording financial events in CouchDB state database providing chaincode-level transaction tracking complementing Fabric's automatic blockchain-level immutable transaction records, enabling fast queries for user/organization transaction history with rich query support.
Common types (GenericAccount and SystemStatus structs) were extracted to common_types.go containing shared utility types used across multiple contracts for administrative operations including lightweight account structures for admin queries and DTOs for system operational state queries. After successful migration of all structs with 100% test pass rate verified, the monolithic types.go file was deleted completing the refactoring. This separation improves code maintainability by isolating domain concerns, enhances navigation with focused files of 30-60 lines each, follows Go best practices for package organization with clear file naming conventions, and simplifies onboarding for new developers with self-documenting file structure.

Transaction Tracking Architecture Documentation: Comprehensive documentation was created explaining the two-level transaction tracking system implemented in the blockchain. Level 1 blockchain-level tracking utilizes Fabric's automatic recording where all invoke operations are recorded in immutable blocks creating a permanent historical record with cryptographic integrity accessible via Fabric SDK block and transaction queries using GetBlockByNumber, GetTransactionByID, and GetBlockByTxID methods. This level provides complete audit trail, tamper-proof history, and regulatory compliance but requires block parsing and sequential scanning for queries.
Level 2 chaincode-level tracking implements explicit Transaction records in CouchDB state database enabling fast queries for user and organization transaction history, rich queries with CouchDB indexes on sender, receiver, amount, and timestamp fields, and application-specific metadata including transaction type, status, and business context. The documentation identifies current coverage gaps in bootstrap operations (DistributeGenesis, InitializeCountryData not creating Transaction records), governance actions (SubmitProposal, VoteOnProposal, ExecuteProposal not tracked), and identity events (CreateUser, EstablishRelationship not logged) for future enhancement. This two-level approach balances immutability requirements with query performance needs.

Load Testing Strategy and Infrastructure Planning: A comprehensive load testing plan was developed to validate the >1,000 TPS throughput requirement from whitepaper specification R6 before production launch. Currently no load testing infrastructure exists and performance baselines are undefined preventing SLA specification. The plan documents Hyperledger Caliper as the official testing framework providing Fabric-specific bindings, multi-worker distributed load generation, comprehensive metrics collection (throughput, latency, success rate), and HTML report generation with performance graphs.
Five test scenarios were designed covering all major chaincode operations: User Onboarding testing genesis distribution at 100 users/second with 30,000 total user creations measuring single-transaction latency and database write performance, Peer-to-Peer Transfers testing sustained 1,000 TPS for 15 minutes measuring transfer validation and balance updates with gossip protocol synchronization, Organization Multi-Signature testing 200 TPS with quorum validation measuring multi-step approval workflows and stakeholder coordination, Governance Voting testing 500 votes/second across 50 concurrent proposals measuring vote aggregation and proposal status updates, and Mixed Workload simulating production traffic with 50% transfers, 20% balance queries, 30% other operations (loans, proposals, relationships).
Five test phases were defined with progressive load increases: Phase 1 Baseline establishes latency baseline at 100 TPS for 5 minutes measuring single-transaction latency, component resource usage, and database query performance, Phase 2 Sustained Load validates target performance at 500/1000/1500 TPS for 15 minutes each measuring throughput stability and latency percentiles (P50, P95, P99), Phase 3 Stress Test finds breaking point with 1000-3000 TPS ramp over 30 minutes identifying maximum sustainable throughput, Phase 4 Endurance validates stability at 1,000 TPS for 2 hours detecting memory leaks and resource exhaustion, and Phase 5 Chaos Engineering validates fault tolerance at 1,000 TPS with random pod kills testing Raft consensus recovery, peer gossip resilience, and automatic failover capabilities.
Success criteria were defined at three levels: Must Pass requirements include ≥1,000 TPS sustained for 15 minutes, P95 latency <5 seconds, >99.9% success rate, and no pod crashes or restarts during testing; Performance Targets include 1,500+ TPS sustained, average latency <2 seconds, CPU usage <70% across all nodes, and linear scaling (2x workers = 2x throughput); Monitoring requirements include tracking 8 Caliper metrics (send rate, throughput, latency distribution, success rate, transaction validation time, block commit time, endorsement time, ordering time) and 16 Prometheus metrics (peer CPU, peer memory, orderer CPU, orderer memory, CouchDB query latency, block production rate, gossip message rate, Raft commit index).
Infrastructure requirements specify a dedicated load testing node with 8 CPU cores, 16GB RAM, 100GB SSD storage, external to K3s cluster to prevent resource contention, Ubuntu 22.04 LTS with Docker installed, Caliper version 0.6.0 with Fabric 2.5 binding, 1 Gbps network connection to cluster via NodePort services for orderer and peer access, and real-time Grafana dashboard monitoring during tests. The implementation timeline spans 10-12 days: Day 1 planning (this document), Day 2 setup (Caliper installation, network configuration, workload module development), Days 3-4 execution (7 hours total for baseline, sustained load, and stress testing), Day 5 analysis (performance report generation, bottleneck identification, optimization recommendations), and Days 6-10 optimization if needed (configuration tuning, resource allocation, code optimization).

Testnet/Mainnet Architecture Specification: A comprehensive architecture specification was developed for dual-environment infrastructure design following industry-standard practices for blockchain network separation and deployment isolation. The document analyzes three architecture options: namespace-based separation (single cluster, separate namespaces - simplest, lowest cost, some resource sharing), cluster-based separation (separate K8s clusters - better isolation, independent scaling, moderate complexity), and multi-region separation (geographically distributed clusters - maximum resilience, highest cost, complex management).
A hybrid implementation strategy was selected with 3-phase progressive migration: Phase 1 (Months 1-2) implements namespace isolation deploying testnet-fabric and mainnet-fabric namespaces in existing cluster with separate NetworkPolicies for network isolation and independent service endpoints; Phase 2 (Months 2-3) migrates to separate Kubernetes clusters with dedicated hardware resources for mainnet ensuring complete resource isolation; Phase 3 (Months 6-12) expands mainnet to multi-region deployment for geographic distribution and disaster recovery.
Kustomize-based configuration management architecture defines base configurations shared between environments in base/ directory (common Fabric component specs, shared ConfigMaps, base security policies) and environment-specific overlays in overlays/testnet/ and overlays/mainnet/ directories (replica counts, resource allocations, domain names, genesis block configurations). Network topology specifications define testnet with 3 orderers (reduced Raft cluster), 2 peers per organization (minimal redundancy), 1 CA per MSP, 2 CouchDB instances, and shared monitoring stack, while mainnet uses 5 orderers (production Raft cluster), 4 peers per organization (full redundancy), 1 CA per MSP with HA, 4 CouchDB instances, and dedicated monitoring with AlertManager.
Security isolation design implements separate RBAC roles (testnet-admin, mainnet-admin) with least privilege, independent NetworkPolicies preventing cross-environment traffic, separate TLS certificate chains (testnet CA, mainnet CA) with different root certificates, and encrypted secrets at rest with separate encryption keys per environment. CI/CD pipeline design automates testnet deployment on every pull request merge with automated health checks and integration tests, requires manual approval for mainnet promotion with change control review and rollback procedures, and implements blue-green deployment strategy for zero-downtime mainnet updates.
Token economics differentiation establishes testnet with unlimited faucet for test tokens, public API for developers to request tokens, automatic reset capability for testing cycles, and no real-world value, while mainnet implements genesis distribution (5 billion users, 6 phases), 1.25 trillion total supply (immutable), velocity tax and transaction fees (production economics), and real-world financial value with regulatory compliance. An 8-month implementation roadmap provides weekly task breakdown from Month 1 namespace setup through Month 8 mainnet stabilization with detailed deliverables and success criteria for each phase.
