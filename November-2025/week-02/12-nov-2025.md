### Duplicate Prometheus Deployment Resolution

The root cause of widespread Grafana "No Data" errors was identified as two Prometheus instances running simultaneously: an old Deployment with incorrect configuration including duplicate kube-state-metrics scraping, and a new StatefulSet with correct configuration. The Prometheus service was routing Grafana traffic to the buggy Deployment instance, causing dashboard queries to return incorrect or empty results.

The old Deployment was creating duplicate metric scraping: the kubernetes-pods job was scraping kube-state-metrics twice (once via static_configs, once via pod discovery with prometheus.io/scrape annotation), resulting in duplicate time series. For example, count(kube_node_info) incorrectly returned 8 nodes instead of the actual 4 nodes in the cluster (4 nodes × 2 scrapes = 8 time series).

The resolution involved deleting the old Deployment, forcing the service to route exclusively to the StatefulSet instance, and verifying all dashboards immediately began displaying correct data. A comprehensive diagnostic report was created documenting the root cause analysis, resolution steps, production deployment checklist, and diagnostic scripts for future troubleshooting. This fix restored full functionality to all monitoring dashboards.

### Duplicate Kube-State-Metrics Scraping Elimination

A specific duplicate scraping issue for kube-state-metrics was identified and resolved through Prometheus configuration refinement. The kube-state-metrics pod had prometheus.io/scrape: "true" annotation, and Prometheus configuration included both an explicit static scrape for kube-state-metrics:8080 and generic pod discovery that matched the annotation, resulting in duplicate time series for all kube-state-metrics metrics.

A relabel_configs drop rule was added to the kubernetes-pods job before the keep rule (relabel order is critical), configured with source_labels: [__meta_kubernetes_pod_label_app], action: drop, and regex: kube-state-metrics. Verification confirmed the fix: before the change, count(kube_node_info) returned 8 with 2 scrape targets for kube-state-metrics; after the change, count(kube_node_info) correctly returned 4 with 1 scrape target.

Impact includes all dashboards using kube_node_info metrics now showing correct values, reduced unnecessary scrape load on the kube-state-metrics pod, and Prometheus storage reduced by eliminating duplicate metrics. Documentation was added to the monitoring access guide covering this known issue and resolution for future reference.

### Dashboard Job Label Corrections

Critical job label mismatches were identified and corrected across multiple dashboards. The blockchain overview dashboard (01-blockchain-overview.json) was using incorrect Prometheus job labels: queries used job="orderer" and job="peer" while actual Prometheus scrape job labels are "fabric-orderers" and "fabric-peers". This caused the Orderers Online panel to show 0 instead of 5 and Peers Online panel to show 0 instead of 4-6.

Three query expressions were updated: Orderers Online query changed to count(up{job="fabric-orderers"} == 1), Peers Online query changed to count(up{job="fabric-peers"} == 1), and Transaction Rate query updated to rate(ledger_transaction_count{...,job="fabric-peers"}[5m]). Verification confirmed Prometheus has 8 fabric-orderers targets (5 mainnet up, 3 testnet down) and 6 fabric-peers targets (4 mainnet up, 2 testnet down), and queries now return correct data.

The peer performance dashboard (03-peer-performance.json) was completely non-functional with all 14 panels showing "No Data" errors due to all queries using incorrect job="peer" label instead of the actual "fabric-peers" job label. This affected critical operational metrics including endorsement success rate, proposal validation failures, blockchain height per peer, block processing time, chaincode execution duration, and state database query performance. All 12 occurrences of job="peer" were updated to job="fabric-peers", immediately restoring functionality to all 14 panels covering endorsement metrics, blockchain height, performance histograms, and GRPC connections.

### cAdvisor Integration for Container Metrics

Dashboard panels for "Pod CPU Usage" and "Pod Memory Usage" were displaying "No Data" errors due to missing cAdvisor metrics (container_cpu_usage_seconds_total, container_memory_working_set_bytes). Investigation revealed that only node-exporter, kube-state-metrics, and Fabric metrics were being scraped, preventing visibility into individual pod resource consumption.

A new scrape job 'kubernetes-cadvisor' was added to Prometheus configuration, scraping the /metrics/cadvisor endpoint from kubelet on all nodes using Kubernetes service discovery with role: node. Configuration includes scrape path /api/v1/nodes/${node}/proxy/metrics/cadvisor, TLS config using service account CA and bearer token, and relabeling to preserve node labels and add node name as label.

Verification after deployment confirmed 4 of 4 cAdvisor targets discovered and up (100% availability), container metrics now available with 60 time series for fabric namespace, and dashboard panels displaying real-time pod resource usage. This completes the monitoring stack with infrastructure, Kubernetes, and Fabric metrics all operational. cAdvisor is embedded in kubelet requiring no separate deployment, with metrics available at kubelet's /metrics/cadvisor endpoint.

### System Resources Dashboard Label Remediation

The system resources dashboard queries were using non-existent 'nodename' label on node-exporter metrics, causing "No Data" errors across 30+ panels. Root cause analysis revealed that node-exporter metrics expose 'instance' label in IP:port format, while kube-state-metrics use 'node' label in hostname format, creating a fundamental labeling mismatch.

Comprehensive query fixes were applied: replacing nodename=~"\$node" with instance=~".*" for node-exporter metrics, replacing nodename=~"\$node" with node=~"\$node" for kube-state-metrics, updating template variable to query kube_node_info with 'node' label, and fixing column rename mappings in infrastructure tables. Panel type conversions included converting cluster utilization gauges (panels 7-10) to stat panels, adding proper reduceOptions for single-value aggregation, and configuring graphMode for visual context in stat panels.

Affected panels totaling 20+ include Cluster Utilization (4 panels for CPU, Memory, Disk, Network), Per-Node Utilization (4 panels for CPU, Memory, Disk, Load Average), Saturation Indicators (8 panels for Disk IOPS, I/O, Network Traffic/Packets), Error Tracking (3 panels for Network Errors Rx/Tx, Drops), and Infrastructure Details (1 table for Node Resource Capacity). Label mapping was documented: node_cpu_seconds_total uses 'instance' label, node_memory_* uses 'instance' label, node_disk_* uses 'instance' label, node_network_* uses 'instance' label, kube_node_status_* uses 'node' label, and kube_pod_* uses 'node' label. Result: all 47 panels now display data correctly with proper label filtering for the 4-node production cluster (3 control plane + 1 worker).

Comprehensive documentation was added covering complete system resources dashboard architecture, USE Method implementation (Utilization, Saturation, Errors), detailed descriptions for all 47 panels across 7 rows, Prometheus query patterns and metric explanations, inverted pyramid layout for critical-first visibility, troubleshooting procedures for common monitoring issues, capacity planning guidelines and threshold recommendations, integration guide with environment-specific dashboards, template variables for node and namespace filtering, and operational procedures for daily monitoring and incident response.
