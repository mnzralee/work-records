**Title:** Testnet Environment Deployment & Multi-Environment Cluster Architecture Implementation

**Objective:**
To expand the existing K3s cluster from 3 to 4 nodes and deploy a complete testnet blockchain environment with zero-trust network isolation, implementing namespace-based separation between mainnet, testnet, and devnet environments while creating comprehensive operational documentation for deployment procedures, monitoring integration, and development workflows across all three network environments.

---

## Work Carried Out

### Cluster Expansion and Phase 1 Completion

The existing K3s cluster was expanded from 3 nodes to 4 nodes by adding srv1117946 (72.61.116.210) as a dedicated testdev worker node, providing isolated infrastructure for testnet and devnet environments. The new node was configured with the node-role=testdev label for targeted workload scheduling, joined to the existing K3s cluster using the K3s agent installation process, and validated for proper connectivity to the control plane and existing worker nodes.

Cluster resource capacity increased to 32 total CPU cores, 128GB RAM, and 1.6TB storage capacity across all nodes. Three dedicated namespaces were created: fabric (mainnet production), fabric-testnet (testing environment), and fabric-devnet (active development environment). ResourceQuotas were applied to each namespace to prevent resource contention and ensure mainnet production workloads remain unaffected by testing activities. Phase 1 completion was documented in approximately 30 minutes of execution time with zero downtime to existing mainnet operations.

### Zero-Trust Network Isolation with NetworkPolicies

Comprehensive NetworkPolicy configurations were implemented to establish zero-trust isolation between mainnet, testnet, and devnet environments, preventing any cross-namespace traffic while enabling necessary internal communication patterns. Five distinct NetworkPolicy types were created and applied to each environment: default-deny-all establishing baseline security by blocking all ingress and egress traffic, allow-intra-namespace permitting pod-to-pod communication within the same namespace, allow-dns enabling CoreDNS egress for service discovery, allow-monitoring-ingress permitting Prometheus scraping for pods labeled monitoring=true, and allow-external-access enabling client connections for pods labeled external-access=true.

The mainnet NetworkPolicies (applied to fabric namespace) ensure complete isolation from testnet and devnet, protecting production workloads from any potential issues in testing environments. Testnet NetworkPolicies (applied to fabric-testnet namespace) provide identical isolation structure but scoped to the testnet namespace. Devnet NetworkPolicies (applied to fabric-devnet namespace) implement the same zero-trust model for the development environment.

Verification procedures were documented including cross-namespace connectivity testing using curl and netcat commands, DNS resolution validation, Prometheus scrape target verification, and external client access testing. The NetworkPolicy implementation ensures that a critical failure in devnet cannot cascade to testnet or mainnet, provides security boundary enforcement at the network layer, enables safe experimentation with new chaincode versions, and supports compliance requirements for environment isolation.

### Testnet Cryptographic Infrastructure Generation

Complete cryptographic material was generated for the testnet environment using cryptogen with a dedicated configuration file (crypto-config-testnet.yaml). The certificate structure includes OrdererOrgTestnet organization with 3 orderer identities (orderer0, orderer1, orderer2), Org1Testnet peer organization with 1 peer and 1 user, and Org2Testnet peer organization with 1 peer and 1 user. All organizations use the testnet.goodness.exchange domain suffix for clear namespace isolation from mainnet production environment.

A critical enhancement was made to the orderer certificate configuration by adding Subject Alternative Names (SANs) for Kubernetes service DNS names. Each orderer certificate includes full Kubernetes DNS names (e.g., orderer0-ordererorg.fabric-testnet.svc.cluster.local), short DNS names, and localhost entries. This ensures TLS certificate validation succeeds when orderers communicate using cluster DNS names, which is essential for proper Raft consensus cluster formation in the Kubernetes environment.

NodeOUs were enabled for all organizations to support identity classification based on Organizational Unit fields in certificates, enabling role-based access control for admin, peer, client, and orderer identities. A Kubernetes Job was created to automate copying of cryptogen materials from the master node (srv1089618) to the testdev node (srv1117946) where testnet pods run, using sshpass for non-interactive SSH authentication and preserving proper file permissions and directory structure.

### Testnet Channel Configuration and Genesis Block Creation

The configtx.yaml configuration file was extended with testnet-specific profiles for Raft consensus and channel creation. The TestnetConsortium was defined containing both Org1Testnet and Org2Testnet peer organizations, enabling multi-organization channel creation. The TestnetGenesis profile configures the system channel with etcdraft consensus using 3 orderers for F=1 Byzantine fault tolerance, OrdererOrgTestnetMSP as the orderer organization, batch timeout of 2 seconds, maximum 100 messages per block, and 10MB absolute maximum bytes per block.

Orderer endpoints were configured using Kubernetes internal DNS names: orderer0-ordererorg.fabric-testnet.svc.cluster.local:7050, orderer1-ordererorg.fabric-testnet.svc.cluster.local:7050, and orderer2-ordererorg.fabric-testnet.svc.cluster.local:7050. Fabric 2.0 capabilities were enabled for channel, orderer, and application components.

An additional GxChannelTestnet profile was created for application channel creation using the modern Channel Participation API (Fabric 2.3+), which replaces the legacy system channel approach with dynamic channel management. This profile includes Raft consensus configuration, both OrdererOrgTestnet and peer organizations (Org1Testnet, Org2Testnet), proper capabilities configuration, and default policies. The Channel Participation API approach eliminates the system channel requirement, enables dynamic channel creation and joining via admin API, simplifies orderer bootstrapping with BOOTSTRAPMETHOD=none, and improves operational flexibility for channel management.

### Persistent Storage Provisioning for Testnet

Seven PersistentVolumeClaims were created for testnet storage allocation using the local-path StorageClass: 3 orderer PVCs (orderer0-data, orderer1-data, orderer2-data) at 5Gi each for Raft consensus metadata and ledger storage, 2 peer PVCs (peer0-org1-data, peer0-org2-data) at 10Gi each for blockchain ledger data and chaincode execution state, and 2 CouchDB PVCs (couchdb0-org1-data, couchdb0-org2-data) at 2Gi each for state database storage.

Total testnet storage footprint is 44Gi (15Gi for orderers + 20Gi for peers + 4Gi for CouchDB) with ReadWriteOnce access mode. All PVCs are bound to the fabric-testnet namespace with automatic provisioning on srv1117946 (testdev node) via the local-path provisioner. The storage allocation is optimized for testnet workloads which have lower transaction volumes than production mainnet.

### Raft Orderer Cluster Deployment

A complete 3-node Raft consensus cluster was deployed as StatefulSets in the fabric-testnet namespace using the hyperledger/fabric-orderer:2.5.11 image. Each orderer (orderer0, orderer1, orderer2) is configured with TLS enabled using server.crt, server.key, and ca.crt from generated crypto material, system channel genesis block mounted from /blockchain/testnet/system-genesis-block path, MSP identity from orderer-specific directories under OrdererOrgTestnet, operations endpoint on port 9443 for Prometheus metrics export, and admin endpoint on port 9444 for Channel Participation API access (separated from operations port to avoid conflicts).

Resource allocation per orderer includes 250m CPU request with 500m limit, 512Mi memory request with 1Gi limit, and PVC-backed persistent storage for consensus metadata. Node affinity configuration targets node-role=testdev (srv1117946) to isolate testnet workloads. Orderers communicate via Kubernetes DNS using service names: orderer0-ordererorg.fabric-testnet.svc.cluster.local:7050 (and orderer1, orderer2) for Raft cluster formation.

StatefulSet deployment ensures stable network identities, ordered pod creation and termination, and persistent storage binding across pod restarts. The 3-node configuration implements F=1 crash fault tolerance suitable for the testnet environment. Health probes were configured with liveness and readiness checks for container lifecycle management. The deployment was updated to use the Channel Participation API by setting BOOTSTRAPMETHOD=none, enabling modern channel management without genesis file dependency.

### CouchDB State Database Deployment

Two CouchDB 3.4.2 instances were deployed as separate pods to serve as state databases for testnet peer ledgers. Each CouchDB instance (couchdb0-org1, couchdb0-org2) is configured with admin credentials (admin/adminpw) for peer connectivity, 2Gi persistent storage via PVC for state data, resource allocation of 250m CPU request with 500m limit and 512Mi-1Gi memory, node affinity targeting node-role=testdev (srv1117946), and port 5984 exposed for HTTP API access.

Kubernetes DNS service names are couchdb0-org1.fabric-testnet.svc.cluster.local:5984 and couchdb0-org2.fabric-testnet.svc.cluster.local:5984. Dedicated CouchDB instances per organization enable rich queries and JSON document storage for world state data from peer ledgers, provide data isolation and scalability benefits, and support complex chaincode queries using the CouchDB query language.

### Peer Node Deployment with External Chaincode Builder

Two peer StatefulSets (peer0-org1, peer0-org2) were deployed in the fabric-testnet namespace using the hyperledger/fabric-peer:2.5.11 image. Each peer is configured with TLS enabled using server certificate, private key, and CA root certificate from cryptogen materials, MSP identity from organization-specific directories (Org1TestnetMSP/Org2TestnetMSP), CouchDB state database backend connected to the dedicated CouchDB instance for that organization, gossip protocol configuration for peer-to-peer communication within the organization, operations endpoint on port 9443 with Prometheus metrics export, and event service on port 7053 for chaincode event streaming to client applications.

Critical configuration for external chaincode support includes CORE_CHAINCODE_EXTERNALBUILDERS configured with the external builder path, builder scripts mounted from the external-builder-scripts ConfigMap, and support for chaincode running as independent Kubernetes services (chaincode-as-a-service pattern). This enables the ccaas deployment pattern matching production mainnet architecture where chaincode runs independently of peer container lifecycle.

Resource allocation per peer includes 250m CPU request with 1 CPU limit (increased from initial 500m for external chaincode operations), 512Mi memory request with 2Gi limit, and PVC-backed persistent storage (10Gi each) for ledger and state data. Node affinity configuration ensures scheduling on testdev node (srv1117946). Peers communicate via Kubernetes DNS using service names peer0-org1.fabric-testnet.svc.cluster.local:7051 and peer0-org2.fabric-testnet.svc.cluster.local:7051.

Network configuration includes hostAliases for orderer, CouchDB, and chaincode service resolution to bypass DNS timeout issues observed in the testnet namespace. Direct IP mapping was implemented as a workaround for DNS resolution problems during initial deployment.

### External Chaincode Deployment with CCAAS Pattern

The gxtv3 chaincode was deployed using the chaincode-as-a-service (ccaas) pattern where chaincode runs as an independent Kubernetes service instead of a peer-managed container. A Deployment named gxtv3-chaincode was created with 1 replica, using image 10.43.75.195:5000/gxtv3-chaincode-testnet:1.0 with imagePullPolicy: IfNotPresent (image pre-loaded via containerd import).

Environment configuration includes CHAINCODE_ID set to the package ID from peer lifecycle install operation, CHAINCODE_SERVER_ADDRESS set to 0.0.0.0:7052 to listen on all interfaces, and CHAINCODE_TLS_DISABLED set to true since TLS is handled at the peer level. A ClusterIP Service exposes port 7052 for peer-to-chaincode communication using the DNS name gxtv3-chaincode.fabric-testnet.svc.cluster.local:7052.

Health monitoring was configured with liveness probe using TCP check on port 7052 (10s initial delay) and readiness probe using TCP check on port 7052 (5s initial delay), ensuring peers only connect to healthy chaincode instances. Resource allocation includes 250m CPU request with 500m limit and 256Mi memory request with 512Mi limit. Node selector targets testdev node for scheduling.

Benefits of the ccaas pattern include independent chaincode lifecycle from peer containers, easier scaling and updates through Kubernetes Deployment, better resource isolation and monitoring, and alignment with enterprise production deployment practices. A persistent tools pod was created with hyperledger/fabric-tools:2.5.11 image using sleep infinity command to stay running indefinitely, providing an interactive CLI environment for peer lifecycle commands (install, approve, commit), peer chaincode invoke and query operations, osnadmin channel management via Channel Participation API, and debugging network issues.

### Image Distribution Solution for Isolated Nodes

A critical challenge was resolved regarding Docker image distribution to the testdev node which has network isolation preventing direct registry access. A NetworkPolicy (network-policy-registry.yaml) was created allowing egress from fabric-testnet namespace to the registry namespace on TCP port 5000 (Docker registry). However, network connectivity issues persisted.

An alternative solution was implemented using a privileged Pod (image-loader.yaml) with host access to the K3s containerd runtime. The pod mounts /run/k3s/containerd for ctr CLI access and /var/lib/rancher/k3s/agent for image storage, using the rancher/k3s:v1.33.5-k3s1 image which includes the ctr binary. The workaround workflow involves building the image on the master node, saving the image to tarball with docker save, streaming the tarball to the image-loader pod via kubectl exec, importing to containerd with ctr -n k8s.io image import, tagging with registry reference for Kubernetes compatibility, and deploying pods with imagePullPolicy: IfNotPresent to use the locally imported image.

A Kubernetes Job was created to automate this Docker image import process from master node to testdev node's containerd runtime. The Job runs on the master node with Docker socket access, saves the specified Docker image to tarball, streams the tarball via kubectl exec to the image-loader pod on testdev, imports directly to containerd, and tags with registry reference. Configuration parameters include IMAGE_NAME (Docker image to transfer), IMAGE_TAG (version tag, default: latest), and TARGET_IMAGE (registry reference for Kubernetes). This automation provides a reproducible image distribution workflow, eliminates network dependency for image pulls, and supports CI/CD integration for chaincode updates.

### Testnet Configuration Troubleshooting and Fixes

Several configuration issues were identified and resolved during testnet deployment. The external-builder ConfigMap namespace was corrected from 'fabric' to 'fabric-testnet' to match the testnet environment deployment target. The gxtv3-chaincode CHAINCODE_ID environment variable was updated to match the ccaas package ID (4eecb83f...) from the peer lifecycle install operation.

An incorrect hostAlias for gxtv3-chaincode was removed from peers-v2.yaml which was pointing to an old pod IP (10.42.3.92) and preventing proper DNS resolution to the chaincode service. The tools pod peer IP addresses were updated to current values (10.42.3.111, 10.42.3.112) as pods had been recreated with new addresses.

A corrected ccaas package was created with the proper DNS name gxtv3-chaincode.fabric-testnet.svc.cluster.local:7052, installed on both peers with package ID gxtv3_1.0:4eecb83f..., and committed as chaincode definition sequence 3. Despite configuration alignment, gRPC handshake timeouts persisted, requiring further investigation comparing testnet setup to the working mainnet configuration.

### Comprehensive Documentation Creation

Extensive documentation was created to support the multi-environment architecture. A README.md was added as the central navigation point for multi-environment blockchain deployment documentation, providing quick navigation, architecture summary, implementation timeline, and common tasks reference.

The unified 4-node cluster architecture specification document defines the complete architecture for mainnet, testnet, and devnet deployment on a single K3s cluster with namespace-based isolation. It includes resource planning details (32 cores, 128GB RAM, 1.6TB storage), node placement strategy (3 primary nodes for mainnet, 1 testdev node for testnet/devnet), network topology with separate PKI per environment, and detailed component specifications for each environment.

A step-by-step implementation guide provides a detailed 5-7 day implementation plan for adding new nodes and deploying testnet/devnet environments, including pre-flight checks, K3s agent installation commands, namespace creation with ResourceQuotas, node labeling strategy, validation procedures, and rollback instructions.

The network isolation configuration guide documents complete NetworkPolicy configurations for zero-trust isolation between environments, including default-deny-all policies, intra-namespace communication rules, DNS egress configuration, monitoring ingress exceptions, external access controls, and comprehensive testing procedures.

A comprehensive testnet deployment guide provides complete procedures including cryptogen configuration, configtx profiles for Raft consensus, Kubernetes manifests with node affinity, channel creation commands, chaincode deployment steps, and validation procedures. Total testnet resources documented: 10 pods, 44Gi storage.

A devnet deployment guide documents the minimal devnet setup with Solo orderer (1s block time), 2 peers, 2 CAs, and verbose DEBUG logging for active development, including deployment automation scripts, rapid chaincode upgrade procedures, real-time debugging workflows, and complete reset functionality. Total devnet resources: 7 pods, 19Gi storage with lower CPU/memory quotas.

A unified monitoring integration guide configures a single Prometheus/Grafana/Loki stack to monitor all three environments with environment-specific labels and filtering, including ServiceMonitor configurations for cross-namespace scraping, multi-environment Grafana dashboards with dropdown filters, Loki log aggregation with environment tags, and tiered alerting rules (critical for mainnet, warning for testnet, info for devnet).
