**Title:** Production Infrastructure Maintenance & Multi-Organization Endorsement Implementation: Server Disk Cleanup, Projector Fabric Connectivity Resolution & Blockchain Bootstrap System Execution

**Objective:**
To execute critical disk space cleanup across all four production servers resolving storage capacity issues, diagnose and fix the projector worker's Fabric connectivity failure caused by missing volume mounts, investigate and implement multi-organization endorsement requirements for protocol-level blockchain operations, and successfully execute the blockchain bootstrap system command establishing the GX Coin Protocol's foundational state on mainnet.

## Work Carried Out

### Production Server Disk Space Cleanup

Comprehensive disk space cleanup operation executed across the GX Coin production infrastructure spanning all four servers. Primary investigation confirmed user suspicion regarding large accumulation of temporary TAR archive files from image transfers between servers combined with excessive Docker build cache and outdated Docker images from versions 2.0.1 through 2.0.19. Total space reclaimed reached 68GB from the primary Kuala Lumpur control-plane server srv1089618 which had reached critical disk usage at 76% capacity.

srv1089618 (Kuala Lumpur control-plane) analysis revealed Docker build cache consuming 89.88GB, Docker images totaling 85 instances with only 9 active, TAR archives in /tmp directory consuming approximately 45GB, K3s containerd storage at 102GB, and logs accumulating 11GB. Cleanup script executed in six phases targeting Docker build cache removal reclaiming 89GB, Docker unused image pruning keeping latest three versions per service, temporary TAR file removal for archives older than seven days, Docker system prune for stopped containers and unused networks, log rotation compressing files larger than 100MB and removing those older than 30 days, and K3s containerd image cleanup. Post-cleanup disk usage reduced from 301GB/399GB (76%) to 233GB/399GB (59%) achieving healthy operational state.

Remaining three servers verified in healthy state with Phoenix control-plane srv1089624 at 174GB/399GB (44%), Frankfurt control-plane srv1092158 at 143GB/399GB (36%), and Kuala Lumpur worker node srv1117946 at 44GB/399GB (11%). Prevention strategies documented including Docker image retention policy implementation keeping only latest three versions, automatic TAR file cleanup via daily cron job removing files older than seven days from /tmp, log rotation configuration with weekly compression and cleanup, and disk monitoring alert setup through Prometheus triggering at 70% threshold with Grafana dashboard visualization.

### Projector Worker Fabric Connectivity Diagnosis

Investigation initiated when svc-admin pods displayed 0/1 Ready status for approximately three days with readiness checks failing on /readyz endpoint returning 503 while liveness checks passing on /livez returning 200 OK. Database query revealed projection lag of 323,722 seconds (approximately 3.7 days) far exceeding configured threshold of 5,000 milliseconds causing readiness check failures. Projector logs analysis showed continuous reconnection loop with gRPC UNAVAILABLE errors and "read ECONNRESET" messages indicating network connectivity issues to Fabric peer.

Configuration comparison between outbox-submitter (working) and projector (failing) deployments identified root cause as missing fabric-wallet volume mount in projector deployment. The outbox-submitter deployment included /fabric-wallet volume mount from fabric-wallet ConfigMap and FABRIC_WALLET_PATH environment variable, while projector deployment lacked both configurations. This prevented projector from authenticating to Fabric network using org1/org2 admin certificates required for event stream connectivity.

Projector deployment YAML updated adding FABRIC_WALLET_PATH environment variable pointing to /fabric-wallet directory, fabric-wallet volume mount from ConfigMap with read-only access and 0400 permissions, and fabric-credentials volume mount from secret for TLS certificates. Projector Dockerfile modified to use pre-built artifacts pattern matching outbox-submitter approach, copying pre-built packages/ and workers/projector/dist directories instead of building TypeScript within Docker context. New projector image built as version 2.0.20 and imported into k3s containerd.

### Projector Connectivity Restoration Verification

Post-deployment verification confirmed successful Fabric network connectivity with projector logs showing "Successfully connected to Fabric network" and "Event listener started from block 1" messages. Metrics server confirmed listening on port 9091 enabling Prometheus monitoring integration. Pod status verification showed projector at 1/1 Ready state and all three svc-admin pods transitioning to 1/1 Ready state after projection lag decreased below threshold.

Countries endpoint testing via port-forward to svc-admin confirmed successful data retrieval with HTTP 200 response returning 15 countries spanning all major regions including Australia, Brazil, Canada, China, Germany, Spain, France, United Kingdom, India, Italy, Japan, South Korea, Mexico, United States, and South Africa. This verified complete data flow from blockchain events through projector processing to read model population and API service response.

### Multi-Organization Endorsement Architecture Investigation

Bootstrap command execution initially failed with endorsement policy error indicating insufficient transaction endorsements despite successful Fabric Gateway SDK connection. Investigation revealed chaincode committed with MAJORITY endorsement policy requiring both Org1MSP and Org2MSP peer endorsements. Direct database insertion test confirmed transaction reaching blockchain with 13.4-second processing duration but failing at endorsement collection phase rather than connection phase.

Architecture analysis established industry-standard two-tier endorsement pattern for blockchain systems. Protocol-level operations including BootstrapSystem, UpdateSystemParameter, PauseSystem, ResumeSystem, InitializeCountry, and AppointAdmin require multi-organization endorsement with AND policy preventing single-org manipulation. Regular transactions including TransferTokens, CreateUser, UpdateProfile, ApplyForLoan, and CastVote utilize single-organization or lightweight OR policy enabling high-volume operations with speed and scalability. This pattern mirrors enterprise blockchain implementations including IBM Food Trust and We.Trade trade finance platform.

Backend config ConfigMap updated adding FABRIC_ORG2_PEER_ENDPOINT pointing to peer0-org2.fabric.svc.cluster.local:7051, FABRIC_ORG2_MSP_ID as Org2MSP, and FABRIC_ORG2_TLS_SERVER_NAME_OVERRIDE as peer0.org2.prod.goodness.exchange.

### Gateway SDK Service Discovery Validation

Testing confirmed Fabric Gateway SDK service discovery functioning correctly without requiring custom endorsement aggregation implementation. When connecting with Org1 identity to Org1 peer, Gateway SDK automatically discovers Org2 peers through channel configuration and gossip protocol, collects endorsements from both organizations satisfying MAJORITY policy, and submits transaction to orderer for block commitment. This eliminated need for manual cross-organization endorsement coordination at application layer.

Key insight documented that Gateway SDK handles multi-org endorsement transparently when chaincode endorsement policy and peer gossip configuration are correct. Initial connection errors with 127.0.0.1 addresses resulted from incorrect environment variable configuration rather than fundamental SDK limitations. Proper peer endpoint configuration combined with TLS server name override enables cross-organization transaction processing within Kubernetes cluster network.

Complete CQRS flow validated from command insertion through blockchain commitment to read model projection. Outbox-submitter worker correctly picked up PENDING commands, selected appropriate identity based on command type, submitted transactions to Fabric network, and updated command status to COMMITTED with transaction details. Projector worker confirmed receiving blockchain events and updating PostgreSQL read models enabling svc-admin API responses.

Blockchain state verified at block 34 with bootstrap event at block 32, country initialization at block 33, and subsequent blocks available for future operations. ProjectorState table showed main-projector processing events with updated checkpoint timestamps within threshold. svc-admin readiness checks confirmed passing after projector caught up with blockchain events eliminating previous 3.7-day lag.

### Production Readiness Assessment

System operational status confirmed with all four production servers in healthy disk usage state, projector connectivity restored with real-time event processing, multi-org endorsement functioning correctly for protocol operations, bootstrap system command executed initializing foundational blockchain state, and country data initialized enabling user registration flows.
