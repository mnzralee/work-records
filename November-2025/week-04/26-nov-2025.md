**Title:** Enterprise Progressive Registration System Implementation: Wise.com-Style 7-Step Flow, Enhanced KYR Database Schema, BFF API Layer & Admin Dashboard KYR Review Interface

**Objective:**
To implement an enterprise-level progressive registration system inspired by Wise.com's user experience, establish comprehensive KYR (Know Your Relationship) database schema with identity document tracking and employment verification fields, develop complete backend API endpoints with frontend wizard components for both registration and KYR flows, and enhance the admin dashboard with detailed user review interfaces supporting the multi-stage verification workflow.

---

## Work Carried Out

### Architecture Design Documentation

Comprehensive design documentation created establishing the technical specification for progressive registration and KYR verification flows totaling approximately 15 pages of detailed requirements. The registration flow design specified 7 discrete steps progressing from email collection through OTP verification, name and country capture, date of birth and gender collection (required for Fabric User ID generation), password creation with strength validation, phone number submission, and final phone OTP verification. Each step designed with data persistence ensuring users can resume abandoned registrations within 24-hour expiry window.

KYR verification flow design specified 7 additional steps beginning with personal details verification (pre-filled from registration), mandatory national ID card upload with front and back images, optional passport document collection with explicit skip functionality, work and company details for future organization membership linking, address verification with proof of address document upload, biometric verification placeholder for future implementation, and final review step including PEP (Politically Exposed Person) declaration. Design explicitly excluded Source of Funds fields as GX Protocol operates as independent monetary system not requiring traditional banking compliance fields.

Test OTP strategy documented specifying "111111" as development bypass code enabling end-to-end testing without real SMS or email provider integration. Genesis eligibility criteria documented requiring users between 13-73 years of age for initial token allocation eligibility, with calculation performed during date of birth submission step.

### Prisma Database Schema Extensions

Database schema underwent significant extension adding four new enums and two new models while enhancing the existing UserProfile model with 18 additional fields. RegistrationStep enum defined with 9 states tracking progressive registration from EMAIL_PENDING through EMAIL_VERIFIED, NAME_COUNTRY, DOB_GENDER, PASSWORD, PHONE_PENDING, PHONE_VERIFIED, MIGRATING, to COMPLETED enabling granular progress tracking and resume capability. EmploymentStatus enum defined with 7 categories including EMPLOYED, SELF_EMPLOYED, FREELANCER, STUDENT, RETIRED, UNEMPLOYED, and OTHER supporting comprehensive work verification. Gender enum simplified to MALE and FEMALE values required for deterministic Fabric User ID generation algorithm. AddressType enum defined with CURRENT, PREVIOUS, MAILING, and WORK categories enabling address history tracking.

PendingRegistration model created implementing the progressive registration state machine with email verification fields (email, emailOtpHash, emailOtpExpiresAt, emailOtpAttempts), name and country fields (firstName, lastName, countryCode with foreign key to Country table), date of birth and gender fields, password hash storage, phone verification fields mirroring email structure, progress tracking via currentStep enum, security audit fields (ipAddress, userAgent) for fraud detection, migration tracking (migratedToProfileId, migratedAt) linking completed registrations to UserProfile records, and automatic expiry timestamp defaulting to 24 hours from creation.

Address model created supporting KYR address verification with type classification, full address fields (addressLine1, addressLine2, city, stateProvince, postalCode, countryCode), proof of address document tracking (proofDocumentUrl, proofDocumentHash, proofDocumentType), verification status with admin audit trail (isVerified, verifiedAt, verifiedBy), and history tracking with validFrom and validTo dates enabling address change history.

UserProfile model enhanced with middleName for complete legal name capture, placeOfBirth for birth location verification, national ID fields (nationalIdNumber, nationalIdIssuedAt, nationalIdExpiresAt) marked as mandatory for KYR completion, passport fields (passportNumber, passportIssuingCountry, passportIssuedAt, passportExpiresAt) marked as optional, employment fields (employmentStatus enum, jobTitle, companyName, industry, workEmail, workPhoneNum) supporting work verification and future organization membership, and PEP compliance fields (isPEP boolean, pepDetails text) for politically exposed person declaration. Schema validated successfully with npx prisma validate confirming all relations and constraints properly defined.

### Registration Backend API Implementation

Registration routes module created defining 10 endpoints supporting the complete progressive flow. POST /api/v1/registration/email handles step 1 email submission creating PendingRegistration record. POST /api/v1/registration/email/verify handles step 2 OTP verification with attempt tracking. POST /api/v1/registration/email/resend enables OTP regeneration with rate limiting. POST /api/v1/registration/name-country handles step 3 collecting first name, last name, and ISO country code with database validation. POST /api/v1/registration/dob-gender handles step 4 collecting date of birth and gender while calculating genesis eligibility. POST /api/v1/registration/password handles step 5 with strength validation and bcrypt hashing. POST /api/v1/registration/phone handles step 6 phone submission with uniqueness validation. POST /api/v1/registration/phone/verify handles step 7 final OTP verification triggering migration to UserProfile. POST /api/v1/registration/phone/resend enables phone OTP regeneration. GET /api/v1/registration/:id enables registration status retrieval for resume functionality.

Registration service implementation totaled 873 lines of business logic handling all registration operations. submitEmail function creates PendingRegistration with generated OTP hash using bcrypt, sets 24-hour expiry, and logs IP address and user agent for security audit. verifyEmailOtp function validates OTP against stored hash, tracks failed attempts with maximum 5 limit, and transitions to EMAIL_VERIFIED step on success. submitNameCountry function validates country code against Country table preventing invalid foreign key references. submitDobGender function validates date format and minimum age of 13 years while calculating genesis eligibility for ages 13-73. submitPassword function enforces strength requirements (8+ characters, uppercase, lowercase, number) and stores bcrypt hash with 10 salt rounds.

submitPhone function validates phone number format and checks uniqueness against both UserProfile and PendingRegistration tables preventing duplicate registrations. verifyPhoneOtp function performs final validation then executes migration transaction creating UserProfile record with all collected data, generating JWT access token with 24-hour expiry and refresh token with 7-day expiry, and marking PendingRegistration as migrated. All OTP operations support test code "111111" for development documented with clear code comments marking sections for production removal.

Registration controller implementation totaled 410 lines handling HTTP request/response cycle with comprehensive input validation. Each endpoint validates required fields returning 400 Bad Request with specific field errors. Email format validation uses regex pattern. Name length validation enforces 2-50 character range. Country code validation expects ISO-2 format. Date format validation expects YYYY-MM-DD pattern. Phone format validation expects international format with country code prefix. All responses follow standardized JSON structure with success boolean, optional message and error strings, and endpoint-specific data fields.

### Registration Frontend Wizard Implementation

RegistrationWizard component created totaling 1,031 lines implementing the complete 7-step user interface. Step 1 presents email input with format validation and submit button. Step 2 presents OTP input with 6-digit field, resend button with 60-second cooldown timer, and test OTP notice displayed in development mode. Step 3 presents first name and last name inputs with country selector dropdown populated from ISO-2 country codes. Step 4 presents date picker for date of birth with age validation (minimum 18 years, maximum 120 years) and gender selection buttons for MALE and FEMALE options with genesis eligibility notification displayed when age qualifies for initial token allocation.

Step 5 presents password input with confirmation field and real-time strength indicators showing 4 requirements (8+ characters, uppercase letter, lowercase letter, number). Step 6 presents phone number input with international format guidance. Step 7 presents final phone OTP verification mirroring step 2 structure. Upon successful completion, component triggers automatic login via NextAuth using returned JWT tokens and redirects to dashboard.

Visual design implements gradient progress bar transitioning from purple (#470A60) to green (#17A210) matching brand guidelines. Framer Motion AnimatePresence provides smooth step transitions with slide and fade animations. Responsive design supports both mobile and desktop viewports with appropriate input sizing and button placement.

Ten BFF (Backend-for-Frontend) API routes created in Next.js app directory proxying requests to backend svc-identity service. Each route handles request validation, error transformation, and response formatting for frontend consumption. Routes include /api/registration/email, /api/registration/email/verify, /api/registration/email/resend, /api/registration/name-country, /api/registration/dob-gender, /api/registration/password, /api/registration/phone, /api/registration/phone/verify, /api/registration/phone/resend, and /api/registration/[registrationId] for status retrieval.

### KYR Wizard Reconstruction

KYRWizard component completely rebuilt expanding from 5 steps to 7 steps per architecture specification. Step 1 Personal Details presents pre-filled information from registration (first name, last name, country) with additional fields for middle name and place of birth verification. Step 2 National ID Card designated as mandatory requirement presents upload interface for front and back images with file hash computation using SHA-256 via Web Crypto API for document integrity verification. Step 3 Passport designated as optional presents similar upload interface with explicit Skip button enabling users without passport to proceed.

Step 4 Work Details presents employment status dropdown (EMPLOYED, SELF_EMPLOYED, FREELANCER, STUDENT, RETIRED, UNEMPLOYED, OTHER), job title input, company name input, industry selector, and optional work email and phone fields. Step 5 Address presents full address form (address line 1 and 2, city, state/province, postal code, country) with proof of address document upload option. Step 6 Biometric presents placeholder interface with "Coming soon" messaging as biometric verification infrastructure remains pending. Step 7 Review presents summary of all submitted information with PEP declaration checkbox requiring user acknowledgment before final submission.

Crypto.subtle fallback implementation added to handle non-HTTPS development contexts where Web Crypto API is unavailable. When crypto.subtle returns undefined, component falls back to simple hash algorithm ensuring development testing remains functional while production environments use full SHA-256 computation. Visual design matches registration wizard with consistent gradient color scheme and Framer Motion transitions.

### Production Deployment and Issue Resolution

Kubernetes ingress configuration discovered missing route for /api/v1/registration path preventing frontend from reaching registration endpoints. Ingress patched using kubectl patch command adding new path entry routing /api/v1/registration prefix to svc-identity service on port 3001. Verification confirmed 200 OK responses from registration endpoints following patch application.

KYR submission flow issue identified where users completing KYR verification remained invisible to admin dashboard. Root cause analysis revealed submitKYC method in users.service.ts created KYCVerification record but failed to update UserProfile status from REGISTERED to PENDING_ADMIN_APPROVAL. Fix implemented wrapping operations in Prisma transaction ensuring atomic creation of KYCVerification record with PENDING status and update of UserProfile status. Two existing users with pending KYC manually updated via database query to correct status enabling admin review.

svc-identity rebuilt as version 2.0.20 incorporating all registration and KYR submission fixes. Image distributed to all three control-plane nodes (srv1089618, srv1089624, srv1092158) using established TAR export and k3s ctr import workflow. Deployment rollout completed successfully with all pods running updated version. Database schema push executed via kubectl port-forward to production PostgreSQL applying all schema changes including new models and UserProfile field additions. Note documented that identityNum column replaced by nationalIdNumber with 1 existing value migrated.
