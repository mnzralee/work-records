Title: Atomic Genesis Distribution & End-to-End On-Chain Registration Stabilization (Fabric Chaincode + CQRS Projector)

Objective:
To eliminate race conditions and partial-state failures in the user onboarding flow by making genesis token allocation occur atomically within the `IdentityContract:CreateUser` transaction, while also hardening the event-processing pipeline (projector/outbox-submitter) for schema and field-name backward compatibility. The goal was to restore a reliable end-to-end wallet registration path (API â†’ outbox â†’ Fabric â†’ projector) and validate the fixes through production-grade deployments and transaction-level verification.

Work Carried Out:
**Projector Event Schema Compatibility (eventVersion Backward Support):** The projectorâ€™s Fabric event ingestion began failing on `UserCreated` events due to a strict requirement for an `eventVersion` field that legacy chaincode events did not emit. To prevent the read-model pipeline from rejecting otherwise valid on-chain activity, the core event validator was updated to treat `eventVersion` as optional and to default to a safe baseline version (`"1.0"`) when omitted. This preserved strictness when the field exists, but enabled backward-compatible processing for existing chaincode payloads.

**Lenient Validation Mode to Prevent Silent Event Skips:** Beyond the missing `eventVersion`, strict schema enforcement risked turning transient schema drift into permanent data divergence (events skipped â†’ projections never reconcile). The projector was reconfigured to run in a lenient validation mode that logs warnings for schema mismatches while continuing to process the event stream. This ensured operational continuity during the transition period where chaincode event payloads and TypeScript schemas were being aligned.

**Field Naming Bridging for Fabric Payloads (userID vs userId):** A practical integration mismatch was identified between Go-style exported field naming in chaincode (`userID`) and the handler expectations in the projector (`userId`). The event handler logic was hardened to accept both representations using a fallback assignment (`payload.userID || payload.userId`) so that `UserCreated` events would reliably map to a `fabricUserId` even when the payload came from differing serialization conventions.

**Projector Fallback Simplification (Avoiding Incorrect Auto-Creation):** A defensive â€ścreate fallbackâ€ť path existed that attempted to create missing user records when encountering unknown identifiers. In practice this was unsafe because chaincode-originated event payloads can contain fields not present in the database schema (e.g., `identityNum`), causing the fallback path to fail noisily and potentially creating incomplete or invalid records. The fallback was removed and replaced with warning-level logging for unknown `fabricUserId`, ensuring the projector remains resilient without inventing partial state.

**Atomic Genesis Distribution Embedded into CreateUser (Core Architectural Change):** The genesis distribution mechanism previously relied on a separate outbox command (e.g., `DISTRIBUTE_GENESIS`) after user creation, which introduced race conditions between identity persistence and token allocation, plus the risk of partial completion (user exists, allocation missing) if downstream processing failed. The architecture was refactored so that genesis allocation occurs immediately within `CreateUser` as part of the same Fabric transaction: the user is written to the ledger and then `distributeGenesisForNewUser(ctx, &user)` is executed inline, ensuring the entire onboarding operation either commits together or rolls back together.

**Read-After-Write Isolation Fix in Fabric (Passing User by Pointer):** A key Fabric constraint was addressed during the atomic refactor: within the same transaction, `GetState` cannot reliably observe uncommitted `PutState` writes (read-after-write visibility is not guaranteed in the endorsement simulation model). The initial designâ€”writing the user and then re-reading it from the ledgerâ€”would return `nil` and break the distribution logic. This was resolved by threading the in-memory `User` object through the genesis distribution path (passing a pointer directly), eliminating the need for an immediate re-read and ensuring nationality/age/userId are available deterministically inside the same transaction context.

**Tokenomics Contract Refactor for Dual Invocation Paths (Admin vs Inline Atomic):** To keep operational flexibility while enabling the new atomic path, genesis distribution logic was split into internal functions: one internal variant that reads the user from the ledger (supporting manual/admin catch-up calls), and a second internal variant that accepts a `User` pointer directly (supporting the atomic `CreateUser` flow). This refactor preserved the full 6-phase tokenomics model (phase gating, age eligibility rules, pool transfers, country-level allocation tracking, and event emission) while removing the need for a separate backend distribution command.

**Cross-Contract Invocation Constraint (BootstrapSystem txid Conflict) and Direct Ledger Helpers:** A critical Fabric limitation was also resolved for system bootstrap: Hyperledger Fabric does not allow `InvokeChaincode` calls to the same chaincode within a single transaction (the endorsement simulation rejects this with a transaction ID conflict, commonly surfaced as a â€śtxid existsâ€ť failure). This impacted `BootstrapSystem` when it attempted to initialize genesis pools by calling tokenomics functions via `InvokeChaincode`, resulting in commands appearing committed while pools remained uninitialized. The fix replaced same-chaincode invocation with direct ledger operations via package-level helper functions (e.g., `readBalance`, `creditAccount`) built on `GetState`/`PutState`, including overflow protection and safe account creation semantics.

**Outbox-Submitter and Registration Pipeline Hardening (Field Name Backward Compatibility):** To maintain compatibility across evolving payload contracts, the outbox-submitter was updated to support both `countryCode` and `nationality` naming when processing genesis-related payloads. This prevented undefined argument failures when older/newer producers emitted slightly different payload shapes, and reduced the operational burden during schema evolution.

**End-to-End On-Chain Registration Restore (DNS + BiometricHash Validation):** Multiple blockers prevented successful user registration via the outbox pattern. First, Fabric Gateway peer discovery was returning peer hostnames based on TLS identities (e.g., `peer0.org1.prod.goodness.exchange:7051`) that were not resolvable from within the Kubernetes cluster, causing connection attempts to fall back to `127.0.0.1`. This was resolved by adding the external peer/orderer hostnames to CoreDNS NodeHosts, mapping them to internal service IPs so gateway discovery resolves correctly in-cluster. Second, the chaincodeâ€™s biometric hash validation requires a 64-character SHA-256 hex string, but the backend registration flow was incorrectly sending a bcrypt password hash (60 characters, `$2b$...`). The registration service was corrected to generate a deterministic SHA-256 hash using stable user identifiers (e.g., profileId, fabricUserId, and dateOfBirth ISO timestamp), restoring compliance with chaincode validation.

**Production Deployment and Verification (Chaincode + Services):** The updated chaincode was packaged and deployed to the production Kubernetes Fabric network, and the lifecycle was advanced to a new committed definition (`gxtv3` version 2.9, sequence 13) with Org1 and Org2 approvals recorded. Deployment work included building and importing the updated chaincode image, installing the CCAAS package on the required peers, committing the definition, and updating deployment environment variables. The backend service changes were also containerized and rolled out, and successful registration was validated both by direct chaincode invocation and by observing a full outbox-submitter transaction reaching `COMMITTED` with a concrete Fabric transaction ID and block inclusion.

Outcome:
The user onboarding workflow was stabilized end-to-end: user creation now triggers genesis distribution atomically within a single Fabric transaction, eliminating race conditions and partial states, while projector/outbox-submitter compatibility fixes ensured events and payloads continue to flow during schema evolution. Production deployments were completed and validated through successful on-chain registration transactions and confirmed system pool initialization.
