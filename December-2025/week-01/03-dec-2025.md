Title: Enterprise Registration Hardening, Batch Blockchain Onboarding & Google Drive Document Storage Architecture (Full-Stack Integration)

Objective:
To deliver production-ready user registration capabilities by implementing comprehensive OWASP-compliant input validation across the entire 7-step flow, establishing a proper KYC-to-blockchain separation of concerns via batch processing workflows, and architecting a secure, virus-scanned document storage system leveraging Google Drive API with ClamAV integration. The parallel objective was to resolve country support gaps (Kosovo) and enhance UX with password visibility controls, ensuring a complete end-to-end registration experience ready for mainnet deployment.

Work Carried Out:
**Comprehensive Input Validation and Sanitization Framework (OWASP Compliance):** A new validation utilities module was created within `@gx/core-http` to centralize all input validation logic across registration endpoints. The framework implements RFC-compliant validation for email (RFC 5322 with domain TLD checks), E.164 international phone number format (7-15 digits with automatic country dial code integration), Unicode-aware name validation supporting international characters while detecting injection patterns (XSS script tags, SQL comment sequences, template injection syntax), and password strength assessment with complexity scoring (weak/medium/strong based on uppercase, lowercase, numeric, and special character composition). Additional validators were added for date-of-birth with age range constraints (13-120 years), gender normalization (case-insensitive MALE/FEMALE output), ISO 3166-1 alpha-2 country codes, UUID formats (v1-v5), and 6-digit OTP codes. All validation functions return structured results including sanitized values and specific error codes, enabling consistent error propagation through the stack.

**Standardized Error Code System for Client-Side Error Handling:** A comprehensive error code taxonomy was established with 25+ distinct validation error codes (e.g., `EMAIL_ALREADY_EXISTS`, `PHONE_INVALID_FORMAT`, `OTP_EXPIRED`, `PASSWORD_TOO_WEAK`, `REGISTRATION_EXPIRED`) that map backend validation failures to user-friendly frontend messages. The registration controller was completely refactored to use the new validation utilities for all seven steps, with each endpoint returning structured error responses containing a human-readable message, machine-readable error code, field identifier, and HTTP status code. This allows the frontend to display contextual, field-specific error messages and handle cases like registration expiration by automatically resetting the wizard state to step 1.

**Frontend Registration Wizard Enhancement with Modular Feature Flags:** The registration wizard received a significant UX overhaul including client-side error code mapping (30+ mappings from backend codes to localized messages), enhanced error handling that extracts structured error details from API responses, and dynamic phone number formatting with country dial code auto-population based on selected country. A modular feature flags system was implemented in `lib/config/registration-features.ts` allowing individual registration steps and features to be toggled via environment variables (e.g., `NEXT_PUBLIC_FEATURE_EMAIL_VERIFICATION`, `NEXT_PUBLIC_FEATURE_PHONE_VERIFICATION`), enabling flexible testing scenarios and gradual feature rollouts. The wizard was refactored from numeric step tracking to ID-based navigation, making the step flow dynamically configurable based on active feature flags.

**Batch Blockchain Registration Workflow (KYC-to-Blockchain Separation):** A critical architectural decision was made to separate user registration, KYC approval, and blockchain registration into three distinct phases, addressing scalability and operational control requirements. The initial 7-step registration flow creates users with `REGISTERED` status and `NOT_REGISTERED` onchainStatus, representing completed identity collection but not yet blockchain-committed. After admin KYC document review and approval via a dedicated endpoint, users transition to `ACTIVE` status while remaining `NOT_REGISTERED` onchain. A new batch approval workflow was implemented in the admin service (`batchApproveForBlockchain`) that queries for all KYC-approved users pending blockchain registration, generates deterministic Fabric User IDs using the existing encoding algorithm (`CC CCC AANNNN TCCCC NNNN` format embedding country, DOB, gender, and account type), and creates `CREATE_USER` outbox commands in a single database transaction per user. This ensures atomic assignment of fabricUserId and outbox command creation, preventing partial states where a user has an ID but no pending blockchain transaction.

**Kosovo Country Support Resolution via Database and Blockchain Synchronization:** A production issue was identified where Kosovo (XK) was missing from both the PostgreSQL database and the Fabric blockchain despite being present in seed data, causing registration failures for users selecting Kosovo. The resolution involved manually inserting the Kosovo country record into the database and executing `AdminContract:InitializeCountryData` on the mainnet Fabric network to establish Kosovo's population percentage (0.023%) and phase-wise genesis allocations (23k/46k/69k/92k/356.5k/563.5k for phases 1-6). This required dual-organization endorsement (peer0-org1 + peer0-org2) to satisfy the channel's endorsement policy, and external DNS name usage for orderers to avoid Kubernetes internal DNS resolution failures. Verification confirmed 235 countries now initialized across both database and blockchain layers.

**Password Visibility Toggle for Enhanced User Experience:** Based on user feedback, password visibility toggles were implemented in both the registration wizard and login form using Eye/EyeOff icons from lucide-react. The implementation maintains independent state for password and confirm-password fields, uses `tabIndex={-1}` on toggle buttons to prevent disruption of keyboard navigation flow, includes accessible `aria-label` attributes for screen reader support, and preserves the existing floating label styling pattern. This provides users with control over password visibility during entry while maintaining security best practices.

**Document Storage Architecture Design and Implementation (Google Drive + ClamAV):** A comprehensive document storage system was architected and implemented to support KYC document uploads, addressing requirements for 2TB storage capacity (user's existing personal Google Drive), administrative document access, and virus scanning. A new shared package `@gx/core-storage` was created providing storage abstraction through a `StorageProvider` interface (enabling future migration to S3 or Azure Blob), `VirusScanner` interface for malware detection, and a combined `DocumentStorageService` that enforces virus scanning before storage. The Google Drive provider implementation uses service account authentication with JSON key credentials, supports automatic folder hierarchy creation (`GX-Protocol-Documents/users/{profileId}/kyc/`, `organizations/{orgId}/`, `relationships/{relId}/`), generates signed URLs for time-limited administrative access, and maintains a folder cache to minimize Drive API calls.

**ClamAV Integration and Kubernetes Deployment Preparation:** The virus scanning component was implemented using the ClamAV antivirus engine with stream-based scanning for memory efficiency, lazy connection initialization with caching to handle credential-not-yet-configured scenarios, configurable bypass mode for development environments, and health check endpoints for monitoring integration. Kubernetes manifests were prepared for ClamAV deployment including a StatefulSet with 5Gi persistent volume for virus definition storage, resource limits (500m CPU, 2Gi RAM), startup probes allowing 5 minutes for initial virus database download, and a ClusterIP service exposing the clamd protocol on port 3310. Configuration includes 10MB maximum file size, support for multiple file format scanning (PE, ELF, OLE2, PDF, HTML, archives), and 4 worker threads for concurrent scan operations.

**Document Upload API Endpoints and Multipart Form Handling:** The svc-identity service was extended with document management endpoints using multer middleware for multipart form-data handling with memory storage (enabling buffer access for scanning), 10MB file size limits matching ClamAV configuration, and MIME type validation against allowed document types. The document upload flow validates file size and type, computes SHA-256 hashes for integrity verification, invokes ClamAV scanning with automatic rejection of infected files, uploads to Google Drive with metadata preservation, creates `KYCDocument` database records with automatic linkage to user profiles and KYC submissions, and logs security incidents for detected viruses. Additional endpoints support listing user documents, generating signed URLs for admin access (with expiration), document deletion from both storage and database, health checks for storage and scanner availability, and constraint queries for client-side validation configuration.

**Structured Error Propagation Across All Registration Routes:** All registration API routes in the frontend BFF layer were refactored to use a new `getApiErrorDetails` helper that extracts structured error information from Axios errors, properly forwards backend validation error codes to the client, preserves field identifiers for inline error display, and handles non-Axios errors with sensible defaults. This ensures that backend validation failures (e.g., `EMAIL_ALREADY_EXISTS` from database uniqueness checks) correctly propagate through the Next.js API routes to the frontend wizard, enabling accurate user feedback without losing error context at middleware boundaries.

Outcome:
The user registration flow was hardened to production standards with comprehensive OWASP-compliant validation, standardized error handling, and proper separation of registration/KYC/blockchain phases via batch processing workflows. A complete document storage system was implemented using Google Drive API with ClamAV virus scanning, ready for deployment pending GCP service account configuration. Kosovo country support was restored across database and blockchain, and UX enhancements (password visibility, dial code auto-fill) were delivered.
