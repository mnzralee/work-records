Title: Transaction Fee Display Resolution, Smart Currency Formatting System, Enterprise KYC Profile Page Implementation & BFF Architecture Pattern

Objective:
To resolve the critical production issue where transaction fees stopped being captured in transaction history after blockchain block 91 despite correct on-chain calculations, implement a smart currency display formatting system dynamically showing amounts in Qirat (Q) or GX units based on value thresholds to improve user comprehension of small versus large amounts, develop a comprehensive enterprise-grade KYC profile page displaying all verified user registration data in read-only format with mobile-first responsive design, extend backend API services to return complete KYC/address/employment/blockchain identity information via single unified endpoint, implement Backend-For-Frontend (BFF) pattern to resolve CORS issues affecting relationships and profile API calls, and update Kubernetes Ingress configuration to expose missing svc-identity API routes causing production 404 errors.

Work Carried Out:
**Root Cause Analysis of Missing Transaction Fee Records Post-Block 91:** Production monitoring revealed that transaction fees calculated correctly on-chain (confirmed via CouchDB state database queries showing accurate fee amounts in world state) were not appearing in PostgreSQL Transaction table records or frontend transaction history displays after block 91, despite working correctly in earlier blocks (90, 91 showing totalFee values). Raw blockchain block data parsing was performed extracting event payloads from multiple blocks: block 90 showed `"totalFee": 15`, block 91 showed `"totalFee": 7500`, but block 96 and subsequent blocks showed `"totalFee": 0` despite actual fees being charged. Further investigation of block 99 raw payload revealed the chaincode now emitted `"fee": 3750` instead of `totalFee`, indicating a field name change. Timeline analysis established that the chaincode was updated on December 10 at 08:17 changing from `totalFee` to `fee` field naming, the projector fix (commit 36ebb6f) was committed at 11:18 adding fallback logic to handle both field names, but the running projector pod was started at 06:37 using image v2.0.7 which predated the fix, meaning all events after block 91 used the new `fee` field name while the deployed projector still expected `totalFee`.

**Projector Image Rebuild and Deployment for Fee Field Name Handling:** The projector worker was rebuilt from source with npm workspaces compiling the TypeScript codebase (`npm run build --filter=projector`), packaging the fix from commit 36ebb6f which added robust field name handling: `const totalFee = typeof payload.fee === 'string' ? parseFloat(payload.fee) : (payload.fee || payload.totalFee || 0)` supporting both old and new field naming with string-to-number conversion. A new Docker image was created tagged as gx-protocol/projector:2.0.8, imported to the local k3s containerd runtime via `docker save | sudo k3s ctr images import -` command, and the Kubernetes deployment was updated to reference the new image tag. The projector pod was restarted establishing fresh PostgreSQL database connections. The outbox-submitter worker was also restarted to clear stale gRPC connections to Fabric peers (had cached 127.0.0.1:7051 addresses from previous network configurations).

**Transaction Fee Verification with Cross-Border Transfer Testing:** A 1 GX cross-border P2P transfer was executed between Sri Lankan and US test users to verify the complete fee calculation and recording pipeline. Projector logs confirmed successful processing of TransferWithFeesCompleted event from block 100: fromUserId LK FF3 ABL912 0WLUY 6025, toUserId US A12 XYZ789 TEST1 3985, amount 1,000,000 Qirat (1 GX), totalFee 1,500 Qirat (0.15% cross-border rate), senderFee 1,500 (sender-pays-all model), receiverFee 0, netReceived 1,000,000, transactionType P2P, scope CROSS_BORDER. Database verification query `SELECT type, amount, fee, counterparty FROM "Transaction" ORDER BY timestamp DESC LIMIT 3` returned three correctly created records: FEE type with amount 1,500 and counterparty SYSTEM_OPERATIONS_FUND, RECEIVED type with amount 1,000,000 and counterparty showing sender's fabricUserId, SENT type with amount 1,000,000, fee 1,500, and counterparty showing receiver's fabricUserId. This confirmed the full pipeline (chaincode fee calculation → event emission → projector capture → database insert → API query → frontend display) now functioned correctly for all new transactions after block 100.

**Smart Currency Display Formatting System with Q and GX Unit Logic:** A comprehensive currency formatting system was implemented to address user confusion between Qirat (1/1,000,000th of a GX) and GX coin displays, where all amounts were previously displayed as GX with decimals (e.g., 0.001500 X for 1,500 Qirat) making small amounts difficult to comprehend. A centralized `formatCurrency` utility function was created in lib/utils.ts implementing threshold-based formatting logic: amounts below 1,000,000 Qirat (less than 1 GX) display in Qirat format with "Q" suffix using locale-aware integer formatting (e.g., "10Q", "1,500Q", "100,000Q", "999,999Q"), while amounts at or above 1,000,000 Qirat display in GX format with appropriate decimal places and "GX" suffix (e.g., "1 GX", "1.5 GX", "10 GX"). The function handles both number and string inputs with proper NaN handling, uses Intl.NumberFormat for locale-aware thousand separators, and gracefully handles undefined/null values returning "0Q" as default. All currency-related components were migrated to use this shared utility: FormattedBalance.tsx changed "X" branding to "GX", RecentTransactions.tsx and RecentActivity.tsx replaced local formatting with shared utility, NewTransaction.tsx updated formatXCoins to implement Q/GX threshold logic, BalanceCard.tsx implemented dynamic Q/GX display with separate getBalanceValue and getBalanceUnit helper functions for two-part display requirements, and transactions/page.tsx adopted the shared utility for consistent history displays.

**Enterprise KYC Profile Page with Comprehensive Read-Only Data Display:** A complete enterprise-grade profile settings page was implemented at /settings/profile displaying all verified user registration and KYC data in mobile-first responsive design. The page features 7 collapsible sections with smooth expand/collapse animations: Personal Information (name, email, phone, date of birth, gender, nationality, place of birth), National ID (government ID number with issue and expiry dates), Passport (passport number with issuing country and validity dates), Residential Address (full current address with verification status badge), Employment Information (employment status, job title, company name, industry, work email and phone), Blockchain Identity (Fabric User ID with copy button, on-chain registration status, registration timestamp), and Account Status (account creation date, last update timestamp, status badge with color coding, lock information if applicable, admin review details). Each section uses a reusable Section component implementing collapsible container logic with header click handling and expand state management. Individual fields are rendered using an InfoField component providing label/value pairs with optional data masking capabilities (later removed per user feedback), verification checkmarks for KYC-verified fields, "No data" indicators for empty/null fields, and proper date formatting for timestamp fields. The page header includes profile avatar with verification badge, full name display, and Fabric User ID badge, with prominent notices explaining that data is verified and immutable (cannot be edited, contact support for corrections).

**Backend API Extension for Comprehensive Profile Data Retrieval:** The svc-identity getUserProfile service method was significantly enhanced to return all KYC, address, employment, and blockchain identity information in a single API call, eliminating the need for multiple frontend requests. The Prisma query was extended with includes for related tables: addresses table filtered to isCurrent: true with take: 1 limit to fetch only the active residential address, and nationalityCountry table with select for countryName to resolve country code to human-readable name via the Country reference table. The AddressDTO interface was created defining structure for address data including street, city, state, postal code, country, and verification status. The UserProfileDTO interface was comprehensively extended adding 40+ fields covering national ID information (nationalIdNumber, nationalIdIssuedAt, nationalIdExpiresAt), passport details (passportNumber, passportIssuingCountry, passportIssuedAt, passportExpiresAt), employment data (employmentStatus, jobTitle, companyName, industry, workEmail, workPhoneNum), current address as nested AddressDTO object, blockchain identity fields (fabricUserId, onchainStatus, onchainRegisteredAt), account status (isLocked, lockReason, lockedAt), and admin review tracking (reviewedBy, reviewedAt, denialReason). Frontend TypeScript types (types/index.ts) were synchronized with backend DTOs adding UserAddress interface and extending UserProfile with all corresponding fields.

**System Account Friendly Names for Fee Transaction Counterparty Display:** The wallet service's counterpartyName enrichment logic was extended beyond regular user lookup to include human-readable names for system accounts appearing as fee transaction counterparties. A mapping was implemented converting system account identifiers to display-friendly names: SYSTEM_OPERATIONS_FUND → "System Operations Fund" (recipient of all P2P and merchant transaction fees), SYSTEM_TREASURY → "GX Treasury" (central treasury for government operations), SYSTEM_HOARDING_TAX → "Hoarding Tax Pool" (recipient of velocity taxes on idle balances), SYSTEM_LOAN_POOL → "Interest-Free Loan Pool" (source of micro-lending disbursements). This enrichment enables FEE transaction records to display meaningful counterparty names like "To: System Operations Fund" instead of cryptic system identifiers, improving transaction history comprehension and transparency regarding where fees are allocated within the GX Coin economic model.

**Frontend Field Name Mapping Layer for Backend Response Transformation:** A critical discrepancy was identified between backend API field naming conventions and frontend component expectations causing profile page to display "Unknown" for full name, missing phone numbers, and incomplete nationality information. The backend UserProfile model uses firstName, lastName, phoneNum, nationalityCountryCode following database column names, while frontend components expect fname, lname, phone, country following legacy interface definitions. A transformation layer was implemented in the /api/users/me BFF route performing bidirectional field mapping: firstName/lastName mapped to both fname/lname (legacy) and firstName/lastName (new), phoneNum mapped to phone, nationalityCountryCode mapped to country, nationalityCountryName provided for display, and username generated from email prefix when not explicitly provided. This mapping ensures backward compatibility with existing frontend components while gradually migrating to standardized naming conventions.

**Backend-For-Frontend Pattern Implementation for CORS Resolution:** Multiple API calls from the frontend to backend services were experiencing CORS (Cross-Origin Resource Sharing) errors where browser security policies blocked direct requests from https://app.gxcoin.money to https://api.gxcoin.money due to missing Access-Control-Allow-Origin headers in certain response scenarios. The BFF (Backend-For-Frontend) architectural pattern was implemented creating Next.js API routes that proxy requests server-side where CORS restrictions don't apply. Five new API routes were created for relationships management: /api/relationships (GET list, POST create), /api/relationships/[id] (GET detail), /api/relationships/[id]/confirm (POST confirm), /api/relationships/[id]/reject (POST reject). Each BFF route extracts the JWT access token from the user session via getServerSession, forwards the request to the backend identityClient with Authorization header, handles errors appropriately (404, 401, 500), and returns JSON responses to the frontend. The KYR relationships page was updated to call these local /api/relationships routes instead of direct backend URLs, eliminating all CORS errors. Similar BFF routes were created for /api/users/me profile endpoint.

**Kubernetes Ingress Configuration Update for Missing Service Routes:** Production debugging revealed that several svc-identity API endpoints were returning nginx 404 errors despite the service and pods functioning correctly, indicating an infrastructure routing issue rather than application code problem. Port-forwarding directly to the svc-identity service pod confirmed endpoints were operational (curl http://localhost:3001/api/v1/relationships returned proper {"error":"Unauthorized"} instead of nginx 404). Examination of the Kubernetes Ingress resource (gx-backend-ingress in backend-mainnet namespace) revealed it only had path rules for a subset of svc-identity routes (/api/v1/users, /api/v1/wallets, /api/v1/auth), missing routes for recently added features. A kubectl patch command was executed adding 8 missing path rules with Prefix type and svc-identity service backend on port 3001: /api/v1/relationships (KYR relationships management), /api/v1/notifications (in-app notifications), /api/v1/beneficiaries (saved transfer recipients), /api/v1/transfers (P2P transfer commands), /api/v1/commands (outbox command status polling), /api/v1/registration (user registration flow), /api/v1/documents (KYC document upload), /api/v1/validation (real-time field validation). Post-deployment verification confirmed public API now returned proper responses (401 Unauthorized indicating authentication check instead of nginx 404).

**Enhanced Fee Transaction Display with Remark Parsing and Blockchain References:** Transaction history displays across dashboard, send page, and transactions page were enhanced to provide comprehensive fee transparency. FEE transaction remarks following the format "[timestamp] Transaction fee | Transfer: sender -> recipient | Amount: X Q | Rate: Y% (TYPE/SCOPE) | TxID: Z" were parsed to extract fee type (P2P, MERCHANT, B2B, GOVERNMENT), geographic scope (LOCAL, CROSS_BORDER), and fee rate in basis points and percentage. The primary display line for FEE transactions now shows human-readable descriptions like "Cross Border Transfer Fee (0.15%)" instead of generic "Transaction Fee" labels. Secondary text displays the destination pool name (e.g., "To: System Operations Fund") resolved from counterpartyName enrichment. Parent transaction references were added by parsing the onChainTxId field where FEE transactions use the pattern `${parentTxId}_fee`, displaying "Ref: #abc12345" to link fees to their originating transfers. Expanded transaction details were augmented with on-chain transaction ID with copy button for blockchain explorers, block number for audit trail purposes, and parent transaction reference for FEE transactions. The Transaction interface in types/index.ts was extended with optional onChainTxId and blockNumber fields, and the API route passes these fields from backend responses.

**Mobile-First Notification System Unification and Enhancement:** The mobile header notification system was completely overhauled to match the desktop notification experience, replacing the previous basic badge-only implementation with a full-featured notification dropdown. The mobile header now fetches notifications via /api/notifications endpoint with proper loading states, displays dynamic unread count badge (only shown when count > 0), implements mark-as-read functionality on notification tap, renders notification list with type-based color coding (TRANSFER_RECEIVED → green, RELATIONSHIP_INVITED → blue, SYSTEM_ALERT → yellow), includes relative time formatting ("just now", "5m ago", "2h ago", "3d ago"), adds action links for notifications with actionUrl fields enabling direct navigation to relevant pages, provides "View all notifications" link to full /notifications page, and polls for unread count updates every 30 seconds for near real-time updates. Mobile-specific optimizations include dropdown width of calc(100vw - 2rem) for edge-to-edge display with padding, touch-optimized hit areas with active state feedback, and touch-manipulation CSS for improved tap responsiveness. Both mobile and desktop notification dropdowns received dismiss functionality: individual notification X close buttons (appears on hover for desktop, always visible with focus state for mobile), "Clear all" button in dropdown header calling DELETE /api/notifications/clear-all endpoint, automatic unread count decrement when dismissing unread notifications, and hidden scrollbars for cleaner visual appearance. Toast notifications were enhanced with mobile-first design using Sonner Toaster component: top-center positioning for better mobile visibility above content, close button enabled on all toasts for manual dismissal, richColors option for better visual feedback, color-coded backgrounds for success/error/warning/info states, compact text sizing optimized for mobile screens, and styled close button with hover interactions.

**KYR Relationships Page Mobile Improvements and Error Handling Robustness:** The KYR relationships page received comprehensive mobile responsiveness enhancements and error handling improvements. Mobile-first responsive adjustments included bottom padding for mobile navigation clearance (pb-20 sm:pb-8 pattern), scaled text sizes appropriate for mobile viewing (text-[10px] sm:text-xs), reduced spacing on mobile (gap-2 sm:gap-3, p-4 sm:p-6), scaled icons for mobile context (h-4 w-4 sm:h-5 sm:w-5), smaller border radius on mobile (rounded-xl sm:rounded-2xl), and truncated long header titles preventing overflow on narrow screens. Error handling was significantly improved addressing multiple failure scenarios: content-type checking before JSON parsing preventing "Unexpected token '<' in JSON" errors when backend returns HTML 404 pages, HTML response detection with user-friendly messages ("Relationships feature is not yet available. This feature will be enabled soon." for 404, "Server error. Please try again later." for 500), network error handling for "Failed to fetch" scenarios showing "Unable to connect to server. Please check your connection and try again.", proper async error propagation in try-catch blocks with specific error type checking, and graceful degradation when API endpoints are unavailable during deployment or maintenance windows.

**Production Deployment Workflow Across Multi-Region Kubernetes Cluster:** The svc-identity v2.0.7 deployment followed enterprise-grade procedures for the distributed 3-node Kubernetes cluster. The Turborepo build system compiled all 17 packages in the monorepo with proper dependency resolution and TypeScript transpilation. Docker image gx-protocol/svc-identity:2.0.7 was built locally with multi-stage Dockerfile optimizing layer caching. The image was imported to k3s containerd on the local control-plane node (srv1089618 in Kuala Lumpur) using `docker save | sudo k3s ctr images import -` command. Remote nodes (srv1089624 in Phoenix, srv1092158 in Frankfurt) required SSH image transfer: `docker save gx-protocol/svc-identity:2.0.7 | ssh root@<node-ip> "k3s ctr images import -"` piping the image stream over SSH and importing directly on remote containerd runtimes. The Kubernetes deployment was updated with `kubectl set image deployment/svc-identity svc-identity=gx-protocol/svc-identity:2.0.7 -n backend-mainnet` triggering rolling update. Pod rollout was monitored with `kubectl rollout status` verifying all 3 pods (one per node for high availability) successfully transitioned to Running state with the new image. Frontend updates were deployed by pushing commits to the dev branch triggering automated Vercel CI/CD pipeline for build and deployment to production CDN.
