Title: Enterprise Concurrency Architecture, Transaction Fee System Implementation, KYR Trust Score Platform & Email Service Integration

Objective:
To implement enterprise-grade concurrency controls across the full blockchain integration stack (chaincode, outbox-submitter worker, projector worker) addressing critical race conditions and idempotency gaps identified during production load testing, develop and deploy a comprehensive transaction fee system with sender-pays-all fee policy aligned with industry standards (Visa, Mastercard, PayPal), implement the Know Your Relationship (KYR) trust score platform with mobile-first frontend and full-stack API endpoints for relationship verification, enhance transaction history displays with counterparty name resolution, and integrate Resend email service for relationship invitations and OTP delivery.

Work Carried Out:
**Comprehensive Concurrency Analysis and Risk Assessment Across All Layers:** A comprehensive end-to-end concurrency analysis was conducted examining potential race conditions, TOCTOU vulnerabilities, and idempotency failures across the chaincode smart contracts, backend workers (outbox-submitter, projector), and database layer. The outbox-submitter was found to use a non-atomic `findMany` + `updateMany` pattern allowing multiple worker replicas to claim the same OutboxCommand records for processing, leading to duplicate blockchain transaction submissions. The lock timeout of 30 seconds was insufficient for Hyperledger Fabric's Raft consensus under load which can exceed 30 seconds for multi-organization endorsement. The projector worker lacked idempotency checks entirely, meaning event replay after pod crashes would create duplicate Transaction records in PostgreSQL. Multi-instance projector deployments risked concurrent processing of the same blockchain events without distributed locking. The chaincode TransferInternal function had no idempotency protection for retry scenarios. A severity-ranked recommendations document (CONCURRENCY_ANALYSIS.md) was created cataloging all identified issues with code examples and mitigation strategies.

**Outbox-Submitter Pessimistic Locking with PostgreSQL FOR UPDATE SKIP LOCKED:** The outbox-submitter worker was refactored to implement atomic command claiming using PostgreSQL's `FOR UPDATE SKIP LOCKED` row-level locking mechanism. The new implementation uses a Common Table Expression (CTE) to perform atomic SELECT + UPDATE in a single database round-trip: a `claimed` CTE selects PENDING or PROCESSING_RETRY commands with `FOR UPDATE SKIP LOCKED` (skipping rows already locked by other workers), then an UPDATE statement sets status to LOCKED and assigns the workerId. This pattern guarantees disjoint command sets across N worker replicas with no blocking or duplicate claims. Lock timeout was increased from 30 seconds to 5 minutes (300,000ms) to accommodate Fabric's multi-organization endorsement latency. Optimistic locking was added to status update operations, verifying that status is still LOCKED and lockedBy matches the current workerId before committing state transitions to COMMITTED or FAILED, preventing stale updates if lock ownership changed during processing.

**Projector Worker Idempotency Checks and Distributed Locking:** The projector worker was enhanced with comprehensive idempotency protection using a two-tier caching strategy: in-memory LRU cache for O(1) duplicate detection of recently processed transactions, plus database fallback querying the Transaction table for `onChainTxId` uniqueness. The `isTransactionAlreadyProcessed` method checks both cache tiers before allowing any transaction-creating event handler to proceed, preventing duplicate records during block replay scenarios (checkpoint resets, pod restarts mid-batch). PostgreSQL advisory locks were implemented using `pg_try_advisory_lock` with a deterministic lock ID derived from `tenantId` and `channel` name, ensuring only one projector instance processes events at any time across the Kubernetes deployment. Advisory locks are session-scoped and automatically released on pod termination, avoiding orphaned locks. Multi-table updates for WalletFrozen and WalletUnfrozen events were wrapped in Prisma transactions ensuring atomicity across UserProfile and Wallet table modifications.

**Chaincode Idempotency Keys and MVCC Documentation:** The chaincode layer received idempotency support via two new helper functions in helpers.go: `checkAndSetIdempotencyKey` performs atomic check-and-set operations storing idempotency keys on the ledger with transaction ID and timestamp, while `isOperationProcessed` provides read-only verification. The TransferInternal function signature was extended with an optional `idempotencyKey` parameter (variadic string slice allowing backward compatibility), enabling exactly-once semantics for retry scenarios where the outbox-submitter resubmits failed commands. Comprehensive MVCC (Multi-Version Concurrency Control) documentation was added explaining Fabric's built-in double-spending protection: when a transaction reads a balance, Fabric records that read in the transaction's "read set," and at commit time validates no other committed transaction modified that balance since the read occurred, failing with MVCC_READ_CONFLICT if concurrent modifications are detected.

**Transaction Fee System Architecture and TransferWithFees Chaincode Function:** Investigation revealed that P2P transfers via the backend API were not charging fees because they used the `TransferInternal` chaincode function designed for fee-exempt system operations (genesis distribution, loan disbursements, treasury deposits). A new `TransferWithFees` function was implemented in tokenomics_contract.go providing automatic fee calculation based on transaction type (P2P, MERCHANT, B2B, GOVERNMENT) and geographic scope (LOCAL, CROSS_BORDER) using the existing `_calculateTransactionFee` algorithm. Fee exemption logic was encapsulated in `isFeeExemptAccount` and `shouldChargeTransferFee` helpers identifying system pools (SYSTEM_USER_GENESIS_POOL, SYSTEM_LOAN_POOL, SYSTEM_CHARITABLE_POOL, etc.) and country treasury accounts as fee-exempt, while charging progressive fees for user-to-user transfers: P2P local transfers below 3 GX are free, 3-100 GX incur 0.05% fees, above 100 GX incur 0.1% fees, with cross-border P2P transfers charged at 0.15% (0-100 GX) or 0.25% (above 100 GX). The function returns a detailed TransferWithFeesResult struct including amount, totalFee, senderFee, receiverFee, netReceived, and emits a TransferWithFeesCompleted event with comprehensive audit data (ISO 8601 timestamps, transaction type, scope, fee breakdown in both absolute and percentage terms, Fabric transaction ID).

**Fee Policy Refinement to Sender-Pays-All Industry Standard Model:** The transaction fee policy was refined from a 50-50 split (sender pays 50%, receiver pays 50% for P2P transfers) to a sender-pays-all model aligning with industry standards practiced by Visa, Mastercard, PayPal, Wise, and other payment networks where the party initiating the transfer bears 100% of the cost. Configuration constants in consts.go were updated: `FeeDistributionSenderPct` set to 100, `FeeDistributionReceiverPct` set to 0, with comprehensive documentation explaining the rationale. Both the `Transfer` and `TransferWithFees` functions were updated to debit the sender for `amount + totalFee` while crediting the receiver the full `amount` with no deductions. Transaction remarks and event payloads were enhanced with enterprise audit trail data including ISO 8601 timestamps, transaction type classification, geographic scope, fee amount with basis points and percentage, and Fabric transaction ID for blockchain correlation, supporting regulatory compliance and forensic analysis requirements.

**Backend Integration of TransferWithFees with Projector Event Handling:** The outbox-submitter worker's command mapping was updated to use `TransferWithFees` instead of `TransferInternal` for the TRANSFER_TOKENS command type, passing fromUserId, toUserId, amount, optional txType hint, remark, and idempotencyKey as function arguments. The projector worker received a new `handleTransferWithFeesCompleted` event handler processing the enriched event payloads: after idempotency verification, it creates three Transaction records in a single database transaction (SENT for sender showing gross amount and total fee, RECEIVED for receiver showing net amount received after fee deduction, FEE for SYSTEM_OPERATIONS_FUND tracking fee collection), and optionally updates wallet cachedBalance fields if not already synchronized. The handler properly parses the `fee` field from chaincode events (previously expected `totalFee`, added fallback logic for backward compatibility) and correctly implements the sender-pays-all model with receiverFee set to 0.

**Genesis Transaction Backfill for Historical Block Gap:** Production transaction history revealed that users' 500 GX genesis allocations were not visible in their transaction lists. Root cause analysis identified that the projector started from block 89 (stored in ProjectorState.lastBlock), missing all earlier blocks (1-10) containing GenesisDistributed events. Rather than resetting the checkpoint and reprocessing 89 blocks risking other data inconsistencies, a targeted SQL migration was executed to backfill GENESIS transaction records for all wallets lacking them: the query performed a LEFT JOIN between Wallet and Transaction tables filtering for missing GENESIS records, then inserted Transaction rows with type GENESIS, amount 500,000,000 Qirat (500 GX), counterparty SYSTEM_USER_GENESIS_POOL, remark "Genesis allocation: Phase 1 User (500 GX)", timestamp matching wallet creation date, and blockNumber 1. The migration successfully backfilled 5 genesis transactions for testflow@example.com, user1@test.com, user2@test.com, and two e2e test users.

**Transaction Fee Testing and Verification Across Cross-Border Scenarios:** Comprehensive transaction fee testing was conducted using cross-border P2P transfers between a Sri Lankan user (LK fabricUserId) and a US user (US fabricUserId) to verify the fee calculation logic. Test 1: A 10,000 Qirat (0.01 GX) transfer was executed triggering the CROSS_BORDER P2P fee rate of 0.15%, resulting in a 15 Qirat fee split as 7 Qirat sender fee and 8 Qirat receiver fee, correctly recorded in three Transaction records (SENT, RECEIVED, FEE). Test 2: A 5,000,000 Qirat (5 GX) transfer was executed with the same 0.15% CROSS_BORDER rate, producing a 7,500 Qirat total fee split as 3,750 Qirat each, with receiver net amount of 4,996,250 Qirat (5,000,000 - 3,750) correctly recorded. Testing confirmed that the 3 GX free threshold applies ONLY to LOCAL P2P transactions, not cross-border transfers which have no free tier. Blockchain height advanced to block 91 during testing, all fees correctly routed to SYSTEM_OPERATIONS_FUND, and wallet balances accurately reflected gross debits (sender) and net credits (receiver).

**Enterprise KYR Trust Score Platform - Frontend Mobile-First Implementation:** A comprehensive Know Your Relationship (KYR) trust score system was implemented enabling users to build verifiable family and business relationship trees for economic trust assessment. **Enterprise KYR Trust Score Platform - Backend API and Service Layer:** Complete backend API implementation was developed in apps/svc-identity including DTOs (types/dtos.ts) defining RelationType enum (9 types: FATHER, MOTHER, SPOUSE, CHILD, SIBLING, BUSINESS_PARTNER, DIRECTOR, WORKPLACE_ASSOCIATE, FRIEND), RelationshipStatus enum (6 statuses: PENDING, CONFIRMED, REJECTED, DECEASED, DIVORCED, NOT_APPLICABLE), and interfaces for CreateRelationshipRequestDTO, RelationshipDTO, TrustScoreDTO, RelationshipsResponseDTO, RelationshipDetailDTO. The backend wallet service (apps/svc-identity/src/services/wallet.service.ts) was enhanced with counterpartyName enrichment in the TransactionDTO interface, implementing batch lookup of counterparty names via a single Prisma query joining UserProfile on fabricUserId, extracting firstName and lastName fields, building a Map for O(1) name resolution, and filtering out system accounts (SYSTEM, SYSTEM_*) from lookup. The getTransactionHistory method creates a unique set of counterparty IDs from returned transactions, performs the batch UserProfile query with fabricUserId IN clause avoiding N+1 query patterns, and enriches each TransactionDTO with the resolved counterpartyName. **Production Deployments and Critical Bug Fix for Prisma Field Name Mismatch:** The svc-identity service was deployed twice during the session: v2.0.8 with initial counterpartyName enrichment, and v2.0.9 fixing a critical PrismaClientValidationError caused by incorrect field names. The initial implementation used `fname` and `lname` in the Prisma select clause, but the actual UserProfile schema defines `firstName` and `lastName` fields, causing 500 errors on all transaction API calls with "Invalid value for argument" validation failures. The fix corrected the field names, added TypeScript type annotations for the counterpartyProfiles array explicitly declaring `{ fabricUserId: string | null; firstName: string; lastName: string }`, and added a guard clause to skip the Prisma query entirely when the counterpartyIds array is empty (preventing unnecessary database calls for transactions with only system counterparties). Docker images were built for both versions, imported to k3s on all three control-plane nodes (srv1089618 Malaysia, srv1089624 Phoenix, srv1092158 Frankfurt) via SSH image pipes, and deployed to the backend-mainnet namespace with kubectl set image commands. All 3 svc-identity pods successfully rolled out to v2.0.9, API testing confirmed counterpartyName correctly populated as "usertwo test" for regular users and null for system accounts, and frontend transaction displays correctly showed full names in the two-line pattern.

**Endorsement Policy Configuration Fix for Multi-Organization Raft Network:** The outbox-submitter encountered "no combination of peers can be derived which satisfy the endorsement policy" errors when submitting transactions without explicit endorsing organization specifications. The issue stemmed from the chaincode's MAJORITY endorsement policy requiring signatures from both Org1MSP and Org2MSP in the 5-node Raft network (3 Org1 peers, 2 Org2 peers), but the submitTransaction call used automatic peer selection which sometimes chose an insufficient combination. The fix changed `submitTransaction` to `submitTransactionWithOptions` with explicit endorsingOrganizations: ['Org1MSP', 'Org2MSP'] array, guaranteeing that at least one peer from each organization endorses every transaction before submission, satisfying the MAJORITY policy that requires more than 50% of organizations (2 out of 2) to approve.
