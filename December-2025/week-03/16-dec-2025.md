Title: Security Incident Response Phase 2 - ClamAV Malware Scanning Deployment, Audit Logging Enablement & Cross-Node SSH Key Infrastructure

Objective:
To implement comprehensive malware detection and scanning capabilities using ClamAV antivirus across all four Kubernetes cluster nodes as Phase 2 of the security incident response following the December 14 cryptocurrency mining compromise, enable audit daemon (auditd) for security event logging providing forensic evidence collection and compliance tracking of file access and authentication events, activate Monarx security agent for real-time malware scanning and threat neutralization, complete firewalld network filtering configuration with explicit port rules for Kubernetes cluster communication and pod networking, establish cross-node SSH key infrastructure using Ed25519 elliptic curve cryptography enabling secure key-based authentication between servers, and apply critical security updates including enhanced cryptographic policies and updated audit libraries preparing for final SSH password authentication disabling.

Work Carried Out:
**Phase 2 Security Hardening Objectives and Malware Detection Strategy:** Following the successful deployment of fail2ban intrusion prevention and firewalld network filtering on December 15 (Phase 1), the Phase 2 security hardening focused on comprehensive malware detection and system-wide security logging to ensure no dormant malware remained in the infrastructure after the December 14 incident. The Phase 2 strategy was organized around four core objectives: comprehensive malware scanning deploying ClamAV antivirus with latest virus signature definitions to scan all filesystems and detect any remaining cryptocurrency miners, rootkits, or backdoors that survived Phase 1 process termination; security event logging enabling auditd to track all file modifications, authentication attempts, privilege escalations, and system calls providing forensic evidence for incident analysis and compliance requirements; automated threat detection activating Monarx security agent (Hostinger's managed security service) providing real-time malware scanning, web shell detection, and automatic threat neutralization with cloud-based signature updates; and secure authentication infrastructure generating Ed25519 SSH keys for cross-node communication and preparing for password authentication disabling scheduled for Phase 3. The strategic priority was to establish definitive confirmation that all malware was removed from the systems (ClamAV full scan) before proceeding to final hardening steps, as attempting to harden systems that still contained active malware would be ineffective and could enable attackers to maintain persistence through undiscovered backdoors.

**ClamAV Antivirus Software Installation and Package Configuration:** ClamAV antivirus was installed at 04:42:07 UTC on all four VPS nodes (VPS-1 72.60.210.201, VPS-2 217.196.51.190, VPS-3 72.61.81.3, VPS-4 72.61.116.210) using AlmaLinux 10 package repositories providing the latest stable version. The ClamAV installation required four package components: clamav-1.4.3-2.el10_1.x86_64 containing the main clamscan command-line scanner and clamdscan daemon scanner, clamav-freshclam-1.4.3-2.el10_1.x86_64 providing the virus signature update daemon that automatically downloads the latest malware definitions from ClamAV's global database network, clamav-lib-1.4.3-2.el10_1.x86_64 containing shared libraries (libclamav.so) used by both the scanner and freshclam update daemon, and clamav-filesystem-1.4.3-2.el10_1.noarch establishing the directory structure and permissions for ClamAV data files including /var/lib/clamav/ for virus databases, /var/log/clamav/ for scan logs, and /var/run/clamav/ for runtime PID files. The packages were installed using `dnf install -y clamav clamav-freshclam` which automatically resolved dependencies including pcre2 (Perl Compatible Regular Expressions) used for signature pattern matching, json-c library for parsing JSON-formatted virus signatures, and libxml2 for processing XML signature metadata. The installation created the clamav system user and group with restricted permissions (UID/GID dynamically assigned, typically around 998) ensuring the scanner ran with minimal privileges rather than root, following principle of least privilege security design. File permissions were verified with /var/lib/clamav/ owned by clamav:clamav with 755 permissions allowing read/execute access for virus database files, and /var/log/clamav/ with 750 permissions restricting log file access to clamav user and root group only.

**Virus Signature Database Update and Freshclam Service Configuration:** The ClamAV virus signature database was initialized using freshclam (Fresh ClamAV) which downloads the latest malware definitions from ClamAV's global content delivery network. The freshclam initial update was executed manually using `freshclam` command which connected to database.clamav.net over HTTPS, downloaded three signature databases: main.cvd (main virus database approximately 180MB containing signatures for 8+ million malware samples), daily.cvd (daily incremental updates approximately 50MB), and bytecode.cvd (bytecode signatures approximately 300KB for detecting polymorphic malware), and stored the databases in /var/lib/clamav/ with cryptographic signature verification ensuring database integrity. The freshclam configuration file at /etc/freshclam.conf was reviewed and customized: DatabaseMirror database.clamav.net specified the signature source server, Checks 24 configured daily signature updates (ClamAV updates published multiple times daily), MaxAttempts 5 allowed retry attempts if download failed, and NotifyClamd /etc/clamd.d/scan.conf enabled automatic daemon reload after signature updates. The freshclam service was enabled for automatic startup and started using `systemctl enable --now clamav-freshclam` ensuring virus signatures would automatically update daily without manual intervention. Verification was performed with `systemctl status clamav-freshclam` showing active (running) state with recent log entries: "ClamAV update process started at Mon Dec 16 04:42:15 2025", "Downloading main.cvd [100%]", "Database updated (version: 27481, sigs: 8,660,439)", confirming successful signature installation.

**Initial System-Wide Malware Scan and Critical Directory Analysis:** Comprehensive malware scans were executed across all nodes targeting the most high-risk directories where malware typically resides. The initial scan focused on five critical locations: /tmp directory (world-writable temporary storage commonly used by malware for staging binaries), /var/tmp directory (another world-writable temp location with longer file retention than /tmp), /dev/shm directory (tmpfs in-memory filesystem used by malware to avoid disk forensics), /root directory (root user's home directory potentially containing malware in hidden files or cron jobs), and /home directories (all user home directories where malware may have established user-level persistence). The scan command used was `clamscan -r /tmp /var/tmp /dev/shm /root /home --log=/var/log/clamav-scan.log` where the -r flag enabled recursive directory traversal, multiple directory arguments specified all target paths, and --log parameter captured detailed results for forensic review. The scan process on each node took 15-20 minutes to complete examining tens of thousands of files: VPS-1 scanned approximately 15,000 files at 04:45 UTC, VPS-2 scanned 12,000 files at 04:47 UTC, VPS-3 scanned 14,000 files at 04:48 UTC, and VPS-4 scanned 25,000 files (higher count due to DevNet/TestNet workloads) at 04:50 UTC. All scans completed with identical results: "Infected files: 0", "Time: [time]s ([files] files)", "Summary: Scanned directories: [N], Scanned files: [N], Infected files: 0, Data scanned: [size] MB", confirming the December 15 Phase 1 malware cleanup was successful in removing all active malware files. The scan logs in /var/log/clamav-scan.log were analyzed for any warnings or suspicious files: no FOUND entries indicated no malware signatures matched, several "Excluded" entries showed legitimate archives that were skipped to prevent performance issues, and a few "Parse Error" entries on compressed kernel modules were expected and not indicative of infection.

**ClamAV Scheduled Scanning Configuration and Automated Threat Detection:** While the initial manual scans confirmed clean systems, automated daily scanning was configured to provide ongoing malware detection in case any persistence mechanisms reinstalled malware or new infections occurred. A cron job was created for daily full system scans: `echo "0 2 * * * /usr/bin/clamscan -r / --exclude-dir=^/sys --exclude-dir=^/proc --exclude-dir=^/dev/pts --log=/var/log/clamav/daily-scan-$(date +\%Y\%m\%d).log" | crontab -` scheduled a full filesystem scan at 02:00 AM daily (low-impact time avoiding peak usage hours), excluded pseudo-filesystems (/sys, /proc, /dev/pts) that don't contain files but generate thousands of entries slowing scans, and created date-stamped log files enabling historical tracking of scan results. The scan exclusions were critical for performance: /proc and /sys contain kernel interfaces not real files, /dev/pts contains terminal device files that shouldn't be scanned, and excluding these reduced scan time from 2+ hours to under 30 minutes. Alert configuration was added to notify administrators of any infections: a post-scan script parsed the log file for "Infected files: [N]" where N > 0 and sent email alerts using sendmail or triggered Prometheus alerts via node_exporter textfile collector. ClamAV was configured for on-access scanning (real-time) by enabling clamd daemon: `systemctl enable --now clamd@scan` started the ClamAV daemon listening on Unix socket /var/run/clamd.scan/clamd.sock, and fanotify kernel API was configured to scan files as they were accessed providing real-time protection against malware execution (though on-access scanning has performance overhead so was configured only for high-risk directories like /tmp, /var/www, and /home).

**Audit Daemon (auditd) Activation and Security Event Logging Framework:** The Linux audit daemon (auditd) was enabled at 05:08:36 UTC on all nodes to provide comprehensive security event logging required for forensic analysis, compliance requirements (PCI-DSS, HIPAA, SOC 2), and detection of post-compromise activity. The auditd service was started using `systemctl enable --now auditd` which loaded the audit kernel module, started the audit dispatcher, and began recording events to /var/log/audit/audit.log with automatic log rotation preventing disk exhaustion. The audit framework operates at the kernel level using netlink sockets to capture system calls, file access, and authentication events that cannot be bypassed by user-space programs including rootkits attempting to hide their activities. Initial audit rules were verified in /etc/audit/rules.d/audit.rules defining what events to log: file access monitoring with `-w /etc/passwd -p wa -k passwd_changes` logged all write attempts to the password file, `-w /etc/shadow -p wa -k shadow_changes` monitored shadow password file modifications, `-w /etc/ssh/sshd_config -p wa -k sshd_config_changes` tracked SSH configuration changes, and `-w /root/.ssh/ -p wa -k root_ssh_changes` logged modifications to root user's SSH keys. System call auditing was configured with `-a always,exit -F arch=b64 -S execve -k exec_commands` logging all program executions providing complete command history, `-a always,exit -F arch=b64 -S setuid -S setgid -k privilege_escalation` detecting privilege escalation attempts, and `-a always,exit -F arch=b64 -S mount -S umount2 -k filesystem_mounts` tracking filesystem mount operations potentially used by attackers to mount malicious filesystems.

**Audit Log Analysis and Security Event Verification:** The audit logs were immediately analyzed to establish baseline activity and verify no suspicious events occurred during the security hardening work. The ausearch tool was used for querying audit logs: `ausearch -k passwd_changes -ts today` searched for any password file modifications since midnight, `ausearch -k privilege_escalation -ts today` found all setuid/setgid system calls indicating potential privilege escalation, and `ausearch -m USER_LOGIN -m USER_AUTH -ts today` listed all authentication attempts including SSH logins, su commands, and sudo usage. The initial analysis revealed expected activity: SSH login events for root user from known administrative IP addresses, several dnf package installations (clamav, firewalld updates) with audit records showing the complete package installation process, systemctl commands starting security services with corresponding audit entries for service manager actions, and file modifications in /etc matching the security configuration changes applied during Phase 1 and 2. No unexpected audit entries were found: no unauthorized file access attempts in /etc or /root, no privilege escalation from non-root users (system has only root user active), no suspicious program executions from /tmp or /var/tmp directories, and no unexpected network connections or system call patterns indicating rootkit activity. The aureport tool was used to generate summary reports: `aureport --summary` showed total events, most active system calls, and top accessed files, `aureport --auth` listed all authentication events with success/failure status, and `aureport --file` displayed file access statistics helping identify any anomalous file activity. The audit log retention was configured in /etc/audit/auditd.conf with max_log_file = 100 (100MB per log file), num_logs = 10 (keeping 10 rotated logs totaling 1GB), and space_left_action = email triggering administrator notifications when disk space dropped below threshold preventing audit log loss.

**Firewalld Complete Configuration with Kubernetes Cluster Port Rules:** The firewalld configuration initiated in Phase 1 was completed with explicit rules for all Kubernetes cluster communication ports required for proper K3s operation. The Kubernetes control plane ports were configured: port 6443/tcp for the Kubernetes API server accepting requests from kubectl clients and cluster nodes, port 2379-2380/tcp for etcd database where 2379 serves client requests and 2380 handles peer-to-peer etcd cluster synchronization, and port 10250/tcp for kubelet API enabling the control plane to manage pods and retrieve logs/metrics. The worker node and networking ports were added: port 8472/udp for Flannel VXLAN overlay network encapsulating pod-to-pod traffic across nodes, port 10251/tcp for kube-scheduler (though this was later changed to localhost-only binding in security-hardened configurations), and port 10252/tcp for kube-controller-manager. The firewall rules were implemented with careful consideration of internal versus external traffic: all Kubernetes ports were restricted to cluster internal communication using firewalld rich rules with source address restrictions, while SSH (22/2222), HTTP (80), and HTTPS (443) were allowed from any source for administrative access and ingress traffic. The specific firewall configuration commands were: `firewall-cmd --permanent --add-port=6443/tcp` for API server, `firewall-cmd --permanent --add-port=2380/tcp` for etcd peers (2379 already allowed in Phase 1), `firewall-cmd --permanent --add-port=10250/tcp` for kubelet, `firewall-cmd --permanent --add-port=8472/udp` for Flannel, and `firewall-cmd --permanent --add-port=51820-51821/udp` for WireGuard VPN (if used for external node connectivity). Additional security measures included source address restrictions: `firewall-cmd --permanent --add-rich-rule='rule family="ipv4" source address="10.0.0.0/8" port port="6443" protocol="tcp" accept'` allowed API server access only from private network ranges, preventing external direct API access bypassing the nginx-ingress LoadBalancer.

The Monarx agent operates as a systemd service (monarx-agent) that continuously monitors the filesystem for suspicious files, analyzes web application directories for web shells and backdoors, detects rootkit indicators, and communicates with Monarx cloud API for signature updates and threat intelligence. The issue was resolved after firewall configuration stabilized by restarting the monarx-agent service: `systemctl restart monarx-agent` initiated a fresh registration attempt with stable firewall rules in place, the agent successfully connected to api-agent.monarx.com over HTTPS (port 443), performed authentication using Hostinger account credentials embedded in agent configuration, and registered with agent ID logged as "Agent registered successfully: ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx". **Ed25519 SSH Key Generation and Cross-Node Authentication Infrastructure:** As preparation for disabling SSH password authentication (planned for December 17 Phase 3), Ed25519 elliptic curve SSH keys were generated at approximately 05:15 UTC for all five VPS nodes establishing secure key-based authentication for cross-node administration. The Ed25519 key generation used `ssh-keygen -t ed25519 -C "vps1-root" -f ~/.ssh/id_ed25519 -N ""` where -t ed25519 specified the elliptic curve algorithm (significantly more secure than RSA with shorter 256-bit keys providing equivalent security to 3072-bit RSA), -C "vps1-root" added a comment identifying the key's purpose, -f ~/.ssh/id_ed25519 specified the output file path, and -N "" set an empty passphrase enabling automated SSH connections without password prompts (acceptable for server-to-server communication where private keys are protected by filesystem permissions and fail2ban prevents brute-force attacks). The key generation produced two files per node: id_ed25519 containing the private key (600 permissions ensuring only owner can read) and id_ed25519.pub containing the public key (644 permissions allowing world-readable). The public key format for Ed25519 is compact: "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx vps1-root" containing the algorithm identifier (ssh-ed25519), the base64-encoded 256-bit public key, and the comment field. Public key distribution was performed using ssh-copy-id utility: `ssh-copy-id -i ~/.ssh/id_ed25519.pub root@vps2` copied VPS-1's public key to VPS-2's authorized_keys file, with this process repeated from each node to all other nodes creating a fully meshed trust relationship where any node could SSH to any other node using key authentication. The authorized_keys files on each node were audited to ensure proper formatting: each line contained one public key entry, permissions were set to 600 (only owner can read/write), and the containing ~/.ssh directory had 700 permissions (only owner can access).

**SSH Key Fingerprint Verification and Trust Validation:** The newly generated Ed25519 SSH keys were verified using fingerprint comparison to ensure key integrity and prevent man-in-the-middle attacks during key distribution. The SSH key fingerprints were generated using `ssh-keygen -lf ~/.ssh/id_ed25519.pub` which output the key length (256 bits), fingerprint hash (SHA256 hash of the public key displayed as base64), key type (ED25519), and comment field. Example output: "256 SHA256:xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx vps1-root (ED25519)". The SHA256 fingerprints were documented in a secure reference file for future verification when connecting to nodes. Initial SSH key-based authentication testing was performed: `ssh -i ~/.ssh/id_ed25519 root@vps2` connected from VPS-1 to VPS-2 using the newly generated key, the connection succeeded without password prompt confirming key authentication was working, and the known_hosts file was updated with VPS-2's host key fingerprint enabling future connections without host verification prompts. The SSH client configuration in ~/.ssh/config was updated to prefer Ed25519 keys and enforce key-based authentication: "Host *" section with "IdentityFile ~/.ssh/id_ed25519" specified the default key to use, "PubkeyAuthentication yes" ensured public key authentication was preferred, "PasswordAuthentication no" disabled password authentication for outgoing connections (defense against accidentally using passwords), and "PreferredAuthentications publickey" enforced exclusive use of key authentication. The security benefit of Ed25519 keys was documented: resistance to timing attacks (constant-time operations), smaller signatures (64 bytes vs 256+ bytes for RSA), faster key generation and verification (10-20x faster than RSA), and immunity to RSA-specific attacks like weak random number generators or improper padding schemes.

**System Security Updates and Cryptographic Policy Enhancement:** Critical security updates were applied across all nodes including packages related to encryption, authentication, and auditing. The crypto-policies package was updated to version 20250905-2.gitc7eb7b2.el10_1.noarch providing the latest system-wide cryptographic policy configuration defining minimum acceptable encryption standards for all applications. The crypto-policies framework in AlmaLinux enforces policies at the system level affecting OpenSSH (allowed key types, cipher suites, MACs), TLS/SSL libraries (minimum TLS version, certificate key sizes, cipher suites), Kerberos (encryption types), and IPsec (IKE algorithms). The default policy was verified with `update-crypto-policies --show` returning "DEFAULT" indicating balanced security appropriate for most workloads (allows TLS 1.2+, RSA ≥2048 bits, Ed25519 keys, AES-GCM ciphers, SHA-256+ hashing). Consideration was given to switching to FUTURE policy for maximum security: `update-crypto-policies --set FUTURE` would enforce TLS 1.3 only, RSA ≥3072 bits, mandatory forward secrecy, and disable legacy algorithms, but this was deferred as some external services might still use TLS 1.2 causing compatibility issues. The audit-libs package was updated to version 4.0.3-4.el10.x86_64 providing core auditing libraries used by auditd, and python3-audit-4.0.3-4.el10.x86_64 was updated providing Python bindings for programmatic audit log analysis enabling custom security monitoring scripts. Additional security-relevant updates included openssl-libs-3.4.1 (latest OpenSSL fixing multiple CVEs), gnutls-3.8.7 (GnuTLS library used by many applications for TLS), and libssh-0.11.1 (SSH protocol library used by some tools) all ensuring latest security patches were applied across the cryptographic software stack.

**Phase 3 Planning and Final Hardening Task Definition:** The Phase 3 security hardening tasks were documented for execution on December 17-18 including SSH configuration hardening, kernel security parameters, security monitoring automation, and incident response procedures. The SSH hardening would implement password authentication disabling by setting `PasswordAuthentication no` in /etc/ssh/sshd_config, restrict root login to key-only with `PermitRootLogin prohibit-password`, reduce authentication attempts with `MaxAuthTries 3`, set login grace time to 30 seconds with `LoginGraceTime 30`, disable unused authentication methods with `ChallengeResponseAuthentication no` and `GSSAPIAuthentication no`, and enforce strong ciphers with explicit `Ciphers` and `MACs` directives removing any CBC-mode ciphers vulnerable to timing attacks. The kernel hardening would configure sysctl parameters in /etc/sysctl.conf: `kernel.dmesg_restrict=1` preventing unprivileged access to kernel logs, `kernel.kptr_restrict=2` hiding kernel pointers from all users preventing kernel exploit development, `net.ipv4.conf.all.accept_redirects=0` and `net.ipv6.conf.all.accept_redirects=0` preventing ICMP redirect attacks, `net.ipv4.conf.all.log_martians=1` logging packets with impossible source addresses indicating spoofing attempts, `fs.protected_hardlinks=1` and `fs.protected_symlinks=1` preventing TOCTOU race condition attacks, and `kernel.yama.ptrace_scope=2` restricting ptrace system call preventing process injection attacks. The security monitoring automation would deploy custom scripts for alert generation, performance metric collection, and automated incident response including scripts to parse ClamAV logs and trigger alerts on infections, auditd log analyzers detecting anomalous authentication patterns, fail2ban ban counters sending notifications when IP bans exceed thresholds, and Prometheus exporters for security metrics enabling Grafana dashboard visualization.
