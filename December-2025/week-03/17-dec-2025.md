To transition from reactive incident response to proactive security operations by deploying comprehensive automated security scanning infrastructure across all four VPS nodes, establishing centralized security reporting mechanisms with structured logging and alerting capabilities, configuring development environment prerequisites to enable Claude Code CLI integration for automated development workflows, and implementing scheduled security monitoring through cron job automation while documenting the current security posture and identifying remaining hardening tasks including SSH password authentication disablement. ## Work Carried Out

**Comprehensive Security Scanning Script Development and Deployment.** The day began with the critical task of establishing automated security monitoring to replace the manual scanning procedures that had been performed during the incident response phases. The security scanning script was developed and deployed to `/root/security-scripts/security-scan.sh` on VPS-2 (217.196.51.190) as the initial test node, with the first automated scan executing at 08:32:19 UTC on December 17, 2025. The script was designed with fifteen comprehensive security check sections to provide complete coverage of potential attack vectors and system compromise indicators. Section 1 implemented crypto miner detection through process name pattern matching against known mining software signatures including xmrig, minerd, kinsing, and other common cryptocurrency mining binaries that had been discovered during the December 14 incident. Section 2 performed process analysis to identify any processes consuming excessive CPU or memory resources, as the original malware detection had been triggered by 100% CPU utilization from mining operations. Section 3 implemented file system security scanning to detect world-writable files that could enable privilege escalation, as well as SUID and SGID binaries that could be exploited for unauthorized access. Section 4 provided network security analysis including enumeration of listening ports using netstat and ss commands, identification of established connections to external IP addresses, and detection of any suspicious outbound connections that might indicate command-and-control communication. Section 5 implemented user account security verification including detection of any additional UID 0 accounts beyond root, enumeration of users with sudo access, and identification of accounts with empty passwords or suspicious shell configurations. Section 6 performed comprehensive SSH security auditing to verify configuration hardening status including root login settings, password authentication status, MaxAuthTries limits, and failed login attempt counts from the authentication logs. Section 7 reviewed all scheduled tasks including user crontabs and system cron directories to detect any persistence mechanisms that attackers might have established. Section 8 provided container security analysis for Docker and Kubernetes environments including detection of privileged containers, containers running as root, and containers with host filesystem mounts that could enable container escape attacks. Section 9 integrated with the ClamAV antivirus scanner deployed during Phase 2 to perform on-demand malware scanning of critical system directories. Section 10 implemented rootkit detection through file system and process hiding detection techniques, verification of system binary integrity, and detection of kernel module tampering. Section 11 verified the operational status of all security services including fail2ban, firewalld, auditd, and the Monarx agent to ensure continuous protection. Section 12 performed system integrity verification through package verification using rpm commands to detect any modifications to installed system files that might indicate compromise. Section 13 analyzed system logs for security-relevant events including failed authentication attempts, privilege escalation activity, and suspicious system calls captured by auditd. Section 14 verified backup system status and integrity to ensure recovery capabilities in case of future incidents. Section 15 generated a comprehensive security summary with color-coded status indicators and alert generation for any findings requiring immediate attention.

**First Automated Security Scan Execution and Analysis.** The first execution of the security scanning script on VPS-2 at 08:32:19 UTC produced a detailed security report that was saved to the centralized reporting directory. The scan report header displayed the hostname (srv1089624), IP address (217.196.51.190), operating system (AlmaLinux 10.0 Purple Lion), and scan timestamp. Section 1 crypto miner detection returned OK status with the message "No crypto miner processes found" confirming that the malware removal performed during Phase 1 and Phase 2 had been successful and no mining processes had been reestablished. Section 6 SSH security audit returned mixed results with two warnings and two OK statuses. The first warning indicated "SSH: Root login is permitted (consider using 'prohibit-password')" identifying that the PermitRootLogin directive in sshd_config was still set to 'yes' rather than 'prohibit-password' which would allow root login only with key-based authentication. The second warning stated "SSH: Password authentication is enabled (consider disabling)" identifying that PasswordAuthentication was still set to 'yes' despite the Ed25519 key infrastructure being deployed during Phase 2. The OK statuses confirmed that MaxAuthTries was properly set to 3 to limit brute-force attempts, and that only 9 failed SSH login attempts had been recorded in the last 24 hours, a significant reduction from the hundreds of attempts seen before fail2ban deployment. The scan also identified one world-writable file at `/etc/monarx-agent.conf` which was flagged for remediation as it could potentially allow unauthorized modification of the Monarx agent configuration by non-privileged users.

**Security Reports Directory Structure Establishment.** A centralized security reporting structure was created to maintain historical scan data and enable trend analysis of security posture over time. The directory `/root/security-reports/` was created on each VPS node to store individual scan outputs with standardized naming conventions. Each scan report was saved with the filename format `scan-{hostname}-{YYYYMMDD}-{HHMMSS}.txt` enabling easy identification and chronological sorting of scan history. The first scan output was saved as `scan-srv1089624-20251217-083219.txt` representing the 08:32:19 UTC execution. A subsequent noon scan scheduled by cron created `scan-srv1089624-20251217-120001.txt` confirming the automated scheduling was operational. Alert files with the naming pattern `ALERT-*.txt` were configured to be generated when any scan section returned critical findings requiring immediate attention. The logging configuration directed all scan execution output to `/var/log/security-scans/security-scan.log` providing a consolidated log file for troubleshooting scan execution issues and maintaining an audit trail of all security monitoring activity. A retention policy of 30 days was established for scan history, with older reports automatically purged to manage storage consumption while maintaining sufficient historical data for security trend analysis.

The installation began with the EPEL repository enablement at 06:47:23 UTC through the installation of epel-release version 10-7.el10_1, providing access to additional packages not available in the base AlmaLinux repositories. The git package version 2.47.3-1.el10_0 was installed at 11:02:01 UTC to enable repository management and version control operations for the GX Protocol codebase. Node.js version 22.19.0-2.el10_0 was installed at 11:02:34 UTC along with nodejs-npm version 10.9.3 at 11:02:33 UTC, providing the JavaScript runtime environment and package manager required for Claude Code CLI and other Node.js-based development tools. The installation timestamps were extracted from the dnf history logs and package installation metadata to document the precise sequence of package deployments. The purpose of this development environment was to enable automated development workflows where Claude Code could assist with code generation, review, and modification tasks directly on the VPS-2 server, reducing the need to transfer code between local development environments and the production infrastructure.

**Cron Job Automation Configuration.** Automated 12-hour security scanning was configured through cron job entries on each VPS node to ensure continuous monitoring without manual intervention. The crontab was edited using the `crontab -e` command to add the security scan schedule entry: `0 */12 * * * /root/security-scripts/security-scan.sh >> /var/log/security-scans/cron.log 2>&1`. This cron expression specifies execution at minute 0 of every 12th hour, resulting in scans at 00:00 UTC (midnight) and 12:00 UTC (noon) each day. The output redirection to `/var/log/security-scans/cron.log` with stderr merged to stdout ensures all execution output including any error messages is captured for troubleshooting. The 12-hour interval was chosen to balance between early detection of security issues and minimizing system resource consumption from frequent full-system scans. This schedule ensures that any security compromise would be detected within a maximum of 12 hours, allowing rapid response before significant damage could occur.

**VPS-Specific Security Scan Script Development.** Recognizing that VPS-2 serves a unique role as the development and build environment within the GX Protocol infrastructure, a specialized security scan script was created as `vps2-security-scan.sh` with additional checks specific to the development workload. The VPS2-specific checks included Docker container security verification to ensure development containers were not running with excessive privileges, verification of known Docker images and detection of any unauthorized container deployments. Node.js application security checks were implemented to verify package integrity through npm audit, detection of known vulnerable dependencies, and verification that development servers were not exposed to external network access. Build environment integrity verification ensured that build tools and compilers had not been tampered with, critical for supply chain security to prevent injection of malicious code during application builds. These additional checks complemented the base security scan and addressed the specific attack surface introduced by development tooling.

The journal matches confirmed monitoring of the sshd.service systemd unit. The actions section showed 0 currently banned IP addresses with approximately 10 total banned IPs since deployment, indicating that fail2ban had successfully identified and blocked multiple brute-force attack sources. The observation that attack volume had decreased since fail2ban deployment suggested that attackers had either moved on to easier targets or had exhausted their IP address pool through progressive banning. The zero currently banned IPs indicated that either the ban periods had expired (configured as 1 hour during Phase 1) or that active attacks had ceased. Importantly, the logs confirmed no successful unauthorized SSH logins had occurred since the security hardening was implemented.

**Security Posture Assessment and Priority Documentation.** Based on the automated scan results and manual verification, the current security posture was documented with identified issues prioritized for remediation. The SSH password authentication remaining enabled was classified as HIGH priority with target remediation date of December 18-21, as this represented the most significant remaining attack vector. The Ed25519 key infrastructure had been deployed during Phase 2, and all that remained was to modify the sshd_config to disable password authentication and configure PermitRootLogin to prohibit-password mode. The world-writable `/etc/monarx-agent.conf` file was classified as LOW priority with target remediation on December 18, as the risk was limited to potential configuration tampering by local users who already had shell access. The SELinux disabled status was classified as MEDIUM priority with remediation deferred to a future maintenance window, as enabling SELinux would require comprehensive testing of all applications to ensure proper labeling and policy configuration. The remediation plan for December 18 was documented to include continuing security scan monitoring, preparing SSH hardening configuration templates, testing key-only authentication with the deployed Ed25519 keys, and documenting the security baseline for ongoing operations.
