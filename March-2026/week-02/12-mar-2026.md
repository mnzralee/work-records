Title: ESLint-Driven Component Architecture Refactoring, E2E Admin Promotion Verification, RBAC API Response Bug Discovery and Fix, Security Hardening of Admin Creation Pipeline, OAuth2 PKCE SSO Flow Debugging, Cross-App Logout Architecture, and Hub Sidebar Integration

Objective:
To resolve ESLint and React Compiler strictness violations through progressive architectural refactoring of a monolithic admin creation page into a well-structured component hierarchy, execute file-by-file commits with gap analysis verification, complete end-to-end Playwright testing of the admin promotion flow uncovering and fixing a systemic API response wrapping bug, harden the admin creation pipeline by closing four security gaps including legacy endpoint disablement and role hierarchy enforcement, debug and resolve a five-layer OAuth2 PKCE SSO flow from the Hub to the administration portal, fix a React hydration mismatch in the SSO callback, design and implement a cross-application logout architecture using redirect chains, and integrate the Hub link into the administration portal sidebar.

Work Carried Out:

**ESLint Remediation as Progressive Architectural Refactoring:** What initially appeared to be routine linter error resolution evolved into four rounds of progressively deeper architectural refinement driven by the React Compiler's strictness rules. The admin creation page began as a 427-line monolith that exceeded the 300-line maximum and had a cyclomatic complexity of 14 against a maximum of 10, alongside six errors and twenty-eight warnings. The first round extracted visual components into a dedicated components directory, reducing the file to 397 lines but exposing memoisation dependency errors where optional chaining on sub-properties produced unstable dependency references that the React Compiler could not optimise. The second round created separate section components and fixed the memoisation dependencies to use full objects, but this revealed setState-in-effect errors where two useEffect blocks called setState synchronously — one auto-selecting a single search result and another clearing selection on search change. The third round eliminated these useEffect anti-patterns by replacing the auto-select effect with a derived value pattern using a pure function that computes the effective participant from state without effects, and replacing the clear-on-change effect by handling it directly in the search change callback. An attempted useRef approach was rejected by the React Compiler's refs rule prohibiting ref access during render, as refs are side-effect channels and not render-time data. The fourth round extracted a custom permissions editor hook that uses render-time state adjustment — tracking the last synced role in a state variable and comparing on each render rather than using an effect to sync derived state from props. This dropped the main component to a complexity of 9, under the maximum of 10. The final architecture comprises six files totalling 580 lines: a 133-line orchestrator page, a 63-line participant card, a 52-line permission checkbox group, a 114-line search section, a 134-line role permissions section, and an 84-line permissions editor hook. All six original errors and twenty-two of twenty-eight warnings were resolved, with the remaining six being pre-existing security object injection warnings unrelated to the changes. An additional fix in the admin hooks file removed explicit generic type parameters from API calls that were causing TypeScript to infer response data as always truthy, triggering unnecessary-condition warnings on null-coalescing fallbacks.

**Enterprise-Discipline File-by-File Commits and Gap Analysis:** Nineteen commits were made across three repositories with each file receiving its own commit and descriptive message explaining the change and its purpose. Seven backend commits covered the extended promotable search output, repository search additions, custom permissions in the promotion schema and use case, the repository port method addition, controller simplification, and module registration. Nine frontend commits covered the type definitions, admin hooks, four new component files, the permissions editor hook, the rewritten page orchestrator, and the button label update. Three parent repository commits recorded the submodule updates and session documentation. All nineteen commits passed lint-staged hooks on the first attempt. Before proceeding to end-to-end testing, a comprehensive gap analysis was executed using three parallel verification tracks — backend, frontend, and integration — confirming one hundred percent implementation completeness across all eleven planned backend items and all eighteen planned frontend features, with API contracts verified as aligned between the two layers.

**E2E Playwright Testing and Permissions Loading Bug Discovery:** With gap analysis confirming completeness, the end-to-end test flow was launched to promote a test participant to the highest administrative role. Initial testing with a standard admin account correctly demonstrated that the role hierarchy function prevents same-rank assignment — a super administrator cannot assign the super administrator role because the assigner's rank must be strictly higher than the target. After logging in as the super owner account with the highest rank, the search, participant card rendering, and role selection all functioned correctly, with the single-result auto-selection working exactly as designed via the derived value pattern. However, expanding the permissions customisation section revealed a critical bug: the permission count displayed zero instead of the expected eighty-plus permissions, the role match indicator showed a custom label instead of the correct role name, and no permission checkboxes rendered. Investigation traced the data flow through the entire chain — from the backend returning a standard wrapped response with success and data fields, through the Next.js rewrite proxy passing the response unmodified, through the Axios client with its identity response interceptor, to the fetch function returning the full wrapper object, into the TanStack Query cache, and finally to the permissions editor hook accessing a permissions property at the top level of the response where it does not exist. The actual permissions array was nested one level deeper inside the data property of the wrapper. The TypeScript type definitions claimed a flat shape but the runtime value was wrapped, meaning the types suppressed compiler warnings without enforcing runtime correctness.

**Systemic RBAC API Response Unwrapping Fix:** The root cause — fetch functions returning the API response wrapper rather than the inner data — was systemic across all eight RBAC fetch functions. The fix applied the response unwrapping pattern already used in the admin hooks file to all eight functions: permissions listing, category-filtered permissions, single permission details, role summaries, role details with permissions array, the cross-role permission matrix, the current admin's effective permissions including a type mapping fix to handle both legacy and current backend response shapes, and specific admin permission summaries. An unused type import was also removed. This fix not only resolved the new promotion page but also repaired the existing roles management page and its permissions editor component, which had contained the same latent bug causing silently empty permission lists that had gone unnoticed because the page was rarely used.

**Successful Admin Promotion E2E Verification:** After deploying the response unwrapping fix, the Playwright flow was re-executed. The participant search returned correct data, the role defaulted to the highest administrative level, and expanding permissions customisation now correctly loaded seventy-one permissions across six categories: User with fourteen, Financial with nineteen, Audit with fourteen, System with twenty-eight, Configuration with fourteen, and Deployment with seven. The role match indicator correctly displayed the role name instead of the custom label. The promotion was submitted successfully, redirecting to the admin list where the promoted user appeared with the correct role and active status, with the admin count increasing from three to four.

**Security Hardening of the Admin Creation Pipeline:** A hard protocol rule mandating that all administrators must be existing on-chain platform participants triggered a comprehensive security audit of the entire admin creation surface area. Four gaps were identified and addressed. The first critical gap was a legacy appointment endpoint that still accepted password-based admin creation without requiring a participant profile, with the admin repository even auto-generating a user profile to bypass KYC and blockchain identity verification. The route was disabled in the routes file with the use case code preserved but unreachable. The second critical gap was the absence of role hierarchy enforcement in the promotion use case — unlike the legacy appointment and role update use cases which both enforce that the assigner's hierarchy level must exceed the target, the new promotion use case had zero hierarchy checking, allowing a super administrator to promote to the same rank. The fix extracted the promoter's role from the JWT context in the controller and added hierarchy validation in the use case matching the pattern used by the legacy appointment flow. The third high-severity gap was that custom permissions accepted arbitrary strings with no validation. The fix added a pre-creation step that fetches all system permissions, builds a valid code set, and rejects any codes not found in the set. A fourth finding regarding KYC field naming was investigated and confirmed as a false positive — standard ORM mapping between Prisma field names and read model property names. Nine commits across two repositories captured these security fixes including a new domain error class, Fabric identity enforcement in the promotable search and promotion use case, legacy endpoint disablement, role hierarchy and permission validation enforcement, controller role extraction, and dependency injection for the RBAC repository.

**OAuth2 PKCE SSO Flow End-to-End Debugging:** The remaining work from the SSO module involved making the Hub-to-administration-portal navigation trigger a seamless OAuth2 PKCE authentication flow rather than requiring manual login. The command centre card originally performed a plain navigation to the portal login page without initiating SSO. The fix added an SSO trigger parameter to the redirect URL, with the portal login form detecting this parameter and auto-initiating the SSO flow using a lazy useState initialisation pattern to satisfy the React Compiler's setState-in-effect rule. The OAuth authorise proxy route then produced a 500 error that appeared to be a framework crash but was diagnosed through direct server log inspection as a backend error response being correctly forwarded. The investigation revealed an HTTP method mismatch where the proxy sent a POST request but the backend expected GET with query parameters, an expired access token issue where the raw cookie reader did not trigger the JWT callback's refresh logic requiring a switch to the session handler that performs token refresh, and a database schema drift where a NOT NULL constraint on a plaintext authorisation code column conflicted with the hash-only storage design requiring a manual column alteration and index removal. After resolving all five layers of issues, the full seventeen-step SSO flow was verified end-to-end through browser automation: from the Hub card click through PKCE parameter generation, redirect to the wallet login with OAuth parameters, session detection and authorise call with fresh token, auth code generation with hash-only storage, redirect to the portal callback with code and state, PKCE verification and token exchange, and final dashboard rendering with the correct user and role. The total redirect chain completes in approximately two to three seconds with warm caches.

**Hub Layout Migration to Standalone Route Group:** The Hub page was migrated from its initial location within the main client route group that renders with sidebar and header chrome to a standalone route group with a minimal authentication-protected layout. This ensures the Hub renders as a clean context selector without dashboard navigation elements, matching its architectural role as a layer above any single application.

**React Hydration Mismatch Fix in SSO Callback:** During Playwright testing of the SSO callback flow, React threw a recoverable hydration error caused by a server-side guard in a component already wrapped in a Suspense boundary. The component's useMemo accessed sessionStorage with a typeof window check that produced an error UI path on the server and a loading spinner path on the client, creating divergent HTML trees that hydration could not reconcile. The fix removed the server guard entirely because the Suspense boundary guarantees the component only renders on the client, making the guard both unnecessary and actively harmful.

**Cross-Application Logout Architecture Design and Implementation:** A fundamental UX and security gap was identified: the two applications have completely independent session lifecycles with different authentication systems — one using a state management store with localStorage and the other using NextAuth with encrypted JWT cookies — meaning a user who logs out of one remains fully authenticated in the other. Three architectural approaches were evaluated. A shared backend session revocation approach was rejected due to the infrastructure overhead of token blacklists and session stores with polling. A postMessage via hidden iframe approach was rejected due to CORS complexity and Content Security Policy fragility. A redirect chain approach was selected for its simplicity and guarantee: on logout, the user is redirected through the other application's remote logout endpoint which clears that application's session using its own native mechanism before redirecting back. Two new endpoints were created: a server-side API route in the wallet application that iterates over all NextAuth cookies and expires them before validating and redirecting to the return URL, and a client-side page in the administration portal that clears localStorage tokens, broadcasts logout to other tabs, and redirects to the validated return URL. Both existing logout handlers in the wallet application and the administration portal's auth store were modified to redirect through the other application's remote-logout endpoint. The administration portal's public routes allowlist was updated to include the remote-logout path. Both endpoints validate the return URL against a strict origin allowlist to prevent open redirect vulnerabilities.

**Hub Link Integration in Administration Portal Sidebar:** The Hub page link was added to the administration portal sidebar under the Dashboard section using the LayoutGrid icon. Because the Hub is a wallet application page, the sidebar link points to the external wallet URL, and the existing active-item detection correctly never highlights it since external URLs cannot match the internal pathname. TypeScript compilation passed in both applications with zero owned errors, and all ESLint checks passed on modified files.
